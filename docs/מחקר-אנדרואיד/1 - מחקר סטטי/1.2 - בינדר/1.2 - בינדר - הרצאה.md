
## בינדר- Binder

### מה זה Binder

הBinder הוא מנגנון התקשורת המרכזי בין תהליכים (IPC - Inter Process Communication) במערכת ההפעלה Android.  
אנדרואיד בנויה כך שכל אפליקציה רצה בתוך תהליך נפרד עם משתמש והרשאות משלה. בגלל הבידוד הזה, אפליקציות לא יכולות לגשת ישירות לזיכרון אחת של השניה.

כדי לאפשר תקשורת בין תהליכים שונים, אנדרואיד משתמשת במנגנון בשם Binder.

הBinder מאפשר:

- קריאה לפונקציות בתהליך אחר
- העברת אובייקטים ונתונים בין תהליכים
- תקשורת בין אפליקציות
- תקשורת בין אפליקציות לשירותי מערכת

למעשה, intent-ים, broadcast-ים, service-ים כל אלו הם מנגנונים שעובדים מעל binder מאחורי הקלעים.
לדוגמה:

- אפליקציה שמבקשת מיקום מה־Location Service
- אפליקציה שמנגנת מוזיקה דרך Media Service
- אפליקציה שמפעילה מצלמה 

כל אלה מתבצעים דרך Binder.

## הארכיטקטורה של Binder

המערכת בנויה מכמה רכיבים:

1. הClient  
    האפליקציה שמבקשת שירות.
2. הServer  
    השירות שמספק את הפונקציונליות.
3. הBinder Driver  
    דרייבר בקרנל של לינוקס שאחראי להעברת ההודעות.
4. הService Manager  
    רשימה של שירותים במערכת.

התהליך: Client → Binder Driver → Server
כאשר Client קורא לפונקציה, Android למעשה שולח הודעה דרך Binder Driver אל השירות.

## הBinder Driver בקרנל של לינוקס

בקרנל של Android Linux קיים דרייבר בשם:

```
/dev/binder
```

האפליקציות לא ניגשות אליו ישירות בדרך כלל.  
ה־Framework של Android משתמש בו כדי לעשות transactions בין תהליכים.

כאשר מתבצעת קריאה לשירות:

1. הקריאה נארזת כ־Parcel (מבנה הנתונים של binder)
2. מועברת ל־Binder driver
3. מועברת לתהליך השני

## מבנה נתונים Parcel

הParcel הוא מבנה נתונים המשמש להעברת מידע דרך Binder.

הParcel יכול להכיל:

- מספר int
- מחרוזת string
- מערך arrays
- אובייקט objects
- פונקציה binder handles

דוגמה:

```java
Parcel parcel = Parcel.obtain();
parcel.writeInt(42);
parcel.writeString("hello");
```

קריאה בצד השני:

```java
int value = parcel.readInt();
String text = parcel.readString();
```

## פעולת Remote Method Invocation

הBinder מאפשר לבצע קריאה לפונקציה שנמצאת בתהליך אחר כאילו היא פונקציה מקומית.
ממש rpc.
לדוגמה:

```
Client → call getLocation()
Server → returns Location
```

מאחורי הקלעים:

1. הקריאה נארזת ל Parcel
2. נשלחת ל Binder driver
3. מגיעה לשירות
4. השירות מפעילה את הפונקציה
5. השירות מחזירה תשובה

# הAIDL

הAndroid Interface Definition Language או בקצרה AIDL היא דרך להגדיר interface של התקשורת בין תהליכים.
לדוגמה, אם נרצה לפתח שירות שמאפשר לאפליקציות אחרות לקרוא לפונקציות מחשבון באפליקציה שלנו, נוכל להגדיר את הממשק הבא:
 קובץ AIDL:
ICalculator.aidl
```java
interface ICalculator {
    int add(int a, int b);
}
```
הקובץ הזה מגדיר interface שמשמש את הbinder לדעת איזה מידע השירות שלנו יכול לקבל, ואיזה פונקציות אנחנו מייצאים דרך binder.

הנה דוגמה למימוש הinterface.
```java
public class CalculatorService extends Service {

    private final ICalculator.Stub binder = new ICalculator.Stub() {
        @Override
        public int add(int a, int b) {
            return a + b;
        }
    };

    @Override
    public IBinder onBind(Intent intent) {
        return binder;
    }
}
```
כאן יצרנו שירות שמאפשר דרך binder לקרוא לפונקציה add שהגדרנו.
```xml
<service
    android:name=".CalculatorService"
    android:exported="true"/>
```
לאחר שנסיים, נייצא את השירות באמצעות exported=true

קוד הלקוח שמתחבר לשירות יראה כך:
```java
private ICalculator service;

ServiceConnection connection = new ServiceConnection() {

    @Override
    public void onServiceConnected(ComponentName name, IBinder binder) {
        service = ICalculator.Stub.asInterface(binder);
    }

    @Override
    public void onServiceDisconnected(ComponentName name) {
        service = null;
    }
};

Intent intent = new Intent();
intent.setComponent(new ComponentName(
        "com.example.binder",
        "com.example.binder.CalculatorService"));

bindService(intent, connection, Context.BIND_AUTO_CREATE);

int result = service.add(2,3);
```

# הStub והProxy

מאחורי הקלעים חיבור Binder יש 2 שירותים שעומדים לפני השרת והלקוח:
### הStub
נמצא בצד השרת.  
מקבל את ה־transactions ומפעיל את הפונקציות.
### הProxy
נמצא בצד הלקוח.  
אורז את הקריאות ל־Parcel ושולח דרך Binder.

```
Client
  |
Proxy
  |
Binder Driver
  |
Stub
  |
Server
```

# בינדר באנדרואיד

רוב השירותים Android עובדים עם Binder.
- שירות הActivityManager, שבין היתר אחראי להעביר intent-ים בין פעילויות- גם עובד עם binder מאחורי הקלעים.
- שירות הPackageManager שאחראי להתקין חבילות.
- שירות הWindowManager שאחראי על הui במכשיר
- שירות הLocationManager שאחראי על הgps

כאשר האפליקציה שלנו נגשת לשירות מסוים כדי לבצע פעולה, היא מבקשת מה״Service Manager״ את השירות.
הService Manager הוא תהליך שממפה את כל השירותים השונים באנדוראיד דרך binder.
לדוגמה כשאנחנו כותבים קוד שניגש לשירות:
```java
ActivityManager am = (ActivityManager)getSystemService(Context.ACTIVITY_SERVICE);
```
מאחורי הקלעים הקריאה עוברת דרך Binder באמצעות Service manager.
הService Manager שומר רשימה של כל השירותים, ויודע למפות אותם לbinder handler.

כאשר אפליקציה מבקשת שירות:
```kotlin
getSystemService()
```
1. אנדרואיד פונה ל Service Manager
2. מקבל Binder handle
3. משתמש בו לקריאות IPC

# בינדר במחקר

בינדר חשוב מאוד במחקר Android.
מאפשר לך להבין בצורה יותר עמוקה תקשורת בין אפליקציות, וכיצד system services עובדים.

הפקודה הבאה מאפשרת לך לראות רשימה של כל השירותים של אפליקציה מסוימת
```bash
service list
```

הפקודה הבאה מאפשרת לך לקרוא לשירותים מסוימים
```bash
service call
```

הפקודה הזו מאפשרת לך לקבל המון מידע על שירותים שונים באפליקציה.
```
dumpsys
```
