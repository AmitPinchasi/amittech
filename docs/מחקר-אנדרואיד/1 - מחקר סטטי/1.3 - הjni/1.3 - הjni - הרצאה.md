## ספריות Native באנדרואיד

בהרבה אפליקציות אנדרואיד חלק מהלוגיקה לא נמצא בקוד Java או Kotlin אלא בקוד **Native** שנכתב ב־C או C++.
קוד כזה מקומפל לספריות `.so` (shared object) 
אחרי שכתבנו קוד בnative, נוכל לקרוא לפונקציות בקוד ממש דרך הjava באמצעות **JNI (Java Native Interface)**.

מפתחים ולעיתים גם malware משתמשים בקוד native כדי:
- להקשות על Reverse Engineering
- להסתיר לוגיקה בקוד native-י
- לבצע פעולות low-level-יות
- להשתמש בספריות קיימות ב-C/C++

לכן חוקר אפליקציות אנדרואיד חייב לדעת לנתח גם **DEX code** וגם **Native code**
# ספריות Native Libraries בתוך APK

בתוך apk ניתן למצוא ספריות native בתיקיה:
```
lib/
   armeabi-v7a/
   arm64-v8a/
   x86/
```

לדוגמה:
```
libnative-lib.so
libcrypto.so
libfoo.so
```

ספריות אלו מכילות קוד מכונה של ARM או x86 (כדי לתמוך בכל הארכטיקטורות)

האפליקציה טוענת את הספריה בזמן ריצה באמצעות:
```java
System.loadLibrary("native-lib");
```

או:
```java
System.load("/data/app/libnative-lib.so");
```
לאחר טעינת הספריה ניתן לקרוא לפונקציות native מתוך Java. 


# JNI – Java Native Interface

הJNI הוא מנגנון שמאפשר לקוד Java לקרוא לפונקציות שנכתבו ב-C או C++.
דוגמה:
```java
public class MainActivity {
    public native String getSecret();
}
```

המילה `native` אומרת שהפונקציה לא ממומשת ב-Java אלא בספריה native.
המימוש ב-C יראה כך:
```c
JNIEXPORT jstring JNICALL
Java_com_example_app_MainActivity_getSecret(JNIEnv *env, jobject obj) {
    return (*env)->NewStringUTF(env, "secret value");
}
```

שם הפונקציה נבנה לפי מבנה:
```
Java_<package>_<class>_<method>
```


# איך חוקרת מזהה פונקציות Native

כאשר מנתחים APK צריך לענות על כמה שאלות מרכזיות:

1. אילו פונקציות native מוגדרות בקוד Java
2. איזו ספריה נטענת
3. איפה המימוש של הפונקציה נמצא בספריה
4. מה הקוד native עושה בפועל

מטרת הניתוח היא לקשר בין:
```
Java Method
      ↓
JNI Function
      ↓
Native Code
```

# שלב 1 – מציאת פונקציות Native ב-DEX

פותחים את האפליקציה ב־**jadx**.
מחפשים פונקציות שמסומנות:
```java
native
```

לדוגמה:
```java
public native int calculateKey(int a, int b);
```

משמעות הדבר:
- הפונקציה לא ממומשת ב-Java
- המימוש נמצא בספריה native

לעיתים גם נראה טעינת ספריה:
```java
static {
    System.loadLibrary("crypto");
}
```

מכאן ניתן להסיק שהקוד נמצא בקובץ:
```
libcrypto.so
```

# שלב 2 – זיהוי הספריה

לאחר שמצאנו את `loadLibrary` אנחנו יודעים איזו ספריה לחפש בתוך ה-APK.
לדוגמה:
```
lib/arm64-v8a/libcrypto.so
```

אפשר לחלץ את הספריה באמצעות:
```
unzip app.apk
```

או:
```
apktool d app.apk
```

לאחר מכן נקבל את הקובץ `.so`.

# דוגמה לפונקציה JNI
```c
JNIEXPORT jstring JNICALL
Java_com_example_app_MainActivity_getSecret(JNIEnv *env, jobject obj)
{
    return (*env)->NewStringUTF(env, "hello");
}
```

מה קורה כאן:

1. הJava קורא לפונקציה native
2. הJNI מעביר את הקריאה לספריה
3. הספריה מחזירה String ל-Java

# מבנה טיפוסי של JNI

חתימת פונקציה JNI תמיד נראית בערך כך:
```c
JNIEXPORT return_type JNICALL
Java_package_class_method(JNIEnv *env, jobject obj, ...)
```

הפרמטרים הראשונים תמיד יהיו:
```
JNIEnv* env
jobject obj
```

# איך מקשרים Java לפונקציה Native
בדרך כלל יש שתי שיטות:

## 1. Static Binding

שם הפונקציה:
```
Java_com_example_app_Class_method
```
ה-JVM מוצא את הפונקציה לפי השם.

## 2. Dynamic Registration

לעיתים האפליקציה משתמשת בפונקציה:
```
RegisterNatives
```
שמקשרת בין סימבולים של פונקציות בnative לבין jni
ואז במקרה כזה הקישור בין Java ל-native נעשה בזמן ריצה. (איי אפשר להבין רק לפי השם, מקשה על המחקר)
כי השם לא יהיה בהכרח `Java_com_example_app_Class_method`

# למה Malware משתמש ב-Native Code

אפליקציות זדוניות לעיתים מעבירות קוד מסוכן ל-native.

סיבות:

1. קוד native קשה יותר לניתוח
2. אפשר להשתמש ב-assembly
3. ניתן להסתיר לוגיקה זדונית
4. ניתן להקשות על static analysis
