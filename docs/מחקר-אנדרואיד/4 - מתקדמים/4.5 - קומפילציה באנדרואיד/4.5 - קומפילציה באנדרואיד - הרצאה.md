## קומפילציה באנדרואיד - OAT and ART Compilation
בשיעור הזה נבין לעומק איך קוד java/kotlin הופך בסופו של דבר לקוד שרץ על המכשיר.
זה חשוב למחקר כי לפעמים הקוד שאנחנו רואים על המכשיר נראה שונה ממה שjadx מראה לנו.

## ההיסטוריה - From Dalvik to ART
כדי להבין את המצב הנוכחי, חשוב להבין את ההיסטוריה.

### הDalvik (עד אנדרואיד 4.4)
בגרסאות ישנות של אנדרואיד, הruntime נקרא **Dalvik**.
הdalvik היה מכונה וירטואלית שהריצה DEX bytecode בזמן אמת - **interpretation**.
כל שורת bytecode תורגמה לקוד מכונה תוך כדי הרצה, מה שהיה איטי.

הDalvik השתמש גם ב-**JIT (Just-In-Time)** compilation - קוד שרץ הרבה פעמים קומפל לnative code בזמן ריצה כדי לשפר ביצועים.
אבל הJIT עבד רק בזמן שהאפליקציה רצה. בכל פעם שפתחו את האפליקציה מחדש, הקוד עבר JIT מחדש.

### הART (מאנדרואיד 5.0)
באנדרואיד 5.0 (Lollipop) גוגל החליפו את Dalvik ב-**ART (Android Runtime)**.
השינוי הגדול: ART הוסיף **AOT (Ahead-Of-Time)** compilation - הקוד מקומפל לnative code **בזמן ההתקנה**, לא בזמן הריצה.

### הגישה המשולבת (מאנדרואיד 7.0)
באנדרואיד 7.0 (Nougat) גוגל שינו שוב את הגישה לגישה משולבת:
- **התקנה** - לא מקמפלים הכל. האפליקציה רצה עם interpretation + JIT
- **פרופיל** - בזמן ריצה, ART אוסף מידע על אילו מתודות נקראות הכי הרבה
- **קומפילציה ברקע** - כשהמכשיר טוען ולא בשימוש, המערכת מקמפלת AOT את המתודות הפופולריות (לפי הפרופיל)
- **הרצה הבאה** - המתודות שקומפלו רצות כnative, השאר ממשיך עם JIT

הגישה הזו נקראת **Profile-Guided Compilation** והיא מאזנת בין מהירות התקנה, מהירות ריצה, וגודל אחסון.

## הכלי dex2oat
הכלי שמבצע את הקומפילציה מDEX לקוד native נקרא **dex2oat** (dex to OAT).
```
classes.dex  -->  dex2oat  -->  .oat / .odex / .vdex / .art
```

הכלי נמצא ב:
```bash
adb shell which dex2oat
# /system/bin/dex2oat
```

### מתי dex2oat רץ
- **בזמן התקנת אפליקציה** - `adb install` או התקנה מPlay Store
- **אחרי עדכון מערכת** - כל האפליקציות עוברות re-compilation (לכן עדכון מערכת לוקח זמן - "optimizing apps")
- **ברקע** - הjob scheduler מריץ קומפילציה כשהמכשיר טוען ולא בשימוש
- **בבוט ראשון** - קומפילציה של אפליקציות מערכת

### רמות קומפילציה - Compiler Filters
אפשר לשלוט ברמת הקומפילציה עם compiler filters:
לכל אפליקציה יש filter שונה.
הנה הפילטרים הקיימים:
- ה**verify** - רק מוודא שהbytecode תקין, לא מקמפל (הכי מהיר להתקנה)
- ה**quicken** - אופטימיזציות קלות לbytecode
- ה**speed-profile** - קומפילציה לפי פרופיל (ברירת המחדל)
- ה**speed** - קומפילציה מלאה של הכל לnative

אפשר לראות את הfilter של אפליקציה:
```bash
adb shell dumpsys package com.example.app | grep -i "compiler\|dex"
```

## קבצי הפלט - Output Files
הdex2oat מייצר כמה קבצים:

### קובץ OAT
הפורמט העיקרי. זה בעצם קובץ **ELF** (כמו ספריית so רגילה בלינוקס) שמכיל:
- ה**oatdata** (בsection של .rodata) - מטאדאטה על הDEX המקורי
- ה**oatexec** (בsection של .text) - הקוד הnative המקומפל

אפשר לוודא שזה ELF:
```bash
adb pull /data/dalvik-cache/arm64/system@framework@boot.oat ./boot.oat
file boot.oat
# ELF 64-bit LSB shared object, ARM aarch64...
```

### קובץ VDEX
מכיל את הDEX bytecode המקורי (אולי עם אופטימיזציות קלות).
הVDEX נשמר בנפרד כדי שאם צריך לעשות re-compilation, לא יצטרכו לחלץ את הDEX מהAPK מחדש.

### קובץ ART
מכיל snapshot של אובייקטים מ-heap - בדרך כלל מחלקות שנטענו מראש (preloaded).
נמצא בעיקר עבור boot image.

### איפה הקבצים נמצאים
קבצי הקומפילציה נמצאים בשני מקומות:

**עבור אפליקציות מערכת (boot image):**
```
/system/framework/oat/<arch>/
/system/framework/arm64/boot.oat
/system/framework/arm64/boot.vdex
/system/framework/arm64/boot.art
```

**עבור אפליקציות מותקנות:**
```
/data/dalvik-cache/<arch>/
```
או בתוך תיקיית הAPK:
```
/data/app/<package>/oat/<arch>/base.odex
/data/app/<package>/oat/<arch>/base.vdex
```

אפשר לראות:
```bash
adb shell ls /data/app/*/oat/arm64/
```

## מחקר OAT - OAT Research

### חילוץ DEX מOAT
לפעמים אנחנו מוצאים על מכשיר קובץ OAT או VDEX ורוצים לחלץ ממנו את הDEX כדי לפתוח אותו בjadx.

**מVDEX:**
הכלי **vdexExtractor** מחלץ DEX מקבצי VDEX:
```bash
vdexExtractor -i base.vdex -o output/
```

**מOAT:**
הכלי **oat2dex** מחלץ DEX מקבצי OAT:
```bash
java -jar oat2dex.jar base.odex
```

### ניתוח הקוד הnative
אם רוצים לראות את הקוד הnative שdex2oat ייצר, אפשר לפתוח את קובץ הOAT בGhidra או IDA.
זה שימושי כשרוצים להבין אופטימיזציות שART ביצע, או כשמחפשים חולשות ברמת הruntime.

### הדגל debuggable
כשאפליקציה מוגדרת כ-`debuggable=true` בmanifest, הdex2oat לא מקמפל אותה לnative (כדי לאפשר debugging).
זה אומר שאפליקציה debuggable תרוץ לאט יותר אבל תהיה קלה יותר לניתוח דינמי.

אפשר לשנות אפליקציה להיות debuggable:
1. פרקי עם apktool
2. שני בmanifest: `android:debuggable="true"`
3. בני מחדש וחתמי

### boot image
הboot image (`boot.oat`) מכיל את כל הframework של אנדרואיד מקומפל.
זה הקוד שהזיגוט טוען. ניתוח שלו מאפשר:
- להבין את הimplementation של API-ים של אנדרואיד
- לחפש חולשות בframework
- להבין מנגנוני אבטחה מבפנים

```bash
adb pull /system/framework/arm64/boot.oat
adb pull /system/framework/arm64/boot.vdex
```

## הזרימה המלאה - Full Pipeline
בואו נראה את הזרימה המלאה ממה שהמפתח כותב ועד מה שרץ על המכשיר:

```
קוד מקור (Java/Kotlin)
        |
        v
    javac / kotlinc        (קומפילציה לjava bytecode)
        |
        v
    .class files
        |
        v
    D8 / R8                (המרה לDEX bytecode + אופטימיזציה)
        |
        v
    classes.dex            (בתוך הAPK)
        |
        v
    dex2oat                (קומפילציה לnative, על המכשיר)
        |
        v
    .oat / .vdex / .art    (קוד native + DEX + heap snapshot)
        |
        v
    ART Runtime            (מריץ native(aot) + JIT + interpretation)
```

## סיכום
- אנדרואיד משתמש בגישה משולבת של **AOT + JIT + interpretation** (קצת מסובך)
- הכלי **dex2oat** מקמפל DEX לקוד native בפורמט OAT
- קבצי OAT הם קבצי ELF עם קוד native, קבצי VDEX מכילים DEX
- אפשר לחלץ DEX מVDEX/OAT עם כלים כמו vdexExtractor ו-oat2dex
- הבנת pipeline הקומפילציה עוזרת לנו להבין מה אנחנו רואים כשחוקרים אפליקציה
