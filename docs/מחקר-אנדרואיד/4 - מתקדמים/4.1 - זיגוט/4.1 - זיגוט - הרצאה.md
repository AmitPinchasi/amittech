## הזיגוט - Zygote
כדי להבין לעומק את אנדרואיד, צריך להבין איך אפליקציות נולדות.
באנדרואיד, כל אפליקציה רצה בפרוסס נפרד. אבל יצירת פרוסס חדש מאפס היא פעולה יקרה - צריך לטעון את כל הספריות של אנדרואיד, לאתחל את הruntime, ועוד.
כדי לפתור את הבעיה הזו, אנדרואיד משתמש במנגנון שנקרא **Zygote** (זיגוט - ביולוגית זו ביצית מופרית, השלב הראשון של חיים חדשים).

## תהליך האתחול המלא - Full Boot Process
לפני שנצלול לזיגוט, בואו נבין את התמונה הגדולה - מה קורה כשמדליקים מכשיר אנדרואיד.

### שלב 1 - הBootloader
כמו בכל מחשב, הדבר הראשון שקורה זה שהחומרה מריצה את הbootloader.
הbootloader הוא קוד שיושב על שבב הזכרון של המכשיר, והתפקיד שלו הוא לטעון את מערכת ההפעלה.
בטלפונים, הbootloader בדרך כלל נעול (locked) - כלומר הוא יטען רק מערכת הפעלה שחתומה דיגיטלית על ידי היצרן.
כשעושים "unlock bootloader" (שלב ראשון בroot של מכשירים אמיתיים), מאפשרים לbootloader לטעון מערכת הפעלה שלא חתומה.

### שלב 2 - הקרנל (לינוקס)
הbootloader טוען את קרנל הלינוקס לזכרון ומעביר אליו שליטה.
הקרנל מאתחל את החומרה - דרייברים למסך, למצלמה, לרשת, לזכרון, ועוד.
הדבר החשוב שקורה כאן הוא שאנדרואיד הוא בסופו של דבר **לינוקס**. כל מה שלמדנו על לינוקס רלוונטי - פרוססים, הרשאות, מערכת קבצים, סיגנלים, הכל.

### שלב 3 - הפרוסס init
אחרי שהקרנל עלה, הוא מריץ את הפרוסס הראשון במערכת - `init` (PID 1).
הinit קורא קבצי קונפיגורציה שנקראים `init.rc` שמגדירים אילו שירותים ופרוססים צריך להריץ.
הוא אחראי על:
- הרמת מערכת הקבצים (mount של partitions)
- הגדרת הרשאות SELinux
- הפעלת שירותים בסיסיים
- והכי חשוב - הפעלת הזיגוט

בקבצי הinit.rc אפשר לראות את השורה שמפעילה את הזיגוט:
```
service zygote /system/bin/app_process64 -Xzygote /system/bin --zygote --start-system-server
```
הפקודה `app_process64` היא התוכנית שמאתחלת את הART ומריצה את הזיגוט.
שימו לב לדגל `--start-system-server` - נחזור לזה בהמשך.

### שלב 4 - הזיגוט עולה
הזיגוט מתחיל לרוץ ומבצע כמה דברים חשובים:

**טעינת ה-ART (Android Runtime)**
הזיגוט מאתחל את מכונת הjava הוירטואלית של אנדרואיד. זה כולל את הheap, הgarbage collector, והJIT compiler.

**טעינת מחלקות מראש - Preloading**
הזיגוט טוען לזכרון אלפי מחלקות java שכל אפליקציה צריכה:
- כל המחלקות של `java.*` (String, ArrayList, HashMap...)
- כל המחלקות של `android.*` (Activity, View, Intent...)
- כל המחלקות של `javax.*` (crypto, net...)
- משאבי מערכת (fonts, drawables בסיסיים)

הרשימה המלאה נמצאת בקובץ `/system/etc/preloaded-classes` על המכשיר:
```bash
adb shell wc -l /system/etc/preloaded-classes
```
בדרך כלל מדובר ביותר מ-7000 מחלקות.

**פתיחת socket**
הזיגוט פותח Unix domain socket בנתיב `/dev/socket/zygote` ומתחיל להאזין לבקשות ליצירת פרוססים חדשים.

### שלב 5 - הsystem_server נולד
הדבר הראשון שהזיגוט עושה אחרי שסיים לטעון הכל, הוא לעשות `fork()` ליצירת הפרוסס `system_server`.
הsystem_server הוא הפרוסס הכי חשוב באנדרואיד - הוא מריץ את כל שירותי המערכת (נלמד עליו בשיעור הבא).

### שלב 6 - המתנה
מעכשיו הזיגוט פשוט מחכה על הsocket. כל פעם שמישהו (הsystem_server) מבקש ליצור פרוסס חדש לאפליקציה, הזיגוט מבצע fork ויוצר אותו.

## מה קורה כשפותחים אפליקציה - App Launch Process
בואו נעקוב צעד אחר צעד על מה שקורה כשמשתמש לוחץ על אייקון של אפליקציה:

### 1. הLauncher שולח Intent
אפליקציית הlauncher (מסך הבית) שולחת intent של `ACTION_MAIN` עם `CATEGORY_LAUNCHER` לActivityManagerService.

### 2. הAMS מעבד את הבקשה
הActivityManagerService (שרץ בsystem_server) בודק:
- האם האפליקציה כבר רצה? אם כן, פשוט מעביר אליה focus
- האם יש הרשאות מתאימות?
- מה הactivity שצריך להפעיל?

### 3. הAMS מבקש פרוסס חדש
אם האפליקציה לא רצה, הAMS קורא ל-`Process.start()` שבתורו שולח הודעה לזיגוט דרך הsocket.
ההודעה כוללת:
- שם המחלקה הראשית (בדרך כלל `android.app.ActivityThread`)
- הUID של האפליקציה (לצורך sandbox)
- הGID-ים (קבוצות הרשאות)
- דגלים נוספים

### 4. הזיגוט עושה fork
הזיגוט מקבל את ההודעה ומבצע קריאת מערכת `fork()`.
מהרגע הזה יש שני פרוססים זהים:
- **הפרוסס האב** (הזיגוט) - חוזר להאזין על הsocket
- **הפרוסס הבן** (האפליקציה החדשה) - ממשיך לשלב הבא

### 5. הפרוסס החדש מתמחה
הפרוסס החדש:
- מגדיר את הUID והGID שלו (sandbox) (היוזר של האפליקציה)
- סוגר את הsocket של הזיגוט (הוא לא צריך אותו)
- טוען את הDEX של האפליקציה מתוך הapk
- יוצר אובייקט `ActivityThread`
- קורא ל-`ActivityThread.main()` שמתחיל את הlooper של האפליקציה
- הAMS אומר לאפליקציה ליצור את הactivity הראשון

### 6. האפליקציה רצה
מכאן האפליקציה רצה כפרוסס עצמאי עם הUID שלה, sandbox משלה, וheap משלה.

## העתקה בעת כתיבה - Copy-on-Write
המנגנון הזה עובד בזכות תכונה של לינוקס שנקראת **Copy-on-Write** (COW).
כשהזיגוט עושה fork, הקרנל לא באמת מעתיק את כל הזכרון של הזיגוט לפרוסס החדש.
במקום זה, שני הפרוססים **משתפים** את אותם דפי זכרון, ודפי הזכרון מסומנים כread-only.

רק כשאחד הפרוססים מנסה **לכתוב** לדף זכרון, הקרנל יוצר עותק של הדף הספציפי הזה ומעדכן את הpage table.

בפועל זה אומר:
- **מהירות** - הfork כמעט מיידי, כי לא מעתיקים כלום
- **חיסכון בזכרון** - כל 7000+ המחלקות שהזיגוט טען קיימות בזכרון פעם אחת בלבד, גם אם 50 אפליקציות רצות. כל אפליקציה "רואה" אותן אבל לא מעתיקה אותן
- **אחידות** - כל אפליקציה מתחילה מאותו מצב בדיוק

כדי לראות את החיסכון הזה בפועל:
```bash
adb shell cat /proc/<pid>/smaps | grep -i shared
```
נראה שחלק גדול מהזכרון הוא shared - כלומר משותף עם הזיגוט.

## הזיגוט מבחינת מחקר - Zygote in Security Research
למה הזיגוט חשוב לנו כחוקרים?

### הזרקת קוד לזיגוט - Zygote Injection
אם נצליח להזריק קוד לפרוסס הזיגוט, הקוד שלנו ירוץ בתוך **כל אפליקציה** שתיפתח אחר כך - כי כולן fork מהזיגוט.
זה הרעיון מאחורי כמה כלים מאוד חשובים:

ה**Xposed Framework**
פריימוורק שמזריק קוד לזיגוט ומאפשר לכתוב מודולים שעושים hook לכל פונקציה java בכל אפליקציה. הוא עובד על ידי החלפת `app_process` בגרסה שטוענת את הXposed framework לפני שהזיגוט מאתחל.

ה**Zygisk (Magisk)**
מנגנון חדש יותר שמגיע כחלק מMagisk. הוא מזריק ספריית so לתהליך הזיגוט, וכך מודולים יכולים לרוץ בתוך כל אפליקציה.
הZygisk הוא שמירה יחסית על backwards compatibility עם API של Xposed אבל עם מימוש חדש ויותר יציב.

שני הכלים האלו פופולריים מאוד בקהילת המחקר כי הם מאפשרים:
- מעקף root detection בכל אפליקציה
- יצירת hook לכל פונקציה java בכל אפליקציה
- שינוי התנהגות המערכת כולה

### פרידה מול הזיגוט - Frida vs Zygote
כשfrida עושה attach לאפליקציה, היא מתחברת לפרוסס שכבר קיים.
אבל לפעמים אנחנו רוצים לתפוס קוד שרץ מאוד מוקדם - למשל בדיקות root שרצות ב-`onCreate` של הactivity הראשי.

בשביל זה יש את spawn mode של frida:
```bash
frida -U -f com.example.app -l script.js
```
הדגל `-f` גורם לfrida לבקש מהזיגוט ליצור את הפרוסס ולתפוס אותו מיד אחרי הfork, לפני שהאפליקציה מתחילה לרוץ. כך אנחנו יכולים לעשות hook עוד לפני שהקוד של האפליקציה רץ.

## בדיקה מעשית - Hands-on
אפשר לראות את הזיגוט בפעולה על המכשיר:
```bash
adb shell ps -e | grep zygote
```
נראה משהו כמו:
```
root  1234  1  ...  zygote64
root  1235  1  ...  zygote
```
יש שני זיגוטים - אחד ל64 ביט ואחד ל32 ביט (לתמיכה באפליקציות ישנות).
שימו לב שהPPID (עמודה שנייה) הוא 1 - כלומר הinit הוא שהריץ אותם.

כדי לראות שכל האפליקציות הן ילדים של הזיגוט:
```bash
adb shell ps -e -o PID,PPID,NAME | head -50
```
נראה שהPPID של כל אפליקציה (com.example...) הוא הPID של הzygote64 או zygote.

אפשר גם לראות את הsocket שהזיגוט מאזין עליו:
```bash
adb shell ls -la /dev/socket/zygote
```
