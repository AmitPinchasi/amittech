## מערכת הקבצים של אנדרואיד - Android Filesystem
באנדרואיד, מערכת הקבצים שונה ממה שאנחנו מכירים מלינוקס רגיל.
יש partitions שונים, תיקיות עם תפקידים מיוחדים, ומנגנוני הגנה שמגבילים גישה.
כחוקרי אנדרואיד, חשוב שנכיר את המבנה הזה לעומק - לדעת איפה כל דבר נמצא ואיך לגשת אליו.

## מחיצות - Partitions
מכשיר אנדרואיד מחולק למחיצות (partitions), כל אחת עם תפקיד שונה.
אפשר לראות את כל המחיצות שמותקנות (mounted) על המכשיר:
```bash
adb shell mount
```
או בצורה יותר קריאה:
```bash
adb shell df -h
```

### מחיצת boot
מכילה את הקרנל ואת הramdisk (מערכת קבצים זמנית שנטענת בעת ההדלקה).
הramdisk מכיל את קובץ הinit הראשון ואת קבצי הrc שמגדירים את תהליך האתחול.
המחיצה הזו היא read-only, ושינוי שלה דורש root ותהליך מיוחד (flashing).
- זה זהה גם ללינוקס סטנדרטי

### מחיצת system (/system)
זו המחיצה הכי חשובה. היא מכילה את מערכת ההפעלה עצמה:
```
/system/
    app/           # אפליקציות מערכת (מותקנות מראש)
    priv-app/      # אפליקציות מערכת עם הרשאות מיוחדות
    framework/     # הספריות הבסיסיות של אנדרואיד (framework.jar, services.jar)
    lib/           # ספריות native של המערכת (32 ביט)
    lib64/         # ספריות native של המערכת (64 ביט)
    bin/           # פקודות שונות (ls, cat, sh, app_process...)
    etc/           # קבצי קונפיגורציה
    fonts/         # פונטים של המערכת
    media/         # צלילים, אנימציית boot
    build.prop     # מאפייני הbuild של המערכת
```

המחיצה הזו היא **read-only** כברירת מחדל (בשונה מלינוקס). גם עם root איי אפשר לערוך או לשנות את הקבצים בה, צריך לעשות remount כדי לכתוב אליה:
```bash
adb shell mount -o rw,remount /system
```

מבחינת מחקר, התיקיות הכי מעניינות כאן הן:
- `framework/` - שם נמצא כל הקוד של הframework של אנדרואיד
- `app/` ו-`priv-app/` - אפליקציות שמותקנות מראש (bloatware, שירותי Google)
- `etc/` - קונפיגורציה, כולל `preloaded-classes` ו-`permissions`

### מחיצת vendor (/vendor)
מכילה דרייברים וספריות ספציפיות ליצרן המכשיר. (כלומר, תוכן שהוא לא ספציפי של אנדרואיד)
לדוגמה, דרייברים של Qualcomm לsnapdragon, או דרייברים של Samsung למסך.
```
/vendor/
    lib/           # ספריות native של היצרן
    lib64/
    etc/           # קונפיגורציה של היצרן
    firmware/      # firmware של חומרה (modem, wifi, bluetooth...)
    app/           # אפליקציות של היצרן
```
גם מחיצה זו היא read-only.

### מחיצת data (/data)
זו המחיצה שמכילה את כל המידע של המשתמש ושל האפליקציות.
זו המחיצה היחידה שהיא **read-write** ומשתנה כל הזמן.
```
/data/
    data/          # תיקיות פרטיות של אפליקציות (/data/data/<package>/)
    app/           # קבצי APK של אפליקציות מותקנות
    dalvik-cache/  # קבצי OAT/DEX שעברו קומפילציה
    system/        # מסדי נתונים של המערכת (accounts, passwords...)
    local/tmp/     # תיקייה זמנית (כאן שמים frida-server!)
    misc/          # מידע מגוון (wifi, bluetooth, profiles...)
    property/      # system properties שנשמרים בין הפעלות
```

כשמבצעים factory reset, המחיצה הזו נמחקת ונוצרת מחדש.

שימו לב - כשאנחנו מתקינים APK עם `adb install`, הAPK נשמר ב-`/data/app/<package>/base.apk` והמידע הפרטי של האפליקציה ב-`/data/data/<package>/`.

### מחיצת cache (/cache)
תיקייה זמנית שמשמשת לעדכוני OTA (עדכוני מערכת), ולcache של recovery.

### מחיצת recovery
מחיצה שמכילה מערכת הפעלה מינימלית שמשמשת לrecovery mode.
בrecovery mode אפשר לעשות factory reset, להתקין עדכונים, ולעשות flash למחיצות.
כלים כמו TWRP מחליפים את הrecovery המקורי ומאפשרים יותר שליטה.

## תיקיות חשובות לחוקרים

### הנתיב /data/data/
כמו שלמדנו, כאן נמצא המידע הפרטי של כל אפליקציה.
כל אפליקציה מקבלת UID ייחודי (משתמש לינוקסי שונה), ותיקיית הdata שלה שייכת לUID הזה:
```bash
adb shell ls -la /data/data/ | head -20
```
נראה שכל תיקייה שייכת למשתמש שונה (u0_a1, u0_a2, וכו') - כל אחד מהם הוא UID ייחודי של אפליקציה.

### הנתיב /data/app/
כאן נמצאים קבצי הAPK של אפליקציות שהמשתמש התקין:
```bash
adb shell ls /data/app/
```
כל אפליקציה נמצאת בתיקייה עם שם מוצפן, ובתוכה הAPK:
```bash
adb shell find /data/app/ -name "base.apk" | head -10
```
זה שימושי כשרוצים לשלוף APK של אפליקציה מותקנת:
```bash
adb shell pm path com.example.app
# יחזיר את הנתיב המלא לAPK
adb pull <path> ./app.apk
```

### הנתיב /proc/
מערכת קבצים וירטואלית שהקרנל יוצר בזמן ריצה. כל פרוסס מקבל תיקייה עם הPID שלו: (כמו בכל מערכת לינוקס סנטדרטית)
```
/proc/<pid>/
    maps       # מפת הזכרון של הפרוסס (אילו ספריות טעונות ואיפה)
    status     # מידע על הפרוסס (שם, UID, זכרון...)
    cmdline    # שורת הפקודה שהפעילה את הפרוסס
    fd/        # קבצים פתוחים
    net/       # מידע רשת (חיבורים פתוחים, פורטים...)
```

זה מאוד שימושי למחקר:
```bash
# לראות אילו ספריות טעונות באפליקציה
adb shell cat /proc/<pid>/maps | grep "\.so"

# לראות חיבורי רשת פתוחים
adb shell cat /proc/<pid>/net/tcp
```

### הנתיב /sys/ 
מערכת קבצים וירטואלית נוספת שחושפת מידע על החומרה והקרנל.
פחות רלוונטי למחקר אפליקציות, אבל שימושי למחקר של המכשיר עצמו.
- נמצא בכל מערכת לינוקס

### הנתיב /sdcard/ (או /storage/emulated/0/)
האחסון הנגיש למשתמש. כאן נמצאים:
```
/sdcard/
    DCIM/          # תמונות מהמצלמה
    Download/      # קבצים שהורדו
    Pictures/      # תמונות
    Android/data/  # אחסון חיצוני של אפליקציות
    Android/obb/   # קבצי expansion של משחקים
```
הנתיב `/sdcard/` הוא בעצם symlink ל-`/storage/emulated/0/`.
באנדרואיד מודרני, גם כאן יש הגבלות גישה בזכות Scoped Storage.

## הרשאות ובעלות 
אנדרואיד משתמש במנגנון ההרשאות של לינוקס כדי לאכוף את הsandbox:
- כל אפליקציה מקבלת UID ייחודי (בדרך כלל בטווח 10000-19999)
- תיקיית הdata של אפליקציה שייכת לUID שלה
- אפליקציות אחרות לא יכולות לגשת לתיקייה (אין read/execute לothers)
```bash
adb shell ls -la /data/data/com.example.app/
# drwxrwx--x  u0_a123  u0_a123  shared_prefs
# drwxrwx--x  u0_a123  u0_a123  databases
```
הUID u0_a123 הוא הבעלים של התיקיות. רק הוא (והroot) יכולים לגשת אליהן.

### מנגנון SELinux
מעבר להרשאות הרגילות של לינוקס, אנדרואיד משתמש גם ב-**SELinux** (Security-Enhanced Linux).
הSELinux מוסיף שכבת הגנה נוספת שמגדירה בצורה מדויקת מה כל פרוסס יכול לעשות - גם אם יש לו הרשאות רגילות.

כדי לבדוק את מצב הSELinux:
```bash
adb shell getenforce
```
- `Enforcing` - SELinux דולק ואוכף את הכללים
- `Permissive` - SELinux מתעד הפרות אבל לא חוסם

כזכור מהשיעור על frida, לפעמים צריך לכבות את SELinux כדי שfrida תעבוד:
```bash
adb shell setenforce 0
```
בגרסאות חדשות של אנדרואיד, קיים חוק selinux שחוסם את root מביצוע ptrace (קריאת מערכת לינוקסית שמאפשרת לגשת לזכרון של תהליכים אחרים)
או בעברית- פרידה לא תעבוד באנדוראיד חדש מבלי לכבות selinux. (כמו שלמדנו בפרק על פרידה)
