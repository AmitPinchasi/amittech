# מה זה ORM ולמה כמעט כל מתכנת בק-אנד נתקל בזה

אם למדתם SQL וחשבתם "מעולה, עכשיו אני יודע לדבר עם מסד נתונים", ואז נכנסתם לפרויקט אמיתי וגיליתם שהקוד בכלל לא כותב שאילתות SQL ישירות אלא קורא לפונקציות כמו User.find() או Customer.objects.filter() - הכרתם ORM. בואו נסביר מה זה ולמה זה כל כך נפוץ.

## אז מה זה בעצם ORM

ORM זה קיצור של Object-Relational Mapping, מיפוי בין אובייקטים לטבלאות רלציוניות. זו שכבת קוד שיושבת בין האפליקציה שלכם לבין מסד הנתונים, ומתרגמת בין שני עולמות שחושבים בצורה שונה.

בקוד שלכם, בפייתון או ג'אווהסקריפט, אתם חושבים במונחים של אובייקטים - למשל אובייקט User עם תכונות name ו-email. במסד הנתונים, אין אובייקטים, יש טבלאות ושורות. ORM הוא הגשר שמאפשר לכם לכתוב "תביא לי משתמש לפי אימייל" בשפת התכנות הרגילה שלכם, והוא מתרגם את זה מאחורי הקלעים לשאילתת SQL אמיתית.

דוגמה בפייתון עם ORM בשם SQLAlchemy:

```python
# עם ORM
user = session.query(User).filter(User.email == 'dana@mail.com').first()

# מה שקורה מאחורי הקלעים
# SELECT * FROM users WHERE email = 'dana@mail.com' LIMIT 1
```

שתי הגרסאות עושות בדיוק אותו דבר. ה-ORM פשוט חוסך לכם לכתוב את השאילתה בעצמכם.

## למה מתכנתים בכלל טורחים להשתמש בזה

- **פחות קוד חוזר.** במקום לכתוב שאילתת SQL בכל מקום בקוד, מגדירים פעם אחת איך אובייקט User נראה, ומשם משתמשים בפונקציות פייתון או ג'אווהסקריפט רגילות.
- **הגנה מובנית מפני SQL Injection.** כשמרכיבים שאילתות SQL ידנית עם חיבור מחרוזות, קל בטעות ליצור פרצת אבטחה. ORM טוב מטפל בזה נכון כברירת מחדל.
- **פחות תלות במסד נתונים ספציפי.** אם כתבתם את הקוד מול ORM, מעבר בין PostgreSQL ל-MySQL הוא לרוב עניין של שינוי הגדרה אחת, לא שכתוב שאילתות.
- **שילוב טבעי עם שפת התכנות.** קל יותר לעבוד עם אובייקטים, בדיקות (tests) ולוגיקה עסקית כשהמידע ממסד הנתונים כבר מגיע כאובייקטים מוכרים, לא כשורות גולמיות.

## אז למה יש מתכנתים שממש לא אוהבים ORM

כאן מגיע החלק הכנה שהרבה מדריכים מדלגים עליו. ORM הוא לא פתרון קסם, ויש לו מחיר אמיתי.

- **אובדן שליטה על השאילתה בפועל.** לפעמים ה-ORM מייצר שאילתת SQL הרבה פחות יעילה ממה שהייתם כותבים ידנית, ואם לא יודעים לבדוק את זה, הביצועים סובלים בלי שרואים למה.
- **בעיית N+1 השאילתות.** זו המלכודת הכי מפורסמת ב-ORM - קוד שנראה תמים לגמרי בסגנון "עבור כל לקוח, תביא את ההזמנות שלו" יכול בטעות לייצר מאות שאילתות נפרדות במקום שאילתה אחת חכמה עם JOIN.
- **הבנה שטחית של SQL.** מתכנתים שסומכים על ORM בלבד, בלי להבין SQL שמאחוריו, נתקעים ברגע שצריך לבצע שאילתה מורכבת שה-ORM לא יודע לבטא בקלות.

## הכלל שאני נותן לתלמידים שלי

תלמדו SQL קודם, לעומק, ורק אחר כך תתחילו להשתמש ב-ORM. זה נשמע כמו סדר הפוך למה שאינטואיטיבי - למה ללמוד את הדרך "הארוכה" קודם - אבל זו בדיוק הדרך שהופכת אתכם למתכנת שיודע לזהות מתי ה-ORM עושה משהו לא יעיל, ולתקן את זה. מתכנת שלמד רק ORM בלי להבין מה קורה מתחתיו נתקע ברגע הראשון שמשהו לא עובד כמו שציפה.

ORM הוא כלי מצוין להאיץ פיתוח יומיומי, אבל הוא לא תחליף להבנה. הוא שכבת נוחות מעל ידע, לא מחליף אותו.

## ORM נפוצים שכדאי להכיר

אם אתם עובדים עם Node.js, Prisma ו-Sequelize הן דוגמאות נפוצות. בפייתון, SQLAlchemy ו-Django ORM הן הסטנדרט. בכל שפה אחרת יש מקבילות דומות, והעקרונות הבסיסיים זהים בכולן.
