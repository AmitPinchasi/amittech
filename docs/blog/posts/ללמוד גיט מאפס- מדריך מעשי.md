# ללמוד גיט מאפס - מדריך מעשי

גיט (Git) הוא לא שפת תכנות אלא כלי לניהול גרסאות קוד - הוא שומר היסטוריה של כל שינוי שעשיתם, ומאפשר לחזור אחורה, לעבוד עם שותפים, ולנסות דברים בלי לפחד "לשבור" את הפרויקט. זהו כלי שכל מתכנת, בכל שפה, חייב להכיר. הנה מדריך מעשי איך בפועל מתחילים ללמוד אותו מאפס.

## מה מתקינים לפני שמתחילים

מתקינים את גיט מהאתר הרשמי שלו, וזמין לכל מערכת הפעלה. מומלץ גם לפתוח חשבון ב-GitHub, שהוא לא גיט עצמו אלא שירות שמארח פרויקטי גיט באינטרנט ומאפשר לשתף אותם - ההבדל בין השניים מבלבל מתחילים רבים, אז כדאי לזכור: גיט הוא הכלי, GitHub הוא אחד המקומות שאפשר לאחסן בו את הפרויקט.

## תוכנית שבוע ראשון

- **ימים 1-2**: הקמת מאגר (repository) חדש עם git init, ופקודת git status - הפקודה שתריצו הכי הרבה בחיים שלכם עם גיט, כי היא מראה בדיוק מה השתנה ומה מצב הפרויקט.
- **ימים 3-4**: המחזור הבסיסי - git add (סימון קבצים לשמירה) ו-git commit (שמירת "תמונת מצב" עם הודעה). תתרגלו לעשות קומיטים קטנים ותכופים במקום קומיט ענק אחד בסוף היום.
- **ימים 5-7**: git log להצגת היסטוריה, ו-git diff לראיית ההבדלים המדויקים בין גרסאות. כאן מתחילים להרגיש למה גיט כל כך שימושי - אתם יכולים לראות בדיוק מה השתנה, מתי, ולמה.

## למה קומיטים קטנים חשובים

טעות נפוצה של מתחילים היא לעבוד שעות בלי לשמור, ואז לעשות קומיט ענק אחד עם הודעה כמו "עדכונים". זה מבטל את כל הערך של גיט - אם משהו נשבר, אין לכם דרך לדעת איזה שינוי ספציפי גרם לזה. קומיט טוב הוא יחידה לוגית קטנה - "הוספתי טופס התחברות", "תיקנתי באג בחישוב המחיר" - עם הודעה שמסבירה מה ולמה, לא רק "שינויים".

## ענפים (branches) - הכלי שהופך את גיט לעוצמתי

אחרי שהמחזור הבסיסי (add, commit, status) יושב טוב, השלב הבא הוא ענפים - היכולת לפתח פיצ'ר חדש בקו נפרד לגמרי מהקוד היציב, ולמזג (merge) אותו בחזרה רק כשהוא מוכן. git branch יוצר ענף חדש, git checkout או git switch עוברים בינו לבין הענף הראשי, ו-git merge משלב ענפים בחזרה. זה נשמע מסובך בהתחלה אבל זה בדיוק מה שמאפשר לעבוד על כמה דברים במקביל בלי שהם "יתנגשו" זה בזה.

## הפרויקט הראשון שכדאי לתרגל עליו

אל תלמדו גיט על פרויקט ריק. הדרך הכי טובה היא לקחת פרויקט קטן שכבר בניתם - למשל מחשבון או רשימת משימות משפה אחרת שלמדתם - ולהתחיל לנהל אותו בגיט מהתחלה: קומיט לכל שינוי משמעותי, ענף נפרד לכל תוסף חדש שאתם מנסים להוסיף. ברגע שתעשו זאת פעם אחת על קוד אמיתי שאכפת לכם ממנו, גיט יפסיק להרגיש מופשט.

## הטעויות הכי נפוצות של מתחילים בגיט

- **לפחד מ-merge conflicts ולהימנע מהם במקום לפתור**. קונפליקט קורה כשגיט לא מצליח לשלב אוטומטית שני שינויים סותרים באותה שורה. זה נשמע דרמטי אבל בפועל זה פשוט דורש שתחליטו ידנית איזו גרסה נכונה - זה חלק נורמלי לגמרי מעבודה עם גיט, לא סימן שמשהו השתבש.
- **לעבוד ישירות על הענף הראשי (main) בכל דבר**. גם כשעובדים לבד, הרגל של ענף נפרד לכל שינוי משמעותי חוסך כאבי ראש כשמתרגלים לעבודה עם צוות.
- **לשכוח קובץ gitignore**, מה שגורם לקבצים שלא צריכים להיות במאגר (כמו קבצי הגדרות אישיים או תיקיות זמניות) להיכנס בטעות לקומיטים.
- **לפחד מפקודות "מתקדמות" כמו git stash או git revert**. אלו לא כלים מסוכנים - הן דווקא רשת ביטחון שעוזרת לתקן טעויות בלי לאבד עבודה.
