# GraphQL מול REST - מה ההבדל ומתי לבחור בכל אחד

כמעט כל מפתח שמתחיל לעבוד עם API נתקל במתישהו בשאלה הזאת - GraphQL או REST. זה נשמע כמו עוד ויכוח טכנולוגי סתמי, אבל יש כאן הבדל אמיתי בגישה שכדאי להבין, לא רק כדי לענות נכון בראיון עבודה אלא כדי לבחור נכון כשבונים API בעצמכם.

## תזכורת קצרה - איך REST עובד

ב-REST כל משאב מקבל כתובת (endpoint) קבועה - `/users/5` מחזיר את המשתמש עם המזהה 5, `/users/5/posts` מחזיר את הפוסטים שלו. השרת מחליט מראש בדיוק אילו שדות חוזרים בכל תשובה. אם כתבתם REST API בעצמכם, אתם מכירים את הדפוס הזה - הוא פשוט וצפוי, וזו נקודת ההשוואה הטבעית להבין ממנה מה GraphQL בעצם משנה.

## אז מה זה בעצם GraphQL

GraphQL הופך את הגישה על הראש. במקום שהשרת יחליט מה חוזר בכל תשובה, הלקוח שולח שאילתה שמתארת בדיוק אילו שדות הוא צריך, והשרת מחזיר בדיוק את זה - לא יותר ולא פחות. יש רק endpoint אחד בדרך כלל, וכל הבקשות עוברות דרכו בפורמט שאילתה מיוחד.

לדוגמה, אם אתם רוצים רק את השם והאימייל של משתמש, אתם שולחים שאילתה שמבקשת בדיוק את שני השדות האלה, ומקבלים בחזרה בדיוק אובייקט עם שני השדות, בלי כל שאר המידע שיש למשתמש הזה במסד הנתונים.

## ההבדל המרכזי - מי שולט בצורת התשובה

זה באמת הלב של כל הסיפור. ב-REST, השרת קובע. אם אתם צריכים רק שדה אחד מתוך עשרה, אתם עדיין מקבלים את כל העשרה (מצב שנקרא over-fetching), ואם אתם צריכים מידע שמפוזר בין כמה משאבים, אתם צריכים לבצע כמה קריאות נפרדות (מצב שנקרא under-fetching). ב-GraphQL, הלקוח קובע. הוא מבקש בדיוק מה שהוא צריך, ממקור אחד, בקריאה אחת.

## היתרונות האמיתיים של GraphQL

- **בדיוק המידע הנדרש, לא יותר.** באפליקציות מובייל, שבהן כל בייט ברשת חשוב, זה יתרון ממשי.
- **קריאה אחת במקום כמה.** אם מסך אחד צריך מידע ממשתמש, מההזמנות שלו ומהעדפותיו, זו שאילתה אחת ולא שלוש קריאות נפרדות.
- **פחות תלות בגרסאות שרת.** אפשר להוסיף שדות חדשים לסכימה בלי לשבור לקוחות קיימים, כי כל לקוח מבקש רק מה שהוא צריך ממילא.

## והחסרונות שפחות מדברים עליהם

- **מורכבות בצד השרת.** בניית סכימת GraphQL, ניהול הרשאות ברמת שדה בודד, ומניעת שאילתות כבדות מדי - כל אלה דורשים תשתית שלא קיימת ב-REST הפשוט.
- **קאשינג הופך מסובך.** ב-REST, קאשינג ברמת HTTP הוא כמעט חינם - כל endpoint הוא כתובת קבועה. ב-GraphQL, בגלל שכל שאילתה שונה, צריך פתרונות קאשינג מיוחדים.
- **עקומת למידה.** צריך ללמוד שפת שאילתות חדשה, ולפעמים זה overkill לצוות קטן שרק צריך API פשוט שעובד.

## מתי לבחור מה בפועל

אם אתם בונים API פשוט עם משאבים ברורים, ובעיקר עובדים מול קליינט אחד או שניים - REST כנראה מספיק, פשוט יותר לתחזק, ויש עליו תיעוד ותשתית עצומה בכל שפה. אם יש לכם כמה סוגי לקוחות שונים (אפליקציית מובייל, אתר, מכשיר IoT) שכל אחד צריך צירוף שונה של נתונים מאותם משאבים, GraphQL יכול לחסוך המון קריאות מיותרות ולפשט את העבודה של צוותי הפרונט.

חשוב לומר - זו לא בחירה של "כל הפרויקט" תמיד. הרבה חברות משתמשות ב-REST לרוב המערכת, ומוסיפות GraphQL רק בשכבה שבאמת סובלת מ-over-fetching, כמו האפליקציה הניידת.

## לסיכום

REST ו-GraphQL לא מתחרים על "מי טוב יותר" באופן מוחלט - הם פשוט פותרים בעיה דומה בגישה שונה לגמרי. REST נותן שליטה לשרת ופשטות, GraphQL נותן שליטה ללקוח וגמישות. תבינו את הצרכים האמיתיים של הפרויקט שלכם, ותבחרו בהתאם - לא כי שמעתם שמישהו אחר משתמש בזה.
