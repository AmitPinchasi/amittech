Claude Code הוא לא סוכן הטרמינל היחיד בעולם. תוך זמן קצר אחרי שקטגוריית "סוכני CLI" הפכה לזרם מרכזי בפיתוח תוכנה, כמעט כל חברת AI גדולה השיקה כלי מקביל, ולצדם צמחו גם כלים בקוד פתוח שמקדימים אפילו את הגל הזה. בהרצאה הזו נסקור את השחקנים המרכזיים - OpenAI Codex CLI, Gemini CLI של Google, ו-aider - נבין איך הם דומים ואיך הם שונים מ-Claude Code, ולמה שווה להכיר יותר מכלי אחד גם אם Claude Code הוא הכלי המרכזי שלכם.

## למה בכלל יש כמה סוכני CLI - the shape of the category

כל סוכני ה-CLI פותרים את אותה בעיה בסיסית: לתת למודל שפה גישה ישירה למערכת הקבצים ולמעטפת, כדי שיוכל לקרוא קוד, לכתוב קוד, ולהריץ פקודות - קומפילציה, טסטים, git - בתוך לולאה אוטונומית. הארכיטקטורה הכללית דומה בכולם: קלט משפה טבעית, תכנון פנימי, שימוש בכלים (קריאת קובץ, כתיבת קובץ, הרצת פקודה), ולולאה שחוזרת על עצמה עד שהמשימה הושלמה או שהמודל עוצר לבקש אישור.

מה שמשתנה בין הכלים הוא: **איזה מודל מריצים מתחת למכסה המנוע**, **איך מנוהלות ההרשאות וההרצה**, **עד כמה הכלי פתוח להרחבה**, **מודל התמחור**, ו**איכות ההתנהגות בפועל** על משימות תכנות אמיתיות. אלה בדיוק הצירים שנשווה בהם.

```text
משותף לכל סוכני ה-CLI:
קלט טבעי -> תכנון -> שימוש בכלים (קריאה/כתיבה/הרצה) -> לולאה עד סיום

מה שמשתנה:
מודל מתחת למכסה המנוע | מדיניות הרשאות | פתיחות להרחבה | תמחור | בשלות בפועל
```

## OpenAI Codex CLI

Codex CLI הוא סוכן הטרמינל של OpenAI, מריץ מודלים מותאמים למשימות תכנות. מבחינת חוויית שימוש הוא מזכיר את הזרימה שלמדתם ב-Claude Code: מתקינים, מריצים פקודה בתיקיית הפרויקט, נותנים משימה בשפה טבעית, והסוכן קורא קבצים, מציע שינויים ומריץ פקודות.

נקודות שווה להכיר:
- יש לו מדיניות אישורים (approvals) שדומה במהות למודל ההרשאות של Claude Code שלמדנו בפרק 2.3 - אפשר לבחור רמת אוטונומיה, מהצבעה על כל פעולה ועד הרצה חופשית יחסית בתוך סביבה מבודדת.
- יש אפשרות להרצה בסביבת ארגז חול (sandbox) מבודדת, כדי להגביל את הנזק האפשרי אם הסוכן טועה או מקבל הוראה עוינת.
- זהו כלי שמפותח על ידי אותה חברה שמפתחת את מודלי GPT, כך שבדרך כלל הוא מקבל גישה מוקדמת לשיפורים במודלים המתמחים בקוד מבית OpenAI.

מבחינת חוויית עבודה, ההבדלים מ-Claude Code הם בעיקר בפרטי המימוש של ניהול ההרשאות, באיכות ובסגנון של המודל שמריץ אותו, ובמידת הבשלות של מנגנוני ההרחבה (כמו MCP, מיומנויות מותאמות, וסוכני משנה) שלמדנו בפרקים 4 ו-6 של הקורס הזה.

מבחינה ארגונית, צוותים שכבר משתמשים בכלים אחרים של OpenAI (כמו ChatGPT Enterprise) לפעמים מוצאים ש-Codex CLI משתלב טבעי יותר בתהליכי האישור והרכש הקיימים אצלם, פשוט כי הוא מגיע מאותו ספק שכבר עבר בדיקות אבטחה וחוזה ארגוני. זה שיקול פרקטי אמיתי שלא תמיד קשור לאיכות הכלי עצמו.

## Gemini CLI

Gemini CLI הוא סוכן הטרמינל של Google, מריץ את משפחת מודלי Gemini. הוא בקוד פתוח, מה שמאפשר לקהילה לתרום, לבדוק ולהרחיב אותו ישירות, וזה יתרון ממשי מבחינת שקיפות לעומת כלים סגורים.

נקודות שווה להכיר:
- אינטגרציה טבעית עם מוצרי Google - כלים שמריצים משימות שנוגעות בענן של Google, או שרוצים להשתמש בהקשר הרחב שיש למודלי Gemini (למשל חלון הקשר ארוך מאוד), ימצאו כאן זרימה נוחה.
- לרוב יש שכבת שימוש חינמית נדיבה יחסית, מה שהופך אותו לנקודת כניסה נגישה למי שרוצה להתנסות בסוכן CLI בלי מחויבות כספית.
- בהיותו קוד פתוח, קל יותר להבין בדיוק מה קורה "מתחת למכסה המנוע" - איך נבנה הפרומפט למערכת, איך מנוהלת הרשאה, ואיך ממומשים הכלים.
- **חלון הקשר גדול במיוחד** - מודלי Gemini ידועים בחלונות הקשר ארוכים יחסית לשוק, מה שיכול להיות יתרון אמיתי כשעובדים על repository ענק שצריך "להכניס" ממנו הרבה קוד בבת אחת לתוך שיחה אחת, בלי להסתמך רק על חיפוש ממוקד.

## aider

aider שונה במהות מהשניים הקודמים - הוא לא מוצר של חברת מודל גדולה, אלא כלי קוד פתוח, ותיק יחסית בקטגוריה (קדם בהרבה את הגל הנוכחי של סוכנים "מלאים"), שמתמקד בזוגיות תכנות - pair programming - מונחית git.

מה שמייחד את aider:
- **אגנוסטי למודל** - במקום להיות קשור לספק אחד, aider מאפשר לחבר כל מודל שיש לו API, כולל מודלי Claude, מודלי OpenAI, ואפילו מודלים מקומיים. זה הופך אותו לכלי גמיש למי שרוצה לבחור מודל לפי משימה או עלות, בלי להתחייב לספק יחיד.
- **מרכזיות ה-diff וה-git** - aider בנוי סביב הרעיון שכל שינוי מוצג כ-diff ברור, וכל שינוי נשמר כקומיט נפרד ב-git עם הודעת קומיט אוטומטית. זה נותן שקיפות גבוהה ומעקב היסטורי טבעי - קל לחזור אחורה לכל שלב.
- **פחות אוטונומי מטבעו** - aider נוטה להיות מדויק וממוקד יותר בעריכות מבוקשות, ופחות "משוחרר" לרוץ עצמאית על משימות ארוכות עם הרצת פקודות מרובות בלי מעורבות. זה יתרון כשרוצים שליטה הדוקה, וחיסרון כשרוצים אוטונומיה מלאה על משימה ארוכה.
- **קליל ומהיר להתקנה** - כלי פייתון פשוט להתקנה, בלי תלות בתשתית כבדה, מה שהופך אותו לפופולרי גם בסביבות מינימליסטיות.

```bash
# דוגמה טיפוסית לזרימת עבודה עם aider
aider --model claude-sonnet-5
> תוסיף ולידציה לשדה האימייל בטופס ההרשמה
# aider מציג diff, מבקש אישור, ומייצר קומיט git אוטומטי
```

## מעבר בין כלים - what actually transfers

חדשות טובות: רוב המיומנויות שלמדתם עד כה בקורס הזה **לא ספציפיות ל-Claude Code**. פירוק משימה לצעדים ברורים, כתיבת פרומפט מדויק עם הקשר מספיק (פרק 2), עבודה במצב תכנון לפני ביצוע (פרק 2.4), אימות תוצאה במקום קבלה עיוורת (פרק 3.6), וניהול קובץ הקשר קבוע לפרויקט (פרק 2.5) - כל אלה עקרונות כלליים שעובדים מול כל סוכן CLI, כי הם נובעים מהאופי הבסיסי של עבודה עם מודל שפה, לא ממימוש ספציפי.

מה שכן משתנה בין הכלים ודורש התאמה:

- **תחביר הפקודה ודגלי ההרצה.** כל כלי מותקן ומופעל קצת אחרת, ויש ללמוד את הפקודות הבסיסיות שלו (התקנה, הפעלה, בחירת מודל, הגדרת רמת אוטונומיה).
- **מיקום ופורמט קובץ ההקשר.** מה ש-CLAUDE.md עושה ל-Claude Code, יש לו מקבילות בכלים אחרים (לפעמים בשם אחר, לפעמים בפורמט אחר) - חשוב לבדוק את התיעוד הרשמי של כל כלי כדי למצוא את המקבילה.
- **מנגנון האישורים הספציפי.** לכל כלי יש דרך משלו לבקש אישור לפני פעולה מסוכנת (הרצת פקודה, מחיקת קובץ) - חשוב להבין אותה לפני שסומכים על הכלי במשימה רגישה.
- **איכות והתנהגות המודל.** מודלים שונים "מרגישים" אחרת בעבודה בפועל - חלקם נוטים ליזום יותר בדיקות עצמאיות, חלקם שמרנים יותר לגבי שינויים גדולים, חלקם מדויקים יותר בעקיבה אחר הוראות ארוכות ומורכבות.

הלקח המרכזי: השקעה בלימוד עומק של סוכן CLI אחד (Claude Code, בקורס הזה) היא השקעה שמניבה תשואה גם בכלים אחרים, כי רוב מה שלמדתם הוא ברמת החשיבה, לא ברמת התחביר.

## איך הכלים שונים בפועל - real differences that matter

מעבר לטבלה, ההבדלים שבאמת משפיעים על עבודה יומיומית הם:

- **איכות המודל למשימה הספציפית.** מודלים שונים מצטיינים בדברים שונים - יש הבדלים בין מודלים ביכולת דיבוג, בעקביות על פני קבצים רבים, ובנטייה "להמציא" פרטים לא נכונים (הזיות). שווה לבדוק בפועל על המשימות האופייניות לכם, לא להסתמך על מוניטין כללי.
- **מנגנוני הרחבה.** תמיכה ב-MCP (פרק 6.1), במיומנויות מותאמות, בהוקים ובסוכני משנה - אלה משתנים מאוד בין הכלים, ומשפיעים ישירות על כמה אפשר "לאלף" את הסוכן לצרכי הפרויקט הספציפי שלכם.
- **מודל התמחור.** יש כלים שעובדים על מנוי חודשי עם מכסת שימוש, ויש כאלה שגובים לפי צריכת API בפועל (טוקנים). ההבדל יכול להיות משמעותי בפרויקטים גדולים או בשימוש כבד.
- **מדיניות הרשאות והרצה.** כמה קל להריץ בארגז חול מבודד, כמה גרנולרית השליטה על אילו פקודות מותר להריץ בלי אישור - נקודה קריטית לעבודה בסביבות רגישות (ראו פרק 8.2 בהמשך הקורס).
- **בשלות ותמיכה קהילתית.** כלים בקוד פתוח נהנים מתרומות ותיקונים מהירים מהקהילה, אבל לפעמים חסרים תמיכה רשמית ברמת ארגון; כלים סגורים בדרך כלל מציעים SLA ותמיכה מסחרית.

## למה כדאי להכיר יותר מכלי אחד - why more than one matters

בקורס הזה Claude Code הוא הכלי המרכזי, ומסיבה טובה - הוא הבשל, העשיר בהרחבות, והכי מתאים ללימוד עומק של פיתוח אג'נטי. אבל יש כמה סיבות מעשיות ממש להכיר גם סוכנים אחרים:

- **גיוון מודלים לפי משימה ועלות.** לפעמים משימה פשוטה וזולה (תיקון טיפוס, כתיבת טסט שגרתי) לא מצדיקה את המודל היקר ביותר, ומודל זול יותר דרך כלי אחר עושה את העבודה מצוין.
- **חוסן מול תקלות והגבלות.** אם יש הפסקת שירות זמנית, הגבלת קצב (rate limit), או תקלה אצל ספק אחד, כלי חלופי מאפשר להמשיך לעבוד בלי לעצור.
- **דרישות ארגוניות.** חלק מהארגונים מחייבים ספק ספציפי מסיבות רגולטוריות, חוזיות או אבטחתיות. מי שמכיר רק כלי אחד "נתקע" כשעובר ארגון.
- **בדיקת שפיות - sanity check.** על משימות קריטיות, שווה לבקש מכלי או מודל שני "חוות דעת שנייה" על אותה בעיה, ולהשוות תשובות. אם שני מודלים שונים מגיעים לאותה מסקנה, זו אינדיקציה חיובית; אם הם חלוקים, זה סימן לחקור לעומק לפני שממשיכים.
- **הבנה עמוקה יותר של הקטגוריה עצמה.** ראיית כמה מימושים שונים לאותו רעיון - סוכן שקורא, כותב ומריץ - מחדדת הבנה של מה חשוב ומה לא ברמת העיצוב הכללי, לא רק ברמת הכלי הספציפי. זו בדיוק אותה מיומנות שנרחיב עליה בהרצאה 5.6, שם נלמד לבחור ולשלב כלים באופן שיטתי.

הכלל המעשי: השקיעו את רוב הזמן שלכם בלימוד עומק של כלי אחד (Claude Code, במקרה של הקורס הזה), אבל הקדישו זמן להתנסות קצרה עם לפחות עוד סוכן CLI אחד, כדי לפתח תחושה למה משותף לכל הקטגוריה ומה ספציפי לכלי אחד.

דרך פשוטה להתחיל: קחו משימה קטנה וברורה שכבר פתרתם בעבר עם Claude Code - למשל תיקון באג ספציפי או הוספת טסט - ונסו לפתור משימה דומה (לא זהה, כדי לא "לזכור" את הפתרון) עם כלי אחר. שימו לב להבדלים באיכות התכנון, בדיוק ההוראות שהכלי מבקש מכם, ובאופן שבו הוא מציג את השינויים המוצעים. תרגיל כזה, שעות ספורות, שווה יותר מקריאת כל השוואה תיאורטית.

## סיכום

- כל סוכני ה-CLI חולקים ארכיטקטורה דומה: קלט טבעי, תכנון, שימוש בכלים (קריאה/כתיבה/הרצה), ולולאה עד השלמת המשימה.
- **OpenAI Codex CLI** מריץ מודלי GPT, עם מדיניות אישורים והרצה בארגז חול, בדומה במהות למודל ההרשאות של Claude Code.
- **Gemini CLI** מריץ מודלי Gemini, קוד פתוח, עם שכבה חינמית נדיבה ואינטגרציה טבעית לעולם Google.
- **aider** הוא כלי קוד פתוח וותיק, אגנוסטי למודל (מריץ כל מודל עם API, כולל Claude), ממוקד עריכה מדויקת עם diff וקומיטים אוטומטיים, ואוטונומיה מתונה יותר מסוכן "מלא".
- ההבדלים המרכזיים בין הכלים: איכות המודל למשימה, עושר מנגנוני ההרחבה, מודל התמחור, מדיניות ההרשאות, ובשלות התמיכה.
- כדאי להכיר יותר מכלי אחד כדי לגוון עלות ומודל, לשמור על חוסן מול תקלות, לעמוד בדרישות ארגוניות, ולקבל חוות דעת שנייה במשימות קריטיות.
- הכלל המעשי: להעמיק בכלי מרכזי אחד, אבל להישאר בקיאים בקטגוריה כולה - זה בדיוק ברוח "לא מתחתנים עם כלי אחד" שמנחה את הפרק הזה.
