בחמש ההרצאות הקודמות סקרנו חמש קטגוריות שונות של כלים: עורך אג'נטי, השלמת קוד, סוכני CLI חלופיים, מחוללי UI, וסקירת קוד אוטומטית. בכל אחת מהן ראינו שהכלי הספציפי - Cursor, Copilot, Codex CLI, v0, CodeRabbit - הוא לא "מתחרה" ב-Claude Code, אלא פותר בעיה אחרת לגמרי, לפעמים בשלב אחר בתהליך, לפעמים בסוג עבודה שונה לחלוטין. ההרצאה הזו סוגרת את הפרק עם מה שבאמת חשוב: איך בוחרים את הכלי הנכון לכל משימה, ואיך בונים זרימת עבודה שמשלבת כמה כלים בלי לאבד שליטה.

---

## למה לא מתחתנים עם כלי אחד - the case against monogamy

הפיתוי הטבעי הוא לבחור כלי אחד, ללמוד אותו לעומק, ולנסות לכופף כל משימה כדי שתתאים לו. זו טעות מכמה סיבות:

- **כל כלי מותאם לצורת בעיה מסוימת.** Copilot מצוין בהשלמת שורה תוך כדי הקלדה, אבל נסיון לבצע דרכו רפקטור של 20 קבצים יהיה איטי ומתסכל בהשוואה לסוכן שתוכנן בדיוק למשימה כזו. v0 מצוין בטיוטת UI ראשונית, אבל חסר את העומק להטמעת לוגיקה עסקית מורכבת.
- **מחיר ההתאמה בכוח גבוה מהמעבר.** הזמן שמושקע ב"לשכנע" כלי לא מתאים לעשות עבודה שהוא לא תוכנן אליה, לרוב יקר יותר מהזמן שלוקח לפתוח כלי אחר, מתאים יותר, ולעבור אליו.
- **הנוף משתנה מהר.** כלי שהיה המוביל הברור לפני שנה יכול להיות מוחלף היום בכלי חדש עם יכולות טובות יותר. מי שבנה זהות מקצועית סביב כלי ספציפי, במקום סביב הכישורים הכלליים (פרומפט טוב, אימות, פירוק משימה), נתקע מאחור כשהנוף זז.
- **כלים שונים מספקים בדיקת שפיות זה לזה.** ראינו בהרצאה 5.3 שחוות דעת שנייה ממודל או כלי אחר יכולה לחשוף טעויות שכלי בודד היה מפספס.

המסקנה: המיומנות שבאמת שווה לפתח היא **לזהות צורת משימה ולבחור כלי מתאים**, לא לשנן תפריטים של כלי ספציפי אחד.

---

## הצירים לבחירת כלי - the decision axes

לפני שבוחרים כלי, שווה לשאול כמה שאלות שממפות את צורת המשימה:

- **מה גודל וטווח השינוי?** תיקון שורה בודדת מול פיצ'ר שנוגע בעשרות קבצים.
- **כמה אוטונומיה רוצים לתת?** שליטה צמודה על כל שורה, מול "תן לזה לרוץ ותחזור עם תוצאה".
- **מה סביבת העבודה?** GUI מקומי מול טרמינל מרוחק/headless מול פלטפורמת git בענן.
- **מה שלב התהליך?** כתיבה ראשונית, עריכה תוך כדי הקלדה, בנייה מאפס, או ביקורת אחרי שהקוד כבר נכתב.
- **כמה קריטי הדיוק?** קוד תשתיתי או אבטחתי דורש זהירות ואימות כפול; פרוטוטייפ חד-פעמי סובל טעויות בקלות רבה יותר.
- **מה מגבלות הארגון?** דרישות ספק, תקציב, מדיניות אבטחה שמגבילה אילו כלים מותר להריץ על קוד רגיש.

שאלות אלה, לא נאמנות לכלי, הן מה שצריך לקבוע כל החלטה.

## טבלת החלטה - a decision table

| סוג המשימה | כלי מומלץ | למה |
|---|---|---|
| שאלה מהירה / הסבר קוד | צ'אט (Claude.ai, Copilot Chat) | לא דורש הקשר פרויקט מלא, תשובה מיידית |
| למידת codebase לא מוכר | סוכן CLI (Claude Code) | חקירה יזומה עמוקה, grep, קריאת קבצים לרוחב הפרויקט |
| הקלדה שוטפת, קוד חוזר | Copilot / Cursor Tab | השלמה מיידית תוך כדי הקלדה, בלי לעצור ולנסח בקשה |
| פיצ'ר ממוקד בכמה קבצים | Claude Code או Cursor Composer | תכנון, ביצוע רב-קבצי, הרצת בדיקות ותיקון עצמי |
| עריכה נקודתית בקובץ בודד | עריכה ממוקדת ב-Cursor / inline edit | diff מהיר וממוקד, בלי צורך בסוכן מלא |
| טיוטת UI/פרוטוטייפ מאפס | v0 / bolt.new | תצוגה מקדימה חיה תוך שניות, בלי סביבה מקומית |
| משימה אוטונומית ארוכה, headless | סוכן CLI (Claude Code, Codex CLI) | עבודה בלי GUI, הרצה ברקע, שילוב ב-CI/CD |
| דיבוג באג עמוק | סוכן CLI עם דיבוג שיטתי | חקירה, השערות, ריצות בדיקה איטרטיביות |
| רפקטור גדול על פני הפרויקט | סוכן CLI עם סוכני משנה | פיצול עבודה למקביל, שליטה מפורשת בהקשר |
| ביקורת PR | סקירה אוטומטית (CodeRabbit) + reviewer אנושי | מסנן ראשון מהיר לפני זמן אדם יקר |
| חוות דעת שנייה על פתרון קריטי | מודל/כלי שונה מזה שכתב את הפתרון | בדיקת שפיות בלתי תלויה |

הטבלה הזו היא נקודת פתיחה, לא חוק ברזל - לצוותים שונים, פרויקטים שונים ותקציבים שונים יהיו וריאציות. אבל הצורה הכללית - להתאים כלי לצורת משימה - נכונה תמיד.

## דוגמה: מה קורה כשבוחרים כלי לא נכון - a worked mistake

כדי להמחיש את המחיר של בחירה לא מתאימה, שווה לראות דוגמה קונקרטית. נניח שמפתח מקבל משימה: "לשנות את מבנה ה-authentication בפרויקט, מ-session מבוסס cookie למנגנון מבוסס JWT, כולל עדכון של כל ה-endpoints, ה-middleware, וטסטי האינטגרציה".

**הגישה הלא נכונה**: פתיחת Copilot Chat וניסיון לתאר את כל המשימה בהודעה אחת ארוכה, בתקווה שהכלי "יבין" ויבצע את כל השינוי. הבעיה: Copilot Chat לא בנוי לתכנן ולבצע משימה רב-שלבית שנוגעת בעשרות קבצים באופן עצמאי - הוא ייתן הצעות טובות לקובץ ספציפי בכל פעם, אבל המפתח יצטרך לנווט ידנית בין כל קובץ, להדביק הקשר שוב ושוב, ולוודא בעצמו עקביות בין הקבצים. זו עבודה מייגעת שמבזבזת בדיוק את הזמן שה-AI אמור לחסוך.

**הגישה הנכונה**: פתיחת Claude Code, ומתן המשימה כמכלול - הסוכן מתכנן את כל השינוי (אולי אף במצב תכנון ייעודי, כפי שלמדנו בפרק 2.4), מזהה את כל הקבצים הרלוונטים בעצמו דרך חיפוש בפרויקט, מבצע את השינוי בעקביות על פני כל הקבצים, ומריץ את סוויטת הטסטים כדי לוודא שהמעבר לא שבר דבר. Copilot עדיין יכול להיות שימושי בשוליים - להאיץ הקלדה של קוד עזר קטן תוך כדי סקירת השינויים - אבל לא כמנוע המרכזי של המשימה.

ההבדל בין שתי הגישות הוא לא "כלי טוב מול כלי גרוע" - הוא **התאמת כלי לצורת המשימה**, בדיוק העיקרון שההרצאה הזו בנויה סביבו.

## דוגמת שרשרת כלים משולבת - a combined chain

כדי להמחיש שילוב אמיתי, הנה זרימה שמחברת כמעט את כל מה שלמדנו בפרק הזה, על פרויקט לדוגמה - הוספת מסך הגדרות חדש לאפליקציית web:

```text
1. v0        - טיוטת UI ראשונית ממה שמתואר בפרומפט, תוך דקות
2. ייצוא     - הקוד עובר ל-repository האמיתי של הפרויקט
3. Claude Code - מתאים את הקוד למוסכמות הפרויקט, מחבר ללוגיקה
                 עסקית אמיתית, מוסיף טסטים, מריץ ומאמת
4. Cursor    - במהלך עבודה על הקוד, עריכות קטנות-בינוניות
                 נעשות בעורך גרפי עם Tab להאצת הקלדה
5. Copilot   - (אם מותקן לצד Cursor) מאיץ עוד יותר הקלדה שוטפת
                 של קוד תבניתי
6. git push  - פתיחת pull request
7. CodeRabbit - סקירה אוטומטית, תופסת בעיות מכניות ודפוסיות
8. reviewer אנושי - מתמקד בשאלות ארכיטקטורה ומוצר, מאשר ומ merge
```

שרשרת כזו לא נבנית במקרה - היא תוצאה של הבנה מה כל כלי עושה טוב, והרכבה שלהם בסדר שהופך את החולשה של כל שלב לחוזק של השלב הבא. הטיוטה המהירה אבל הרדודה של v0 מתחזקת על ידי סוכן מלא; ההקלדה השוטפת מואצת בלי לפגוע באיכות התכנון; והביקורת מפוצלת לשכבה מכנית מהירה ושכבה אנושית ממוקדת.

## סימני אזהרה לבחירת כלי לא נכונה - warning signs

כמה סימנים שמעידים שהכלי הנוכחי לא מתאים למשימה, ושווה לעצור ולהחליף:

- **מוצאים את עצמכם מנסים "לשכנע" כלי הקלדה לעשות משימה רב-שלבית** - למשל לכתוב פרומפט ענק בתוך Copilot Chat כדי לגרום לו לבצע רפקטור על פני קבצים רבים. זה סימן שצריך סוכן CLI מלא.
- **מריצים סוכן אוטונומי כבד למשימה של שורה אחת** - הפעלת מצב סוכן מלא כדי לתקן typo היא בזבוז זמן ועלות; עדיף עריכה ממוקדת ידנית או Tab.
- **מוצאים באגים דומים חוזרים ב-PR-ים אחרי שסקירה אוטומטית כבר "אישרה"** - סימן שהכלי מוגדר בצורה לא מספיק מותאמת לפרויקט, שווה להשקיע בקובץ התצורה שלו.
- **מרגישים תלות מלאה בכלי יחיד בלי אלטרנטיבה** - אם הפסקת שירות של ספק אחד עוצרת את כל הצוות, זה סימן לגיוון כלים לפני שזה קורה, לא אחרי.

## בחירת כלים ברמת צוות מול ברמת פרט - team-level vs individual choice

עד כה דנו בבחירה מנקודת מבט של מפתח בודד שמתאים כלי למשימה נקודתית. אבל בסביבה ארגונית יש שכבה נוספת: איזה כלים הצוות כולו מאמץ באופן רשמי, ולמה זה לא זהה להחלטה האישית.

יש יתרון אמיתי בסטנדרטיזציה ברמת צוות: כשכולם עובדים עם אותו סוכן CLI מרכזי, קל יותר לשתף קובצי הקשר (כמו CLAUDE.md), מיומנויות מותאמות, והוקים משותפים (פרקים 4 ו-6) - כל אלה נבנים פעם אחת ומשרתים את כל הצוות. אימוץ מבוזר לגמרי, שבו כל מפתח בוחר כלי לפי טעם אישי, מאבד את היתרון הזה ומקשה על שיתוף ידע והרגלים.

מצד שני, כלים "משלימים" שלא נוגעים בליבת הקוד המשותף - כמו העדפה אישית בין Cursor ל-VS Code רגיל עם Copilot, או שימוש אישי ב-v0 לטיוטות UI - יכולים להישאר החלטה אישית לגמרי, כי הם לא יוצרים תלות בין חברי הצוות. הכלל הפרקטי: לסטנדרט ברמת צוות את הכלים שיוצרים תשתית משותפת (הסוכן המרכזי, קובצי ההקשר, כלי סקירת הקוד האוטומטית שרץ על כל PR), ולהשאיר גמישות אישית בכלים שמשפיעים בעיקר על זרימת העבודה האישית של כל מפתח.

## שינוי כלי באמצע משימה - switching mid-task

לפעמים מגלים באמצע עבודה שהכלי שבחרתם לא מתאים - התחלתם עם עריכה ממוקדת קטנה ב-Cursor, וגיליתם שהיא מתפרשת לכדי שינוי רב-קבצי משמעותי. אין שום בעיה לעצור ולעבור כלי באמצע - ההשקעה שכבר נעשתה (הבנת הבעיה, החלטות שכבר התקבלו) לא הולכת לאיבוד, כי היא נמצאת אצלכם, לא נעולה בתוך הכלי.

הדרך הפשוטה ביותר: לסכם במשפט או שניים את מה שכבר הובן ומה שכבר הוחלט, ולהעביר את הסיכום הזה כהקשר פתיחה לכלי החדש. זה בדיוק אותו עיקרון שלמדנו בפרק 4.1 על ניהול הקשר - לדעת מתי "לאפס" ולהתחיל שיחה חדשה עם תמצית ברורה, במקום לגרור הקשר תפוח לאורך זמן. המעבר בין כלים הוא רק מקרה פרטי של אותו עיקרון.

---

## עלות כשיקול בבחירה - cost as a factor

מעבר ליכולת הטכנית, שווה לשקול גם עלות בבחירת כלי למשימה. רוב הכלים בפרק הזה מציעים שכבת ניסיון חינמית ומודל תשלום שמבוסס מנוי או צריכת API בפועל. משימה קטנה ושגרתית לא מצדיקה תמיד את המודל היקר ביותר או את הכלי הכי "כבד" - לפעמים Tab-completion חינמי או זול עושה בדיוק את מה שצריך. משימה קריטית וגדולה, לעומת זאת, מצדיקה השקעה בכלי המתאים ביותר, גם אם הוא יקר יותר, כי עלות הטעות גבוהה בהרבה מעלות הכלי. כלל אצבע פשוט: להתאים את גודל ההשקעה לגודל הסיכון והתועלת של המשימה, לא לבחור תמיד את הכלי הזול ביותר או תמיד את החזק ביותר.

## סיכום

- אף כלי בפרק הזה לא "המנצח" הכללי - כל אחד פותר צורת בעיה אחרת: עורך אג'נטי (Cursor), השלמת קוד (Copilot), סוכני CLI חלופיים, מחוללי UI (v0, bolt), וסקירת קוד אוטומטית (CodeRabbit).
- הבחירה הנכונה נגזרת מצירי החלטה: גודל וטווח השינוי, רמת האוטונומיה הרצויה, סביבת העבודה, שלב התהליך, קריטיות הדיוק, ומגבלות ארגוניות.
- טבלת ההחלטה שבנינו נותנת נקודת פתיחה מעשית: התאמת כלי למשימה ספציפית, מ"שאלה מהירה" ועד "רפקטור גדול על פני הפרויקט".
- שרשרת כלים משולבת - למשל v0 לטיוטה, Claude Code לאינטגרציה והקשחה, Cursor/Copilot להקלדה שוטפת, CodeRabbit לביקורת ראשונית - ממנפת את החוזק של כל שלב ומכסה את החולשות של הקודם לו.
- סימני אזהרה לבחירה לא נכונה: "לשכנע" כלי קל לבצע משימה כבדה, להפעיל כלי כבד למשימה זעירה, בעיות חוזרות למרות אישור אוטומטי, ותלות מלאה בכלי יחיד.
- העיקרון המרכזי שחוצה את כל הפרק: המיומנות היקרה באמת היא לדעת לזהות צורת משימה ולבחור כלי בהתאם - לא נאמנות לכלי אחד. הכישורים שלמדתם בפרקים 1-4 (פרומפט טוב, אימות, ניהול הקשר, פירוק משימה) עוברים לכל כלי, וזה מה שהופך אתכם למקצוענים בתחום, לא הידע הטכני הספציפי בכלי בודד.
