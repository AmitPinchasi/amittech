עד עכשיו הקורס בנוי סביב Claude Code - סוכן שרץ בטרמינל, בלי ממשק גרפי, ומדבר איתנו בטקסט. זו לא הדרך היחידה לעבוד עם AI אג'נטי. חלק גדול מהתעשייה עובד מתוך עורך קוד שבו ה-AI משולב עמוק בתוך חוויית העריכה עצמה, לא כתוסף חיצוני אלא כחלק מהארכיטקטורה של העורך. הכלי המוביל בקטגוריה הזו הוא Cursor, ובהרצאה הזו נבין מה הוא בעצם, במה הוא שונה מסוכן טרמינל, ומתי כדאי לפתוח אותו במקום להישאר בטרמינל.

## מה זה Cursor - the agentic IDE

Cursor הוא עורך קוד עצמאי, בנוי כ-fork של VS Code. המשמעות: זה לא תוסף שמתקינים על VS Code קיים, אלא אפליקציה נפרדת שיורשת את כל מה שמוכר מ-VS Code - ערכות נושא, תוספים, קיצורי מקלדת, טרמינל מובנה - ומוסיפה עליו שכבת AI שנארגת עמוק בתוך העורך עצמו, ולא רק בפאנל צד.

ההבדל הזה קריטי. תוסף AI רגיל רואה בעיקר את מה שהמשתמש בוחר להדביק לתוכו. Cursor, לעומת זאת, בנוי מהיסוד כדי לדעת מה קורה בפרויקט: אילו קבצים פתוחים, מה נערך לאחרונה, מה מבנה התיקיות, ולפעמים גם מה כתוב בקבצים שלא פתוחים בכלל, דרך אינדוקס של כל הקוד בפרויקט. זה נותן לו הקשר טבעי בלי שהמפתח צריך להעתיק-להדביק כל הזמן.

חשוב להבין: Cursor הוא **עטיפה** מעל מודלים - הוא לא מאמן מודל משלו. אפשר לבחור באיזה מודל להשתמש (בין השאר מודלים של Claude), ו-Cursor אחראי על הרכבת ההקשר, הצגת השינויים, וניהול הזרימה סביב המודל. זה דומה בעיקרון ל-Claude Code, שגם הוא "רק" מפעיל מודל - ההבדל הוא באיפה החוויה קורית: בתוך עורך גרפי מלא, מול טרמינל טקסטואלי.

## שכבות העבודה בתוך Cursor - the interaction layers

ב-Cursor יש כמה דרכים לעבוד מול ה-AI, כל אחת מתאימה לגודל משימה אחר.

### השלמה חכמה - Cursor Tab

זו ההרחבה הטבעית של השלמת קוד רגילה. בזמן שכותבים, Cursor מציע לא רק את השורה הבאה אלא לפעמים שינוי במספר שורות בבת אחת, ואפילו קפיצה למקום אחר בקובץ שצריך להשתנות בהתאם למה שכתבתם. הרעיון: המערכת לא מנחשת רק "מה המילה הבאה", אלא "מה העריכה הבאה שהמתכנת עומד לבצע" - כולל שינויים בקבצים אחרים שקשורים לעריכה הנוכחית. לוחצים Tab כדי לקבל, ופשוט ממשיכים להקליד כדי לדחות.

### עריכה ממוקדת - inline edit

בחירת קטע קוד ובקשת שינוי ממוקד עליו - "תהפוך את הפונקציה הזו לאסינכרונית", "תוסיף טיפול בשגיאות כאן". המודל רואה את הקובץ הפתוח ואת מה שבחרתם, ומחזיר עריכה מקומית שרואים כ-diff לפני שמאשרים. זה שימושי לשינויים קטנים וממוקדים שלא דורשים חקירה רחבה.

### צ'אט בתוך העורך - chat panel

פאנל שיחה קבוע בצד המסך, שמכיר את הקובץ הפתוח ואפשר להוסיף לו הקשר נוסף (קבצים ספציפיים, תיעוד, תוצאות טרמינל). טוב לשאלות, הסברים, ותכנון לפני עריכה בפועל - דומה במהות למצב התכנון שהכרנו ב-Claude Code, רק בממשק גרפי.

### מצב סוכן - agent / composer mode

זה החלק שהכי דומה ל-Claude Code: נותנים משימה בשפה טבעית, וה-AI פועל באופן אוטונומי על פני כמה קבצים - קורא, כותב, מריץ פקודות טרמינל, ולפעמים אפילו מתקן את עצמו לפי תוצאות בדיקות. ההבדל המרכזי מהחוויה בטרמינל הוא ויזואלי: השינויים מוצגים כ-diff צבעוני בתוך העורך, קובץ אחר קובץ, ואפשר לאשר, לדחות או לערוך כל שינוי בנפרד לפני שהוא נכנס לפרויקט.

```text
Tab           -> השלמה תוך כדי הקלדה, מקומי, מהיר
inline edit   -> עריכה ממוקדת בקטע נבחר
chat          -> שיחה, שאלות, תכנון
agent/composer -> משימה אוטונומית רב-קבצית, עם הרצת פקודות
```

## אינדוקס הפרויקט וכללי הקשר - codebase indexing and rules files

חלק גדול מהחוזק של Cursor מגיע מזה שהוא בונה מראש אינדקס של כל הפרויקט - לא רק הקבצים הפתוחים כרגע. כשמריצים בקשה, המערכת מחפשת בתוך האינדקס הזה את הקטעים הרלוונטיים ומצרפת אותם להקשר, גם אם לא ציינתם אותם בפירוש. זה דומה במהות לחיפוש שסוכן טרמינל עושה בעצמו דרך כלי grep וקריאת קבצים, רק שב-Cursor זה קורה ברקע, אוטומטית, בלי שהמשתמש מפעיל שום פקודה.

היתרון: פחות עבודה ידנית לצירוף הקשר. החיסרון: פחות שליטה - קשה לדעת בדיוק אילו קטעים נכנסו ואילו לא, ולפעמים האינדקס "מפספס" קובץ רלוונטי שסוכן טרמינל היה מוצא בקלות עם `grep` ממוקד.

בדיוק כמו שקובץ CLAUDE.md נותן ל-Claude Code הקשר קבוע על הפרויקט (פרק 2.5), גם ב-Cursor אפשר להגדיר קובצי כללים - **Rules** - שמכילים הנחיות קבועות: מוסכמות קוד, שפות שאסור/מותר להשתמש בהן, איך לבנות טסטים, אילו ספריות מועדפות. הכללים האלה נשמרים כטקסט בתוך הפרויקט, נכנסים ל-git כמו כל קובץ אחר, וכל בקשה לסוכן מתחשבת בהם אוטומטית. מבחינה מושגית זה אותו רעיון בדיוק כמו CLAUDE.md - "הקשר קבוע שלא צריך לחזור ולהקליד בכל פעם" - רק בפורמט ובמיקום ספציפיים ל-Cursor.

```text
CLAUDE.md ב-Claude Code   <-->   קובצי Rules ב-Cursor
שניהם: הקשר פרויקט קבוע, בטקסט, בגרסאות git, נטען אוטומטית לכל בקשה
```

הלקח: המושג "תן לסוכן הקשר קבוע במקום לחזור עליו כל פעם" הוא כישור כללי שעובד בכל כלי אג'נטי, לא ידע ספציפי לכלי אחד.

## מתי Cursor עדיף על סוכן טרמינל - when the IDE wins

אין תשובה אחת נכונה, אבל יש הבדלים אמיתיים שכדאי להכיר.

**יתרונות אמיתיים של Cursor:**

- **משוב ויזואלי מיידי** - רואים diff מודגש בצבעים בתוך הקובץ עצמו, עם היסטוריית גרסאות, ואפשר לאשר או לדחות כל שינוי בנפרד בלחיצה. בטרמינל רואים diff טקסטואלי, שגם הוא ברור אבל פחות מוחשי לעריכה נקודתית.
- **השלמה תוך כדי כתיבה** - Cursor Tab מצוין למקצבי עבודה שבהם רוב הזמן עדיין נכתב קוד ידנית, ורק חלק ממנו מבוקש מה-AI. זו חוויה שאין לה מקבילה טבעית בטרמינל.
- **ניווט מהיר בין קבצים רבים** - עורך גרפי מלא, עם עץ קבצים, חיפוש גלובלי, קפיצה להגדרה - כל הכלים הרגילים של IDE זמינים בלי לצאת מהחוויה.
- **נוח למי שחי בתוך עורך** - מפתחים שהרגל העבודה שלהם בנוי סביב IDE (ולא סביב טרמינל) ירגישו בבית מהר יותר.

**יתרונות אמיתיים של סוכן טרמינל כמו Claude Code:**

- **עבודה headless ומרוחקת** - שרת SSH בלי ממשק גרפי, סביבת CI/CD, קונטיינר - בכל אלה עורך גרפי פשוט לא זמין, וסוכן טרמינל עובד בול אותו דבר.
- **הרצה אוטונומית ארוכה** - משימות רב-שלביות שרצות ברקע דקות ארוכות, כולל הרצת בדיקות וסבבי תיקון עצמי, בלי שצריך לחכות ליד המסך. אפשר גם להריץ כמה מופעים במקביל מתוך כמה חלונות טרמינל.
- **שקיפות ושליטה מפורטת** - קובץ CLAUDE.md, מיומנויות, הוקים, סוכני משנה - כל מנגנוני השליטה שלמדנו בפרקים 4 ו-6 מוגדרים כטקסט וגרסתיים ב-git, ולא תלויים בהגדרות של אפליקציה גרפית ספציפית.
- **סקריפטביליות** - קל לשלב סוכן טרמינל בתוך pipeline אוטומטי, סקריפט, או כלי אחר, כי הוא פשוט תהליך שמריצים משורת פקודה.

טבלת סיכום קצרה:

| היבט | Cursor | Claude Code |
|---|---|---|
| סביבה | עורך גרפי (fork של VS Code) | טרמינל |
| השלמה תוך כדי הקלדה | כן, חזקה (Tab) | לא רלוונטי |
| עבודה headless / מרוחקת | מוגבל | טבעי |
| הרצה אוטונומית ארוכה ברקע | אפשרי, פחות טבעי | טבעי, כולל כמה מופעים במקביל |
| שליטה בהקשר דרך קבצים גרסאתיים | חלקי | מלא (CLAUDE.md, skills, hooks) |
| נוחות ויזואלית לעריכות קטנות | גבוהה | בינונית |

## זרימת עבודה טיפוסית ב-Cursor - a typical workflow

זרימה נפוצה נראית כך:

1. פותחים את הפרויקט ב-Cursor, כמו שהייתם פותחים אותו ב-VS Code רגיל.
2. במהלך כתיבת קוד "רגילה", משתמשים ב-Tab כדי לחסוך הקלדה - השלמת בלוקים שלמים, לא רק מילה בודדת.
3. לשינוי ממוקד בקובץ אחד - למשל לשכתב פונקציה או להוסיף type - משתמשים בעריכה ממוקדת על הקטע הנבחר, בודקים את ה-diff, ומאשרים.
4. למשימה שדורשת שינוי בכמה קבצים בבת אחת - הוספת שדה חדש שצריך לזרום דרך מודל, API ו-UI - עוברים למצב סוכן, נותנים תיאור ברור של המשימה (בדיוק כמו פרומפט טוב ל-Claude Code שלמדנו בפרק 2), ונותנים לו לרוץ.
5. עוברים על כל שינוי בנפרד, קוראים את ה-diff, מאשרים מה שנכון ומתקנים או דוחים מה שלא.
6. מריצים בדיקות מתוך הטרמינל המובנה של Cursor, בדיוק כמו שהייתם עושים בטרמינל רגיל.

שימו לב שהצעד האחרון - **קריאה ואימות של כל diff** - הוא לא אופציונלי. בדיוק כמו שלמדנו לגבי Claude Code בפרק 3.6, גם כאן חלה אותה מיומנות של אימות: הצגה יפה של diff לא הופכת קוד שגוי לנכון. הכלי הוויזואלי מקל על הקריאה, אבל לא פוטר מהחובה לקרוא.

דוגמה קונקרטית לבקשה טובה במצב סוכן, שמדגימה שאותם עקרונות של פרומפט טוב שלמדנו בפרק 2 חלים גם כאן:

```text
תוסיף שדה "phone" למודל המשתמש. תעדכן את:
1. סכמת בסיס הנתונים (migration חדש)
2. ה-API - קלט ופלט של /users
3. טופס העריכה ב-frontend, כולל ולידציה בסיסית
4. הטסטים הקיימים שבודקים את מודל המשתמש
תריץ את סוויטת הטסטים בסוף ותראה לי את התוצאה.
```

בקשה כזו ממחישה בדיוק למה מצב סוכן שונה מהשלמת Tab או מעריכה ממוקדת: היא פורשת על פני כמה קבצים, דורשת תכנון פנימי מצד הסוכן, והיא הזמן שבו כדאי לתת ל-Cursor "לרוץ" ולא להתערב בכל שורה.

## מגבלות ונקודות זהירות - fair limitations

חשוב להיות הוגנים: Cursor הוא כלי חזק, אבל לא פותר את כל הבעיות שדנו בהן בפרק 8 בהמשך הקורס.

- **עלות** - Cursor פועל במודל מנוי, בדרך כלל עם מכסת שימוש חודשית במודלים המתקדמים ביותר; שימוש כבד יכול להגיע למגבלות או לעלות נוספת, בדומה לכל כלי מבוסס מודל שפה.
- **שקיפות הקשר** - כשה-IDE "מבין" את הפרויקט אוטומטית, לפעמים קשה לדעת בדיוק אילו קבצים נכנסו להקשר של הבקשה, בהשוואה לניהול הקשר המפורש שיש בטרמינל (קבצים שמזכירים בפירוש, CLAUDE.md, פקודות סלאש).
- **נעילה לעורך אחד** - עבודה יומיומית ב-Cursor מרגילה אתכם לחוויה ולקיצורים של עורך ספציפי. זה לא רע, אבל חשוב לזכור שהכישורים המרכזיים - כתיבת פרומפט טוב, אימות תוצאה, פירוק משימה - הם כישורים כלליים שעובדים בכל כלי, לא ידע ספציפי ל-Cursor.
- **התנהגות "קופסה שחורה" יחסית** - אין באותה מידה מנגנוני הרחבה גלויים כמו הוקים ומיומנויות מבוססות טקסט; ההתאמה האישית מוגבלת יותר להגדרות שהעורך חושף.

המסקנה המעשית: Cursor הוא כלי מצוין להוסיף לארגז הכלים, במיוחד לעבודה יומיומית שכוללת הרבה עריכות קטנות-בינוניות בתוך עורך. הוא לא מחליף סוכן טרמינל למשימות אוטונומיות ארוכות, headless, או כאלה שדורשות שליטה מדויקת בהקשר. בפרק 5.6 נראה טבלת החלטה מסודרת שמכסה את כל הכלים שנלמד בפרק הזה, ונבין איך לשלב ביניהם בפועל.

כדאי גם לזכור שהתחום הזה משתנה מהר. הגבול בין "עורך עם AI" ל"סוכן טרמינל" כבר לא חד כמו שהיה - יש תוספים שמביאים סוכני טרמינל אל תוך עורכים גרפיים (כולל Claude Code עצמו, שיש לו אינטגרציות עם IDE-ים שונים), ויש כלים כמו Cursor שמוסיפים יכולות אוטונומיה שדומות יותר ויותר לסוכן עצמאי. המסקנה המעשית לא משתנה: ללמוד לחשוב במונחים של "מה צורת המשימה" ו"כמה שליטה אני צריך", ולא להתחתן עם ממשק ספציפי, כי הממשקים ימשיכו להתפתח והכישורים הבסיסיים - ניסוח בקשה ברורה, אימות תוצאה, פירוק משימה לצעדים - יישארו רלוונטיים בכל כלי.

## סיכום

- **Cursor** הוא עורך קוד עצמאי (fork של VS Code) עם AI משולב עמוק בתוך חוויית העריכה, לא תוסף חיצוני.
- ארבע שכבות עבודה עיקריות: **Tab** (השלמה תוך כדי הקלדה), **עריכה ממוקדת** (שינוי בקטע נבחר), **צ'אט** (שאלות ותכנון), ו**מצב סוכן/Composer** (משימה אוטונומית רב-קבצית עם הרצת פקודות).
- Cursor עוטף מודלים קיימים (כולל מודלים של Claude) - הוא לא מאמן מודל משלו, בדיוק כמו ש-Claude Code הוא עטיפה סביב מודל.
- Cursor מצטיין במשוב ויזואלי, בהשלמה תוך כדי הקלדה, ובעבודה נוחה בתוך עורך גרפי מלא.
- סוכן טרמינל כמו Claude Code מצטיין בעבודה headless/מרוחקת, בהרצה אוטונומית ארוכה, ובשליטה מפורשת בהקשר דרך קבצים גרסאתיים (CLAUDE.md, מיומנויות, הוקים).
- גם ב-Cursor חלה אותה חובת אימות שלמדנו לגבי Claude Code - diff יפה לא הופך קוד שגוי לנכון.
- המגבלות ההוגנות: עלות מבוססת מנוי, שקיפות חלקית של מה נכנס להקשר, נעילה יחסית לעורך אחד.
- הבחירה בין הכלים היא לפי צורת המשימה, לא נאמנות לכלי אחד - ונרחיב על כך בהרצאה 5.6.
