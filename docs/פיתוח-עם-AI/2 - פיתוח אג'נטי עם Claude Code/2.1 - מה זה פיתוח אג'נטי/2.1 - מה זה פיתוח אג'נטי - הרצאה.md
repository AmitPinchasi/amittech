רוב המתכנתים פגשו כבר עוזרי קוד מבוססי בינה מלאכותית בצורה של השלמה אוטומטית: אתה מתחיל לכתוב שורה, וכלי כמו Copyright מציע לך את ההמשך. זה שימושי, אבל זה עוזר קטן וכנוע. בפרק הזה נכיר משהו אחר לגמרי: סוכן - agent. סוכן הוא לא רק מנבא את המילה הבאה, אלא ישות שמקבלת מטרה, מפרקת אותה לצעדים, מפעילה כלים אמיתיים במחשב שלך (קוראת קבצים, עורכת אותם, מריצה פקודות), בודקת את התוצאה, ומתקנת את עצמה עד שהמשימה הושלמה. Claude Code הוא סוכן כזה, שרץ בטרמינל, והוא עמוד השדרה של הקורס הזה.

בהרצאה הזו נבין את הרעיון המרכזי - לולאת הסוכן. נראה אילו כלים יש לסוכן, במה זה שונה מהשלמה אוטומטית ומצ'אט רגיל, למה הלולאה כל כך חזקה, ואיפה בדיוק היא נשברת. זו הרצאה מושגית: אין כאן עדיין הרבה פקודות להקליד, אלא מפה מנטלית שתשרת אתכם בכל שאר הפרק.

---

## לולאת הסוכן - the agentic loop

הרעיון שמתחת לכל סוכן הוא לולאה פשוטה שחוזרת על עצמה. בתיעוד הרשמי של Claude Code מתארים אותה בשלושה שלבים שמתמזגים זה בזה: איסוף הקשר, ביצוע פעולה, ואימות התוצאה. נפרק אותם:

```text
1. איסוף הקשר   - קרא קבצים, חפש בקוד, הבן מה קורה
2. תכנון         - החלט מה הצעד הבא כדי להתקדם למטרה
3. פעולה         - הפעל כלי: ערוך קובץ, הרץ פקודה, חפש
4. תצפית         - קרא את הפלט: הצליח? נכשל? מה למדנו?
5. חזרה על הלולאה עד שהמטרה הושגה
```

ההבדל המהותי בין סוכן למודל שפה רגיל הוא שהסוכן **פועל בעולם ומקבל משוב**. מודל שפה לבדו יכול רק לייצר טקסט. ברגע שנותנים לו כלים, כל הפעלת כלי מחזירה מידע חדש שנכנס בחזרה ללולאה ומשפיע על הצעד הבא. זו בדיוק הנקודה: הסוכן לא מנחש את כל הפתרון מראש, אלא מגלה אותו צעד אחרי צעד, בדיוק כמו שמתכנת אנושי עובד.

בואו נראה דוגמה קונקרטית. נניח שביקשתם: "תקן את הבדיקות שנכשלות". סוכן טוב יעשה בערך כך:

```text
> תקן את הבדיקות שנכשלות

1. מריץ את חבילת הבדיקות     -> רואה ש-3 בדיקות נכשלות
2. קורא את הודעת השגיאה       -> "TypeError בקובץ auth.py שורה 42"
3. מחפש את הקובץ auth.py       -> מוצא אותו
4. קורא את auth.py            -> מבין שהפונקציה מחזירה None במקום dict
5. עורך את auth.py            -> מתקן את הבאג
6. מריץ שוב את הבדיקות         -> כולן עוברות. סיום.
```

שימו לב שאף שלב לא תוכנן מראש במלואו. הסוכן החליט להריץ שוב את הבדיקות רק **כי** הוא ערך את הקובץ, והוא ערך את הקובץ רק **כי** קרא את השגיאה. כל צעד נבע מהתצפית של הצעד הקודם. זו הלולאה בפעולה, והיא יכולה לשרשר עשרות פעולות כאלה ולתקן את מסלולה תוך כדי תנועה.

---

## הכלים של הסוכן - tools

בלי כלים, Claude יכול רק לענות בטקסט. הכלים הם מה שהופך אותו לסוכן. התיעוד מחלק את הכלים המובנים לחמש קטגוריות, וכדאי להכיר אותן כי כל בקשה שלכם תתורגם בסופו של דבר לרצף של הפעלות כלים:

| קטגוריה | מה הסוכן יכול לעשות |
| --- | --- |
| פעולות קבצים - file operations | לקרוא קבצים, לערוך קוד, ליצור קבצים חדשים, לשנות שם ולארגן |
| חיפוש - search | למצוא קבצים לפי תבנית, לחפש תוכן עם ביטוי רגולרי, לסרוק בסיס קוד |
| הרצה - execution | להריץ פקודות מעטפת, להפעיל שרתים, להריץ בדיקות, להשתמש ב-git |
| רשת - web | לחפש באינטרנט, למשוך תיעוד, לחפש הודעות שגיאה |
| בינת קוד - code intelligence | לראות שגיאות טיפוס ואזהרות אחרי עריכה, לקפוץ להגדרות, למצוא הפניות |

מאחורי הקטגוריות האלה יש כלים ספציפיים בעלי שמות. שווה להכיר את העיקריים, כי תראו אותם רצים על המסך בזמן עבודה:

```text
Read    - קורא תוכן של קובץ
Edit    - מבצע שינוי מדויק בתוך קובץ קיים
Write   - כותב קובץ חדש (או דורס קיים)
Grep    - חיפוש תוכן לפי ביטוי רגולרי בכל הקוד
Glob    - מציאת קבצים לפי תבנית שם (למשל *.py)
Bash    - מריץ פקודת מעטפת אמיתית במחשב שלך
```

ה-Claude בוחר בעצמו באיזה כלי להשתמש, מתי, ועם אילו ארגומנטים, על סמך הבקשה שלכם ומה שהוא לומד בדרך. אתם לא אומרים לו "קרא את הקובץ הזה ואז הרץ את הפקודה הזאת" - אתם מתארים מטרה, והוא מרכיב את רצף הכלים בעצמו. זו בדיוק המשמעות של המילה "אג'נטי": יש כאן סוכן שמפעיל שיקול דעת, לא סקריפט שמריץ צעדים קבועים.

מעבר לכלים המובנים אפשר להרחיב את היכולות: מיומנויות - skills, שרתי MCP לחיבור לשירותים חיצוניים, hooks לאוטומציה, ותת-סוכנים למשימות מקבילות. את אלה נלמד בפרקים מאוחרים יותר. בשלב הזה מספיק להבין שהבסיס הוא חמש הקטגוריות שלמעלה.

---

## במה זה שונה מהשלמה אוטומטית - autocomplete

ההבדל בין סוכן להשלמה אוטומטית הוא לא הבדל של איכות אלא של סוג. השלמה אוטומטית עובדת ברמת השורה או הבלוק: היא רואה את מה שאתם מקלידים כרגע ומציעה המשך. היא לא יודעת להריץ את הקוד, לא בודקת אם הוא עובד, ולא רואה קבצים אחרים בפרויקט אלא במידה מוגבלת.

הנה ההשוואה המלאה:

| היבט | השלמה אוטומטית | סוכן - Claude Code |
| --- | --- | --- |
| היקף ראייה | הקובץ הנוכחי, לרוב סביבת הסמן | כל הפרויקט, git, טרמינל |
| פעולה | מציע טקסט להקליד | מריץ פקודות, עורך קבצים |
| משוב | אין - לא מריץ כלום | מריץ בדיקות ורואה תוצאה |
| יוזמה | מגיב לסמן שלכם | מפרק מטרה לצעדים בעצמו |
| תיקון עצמי | אין | חוזר על הלולאה עד הצלחה |

השלמה אוטומטית היא כמו מישהו שמסיים לכם משפטים. סוכן הוא כמו עמית עבודה שאתם מוסרים לו משימה שלמה, והוא הולך, חוקר, כותב, בודק וחוזר עם תוצאה. שני הכלים לגיטימיים, אבל הם פותרים בעיות שונות. הקורס הזה מתמקד כולו בגישה האג'נטית, כי היא זו שמשנה את אופן העבודה מהיסוד.

---

## במה זה שונה מצ'אט - chat

השאלה המתבקשת: הרי אפשר לפתוח צ'אט עם מודל שפה, להדביק קוד, ולקבל תשובה. במה זה שונה מסוכן? ההבדל הוא בשלושה דברים.

ראשית, **גישה ישירה לפרויקט**. בצ'אט רגיל אתם צריכים להעתיק ולהדביק כל קובץ ידנית. הסוכן קורא בעצמו את הקבצים שהוא צריך, מתי שהוא צריך אותם, בלי שתעשו כלום. כשאתם שואלים "מה הפרויקט הזה עושה?", הוא סורק את הקבצים ומחזיר תשובה מבוססת על מה שיש בפועל, לא על ניחוש.

שנית, **פעולה במקום המלצה**. צ'אט אומר לכם "תשנה את השורה הזאת ככה". סוכן פשוט משנה אותה (אחרי שיאשר איתכם). ההבדל הזה מצטבר: משימה שדורשת עשרה שינויים בחמישה קבצים היא בצ'אט עשר פעולות העתקה-הדבקה שלכם, ובסוכן היא בקשה אחת.

שלישית, **לולאת המשוב**. צ'אט עונה פעם אחת ועוצר. סוכן מריץ את הקוד, רואה שהוא נכשל, ומתקן - בלי שתחזרו אליו. זו בדיוק היכולת שהצ'אט חסר: לסגור את המעגל בין כתיבה לאימות.

אפשר לחשוב על זה כך: צ'אט הוא מודל שפה עירום. סוכן הוא אותו מודל שפה, עטוף במה שהתיעוד מכנה "מעטפת אג'נטית" - agentic harness - שמספקת לו כלים, ניהול הקשר, וסביבת הרצה. Claude Code הוא המעטפת הזו סביב מודל Claude.

---

## למה הלולאה חזקה - ואיפה היא נשברת

הכוח של הלולאה נובע מהמשוב. מתכנת אנושי לא כותב תוכנית שלמה ומגיש אותה בלי להריץ; הוא כותב מעט, מריץ, רואה שגיאה, מתקן, וחוזר. הסוכן עושה בדיוק את זה, אבל במהירות ובלי להתעייף. הוא יכול לשרשר עשרות איטרציות של "נסה, בדוק, תקן" תוך דקות. ככל שנותנים לו יותר דברים לאמת מולם - בדיקות, הודעות שגיאה, פלט צפוי - כך התוצאה טובה יותר, כי לכל צעד יש אות ברור אם הוא נכון.

אבל אותה עוצמה היא גם מקור הסכנה. הסוכן מריץ **פקודות אמיתיות במחשב שלכם**. אם הוא מבין את המשימה לא נכון, הוא לא סתם נותן עצה גרועה - הוא עלול למחוק קבצים, לשנות דברים שלא התכוונתם אליהם, או להריץ פקודה הרסנית. הנה המקומות העיקריים שבהם הלולאה נשברת:

- **מטרה מעורפלת**: אם הבקשה שלכם לא ברורה, הסוכן ימלא את החללים בהנחות משלו, ולפעמים ההנחות שגויות. "תנקה את הקוד" יכול להתפרש בעשר דרכים.
- **אין מול מה לאמת**: כשאין בדיקות ואין דרך לבדוק את התוצאה, הסוכן עלול "לחשוב" שהוא הצליח כשלמעשה נכשל. הוא זקוק לאות אמת.
- **הקשר שמתמלא**: לסוכן יש חלון הקשר מוגבל. בשיחות ארוכות מאוד מידע מוקדם עלול ללכת לאיבוד, והוא ישכח הנחיה שנתתם בהתחלה.
- **הרעלת קלט**: אם הסוכן קורא קובץ או דף אינטרנט שמכיל הוראות זדוניות, הוא עלול לפעול לפיהן. זו סיבה מרכזית לכך שקיים מנגנון הרשאות.

בגלל כל אלה, Claude Code בנוי סביב שני מנגנוני בטיחות שנפגוש בהמשך הפרק: **הרשאות** - הסוכן מבקש אישור לפני פעולה בעלת השלכות, ו**נקודות שחזור** - checkpoints - כל עריכת קובץ ניתנת לביטול. שני אלה קיימים בדיוק כי הלולאה חזקה מכדי להריץ אותה בעיניים עצומות.

---

## דוגמה קונקרטית לכשל - a concrete failure

כדי שהאזהרות לא יישארו מופשטות, נראה תרחיש כשל אמיתי. נניח שביקשתם "תמחק את קבצי הזמניים בתיקייה". הכוונה שלכם הייתה קבצי `.tmp` בתיקיית הפרויקט. אבל אם ניסחתם את זה בעמימות והסוכן פירש רחב מדי, הוא עלול לנסות פקודת מחיקה גורפת:

```text
> תמחק את הקבצים הזמניים

  Bash  rm -rf ./*   <- פרשנות שגויה ומסוכנת!
  Do you want to run this command? > 3. No
```

כאן בדיוק נכנס מנגנון ההרשאות. הסוכן לא הריץ את הפקודה - הוא **ביקש אישור**, ואתם, שראיתם `rm -rf ./*`, דחיתם והבהרתם: "רק קבצי .tmp, לא הכל". הדחייה תפסה את הטעות לפני שהיא קרתה. זו התמונה המלאה של למה הלולאה חזקה אבל דורשת פיקוח: הסוכן פועל מהר ורחב, ומנגנוני הבטיחות הם מה שהופך את המהירות הזו לבטוחה.

הלקח: ככל שהבקשה ברורה יותר, כך הסוכן קולע לכוונה; וכשהוא בכל זאת סוטה, ההרשאות ונקודות השחזור הן רשת הביטחון. שני אלה הם נושא ההרצאות הבאות בפרק.

---

## הלך הרוח הנכון - delegate, don't dictate

בגלל שהסוכן מפרק מטרות בעצמו, הדרך היעילה לעבוד איתו היא לחשוב עליו כעל עמית עבודה מוכשר, לא כעל מכונה שמקבלת פקודות מדויקות. אתם נותנים הקשר וכיוון, וסומכים עליו שימצא את הפרטים. במקום לומר "קרא את הקובץ הזה, ואז הרץ את הפקודה הזאת, ואז ערוך את השורה ההיא", אומרים:

```text
תהליך התשלום שבור עבור משתמשים עם כרטיס שפג תוקף.
הקוד הרלוונטי נמצא ב-src/payments/. תחקור ותתקן.
```

הסוכן יבין לבד אילו קבצים לקרוא ואילו פקודות להריץ. ככל שהמטרה ברורה ומעוגנת יותר בהקשר (קבצים ספציפיים, מגבלות, דוגמאות), כך יש פחות צורך לתקן אותו תוך כדי. וכשהוא בכל זאת טועה, לא מתחילים מחדש - מכוונים אותו: "זה לא מדויק, הבעיה היא בטיפול בסשן". זו שיחה, לא פקודה בודדת.

## סיכום

- **סוכן** מקבל מטרה, מפרק אותה לצעדים, מפעיל כלים אמיתיים, בודק את התוצאה, ומתקן את עצמו - להבדיל ממודל שפה שרק מייצר טקסט.
- **לולאת הסוכן** היא איסוף הקשר, פעולה עם כלים, תצפית על התוצאה, וחזרה עד להשגת המטרה. כל צעד נובע מהתצפית של הצעד הקודם.
- הסוכן מפעיל **כלים** בחמש קטגוריות: פעולות קבצים (Read, Edit, Write), חיפוש (Grep, Glob), הרצה (Bash), רשת, ובינת קוד. הוא בוחר בעצמו במה להשתמש.
- **השלמה אוטומטית** מציעה טקסט ברמת השורה בלי להריץ כלום; סוכן פועל על כל הפרויקט ובודק את עצמו.
- **צ'אט** עונה פעם אחת ועוצר; סוכן ניגש ישירות לקבצים, פועל במקום להמליץ, וסוגר את מעגל המשוב.
- הלולאה **חזקה** בזכות המשוב - ככל שיש יותר מול מה לאמת, כך התוצאה טובה יותר.
- הלולאה **נשברת** כשהמטרה מעורפלת, כשאין מול מה לאמת, כשההקשר מתמלא, או בהרעלת קלט. בגלל זה קיימים מנגנוני הרשאות ונקודות שחזור.
- הגישה הנכונה: להאציל כמו לעמית מוכשר - לתת הקשר וכיוון, לא להכתיב כל צעד.
