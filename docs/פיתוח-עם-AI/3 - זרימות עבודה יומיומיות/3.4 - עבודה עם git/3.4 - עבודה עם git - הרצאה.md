הסוכן Claude Code רץ בטרמינל, ולכן git הוא כלי טבעי בשבילו בדיוק כמו בשבילכם - הוא יכול להריץ `git status`, `git diff`, `git commit`, לפתוח branch, וליצור PR דרך `gh`. זה נוח מאוד, וזה גם בדיוק המקום שבו נוחות יכולה להפוך לרשלנות: קל מאוד לתת לסוכן "לסיים" עבודה על ידי commit ו-push בלי שאף אחד עבר על מה שבאמת נכנס להיסטוריה. בהרצאה הזו נעבור על כל שלבי העבודה עם git מול סוכן - staging, ניסוח הודעות commit, יצירת branch ו-PR, סקירת diff, פתרון conflicts - ונקבע גדרות הגנה ברורות שמונעות מהנוחות הזו להזיק.

## הכלל המסגרת: הסוכן מריץ git, אתם מאשרים

לפני שנכנסים לפרטים, כלל אחד שחוצה את כל ההרצאה: **אף שינוי לא נכנס להיסטוריית git בלי שראיתם אותו**. זה נכון גם ל-`git add`, גם ל-`git commit`, ובוודאי ל-`git push`. Claude Code בהגדרת ברירת מחדל מבקש אישור לפעולות כאלה, וזה לא עיצוב שרירותי - זה בדיוק שכבת ההגנה שאתם צריכים. אל תסירו אותה כדי "לחסוך זמן". הזמן שנחסך היום הוא הזמן שתשקיעו בעוד שבוע בניקוי היסטוריית git מבולגנת או, גרוע מכך, בלגלות ש-commit נדחף לענף main בלי סקירה.

## Staging: מה נכנס ל-commit

לפני commit יש שאלה שקל לדלג עליה: האם *כל* מה ששונה צריך להיכנס לאותו commit? סוכן שמריץ `git add -A` בלי לחשוב מכניס יחד שינויים לא קשורים - למשל את הפיצ'ר שביקשתם, בתוספת קובץ קונפיגורציה שהשתנה בטעות בזמן ניסוי מקומי.

```text
> תראה לי git status ותציע לי אילו קבצים שייכים ל-commit
  הזה ואילו לא. אל תעשה add לכלום לפני שאני מאשר.
```

```bash
$ git status --short
 M src/services/orderService.ts
 M src/api/controllers/orderController.ts
 M .env.local
?? src/api/__tests__/orders.export.test.ts
```

```text
< src/services/orderController.ts ו-orderController.ts שייכים
  לפיצ'ר ה-export. הבדיקה החדשה גם שייכת. .env.local נראה
  כמו קובץ סביבה מקומי שהשתנה בטעות - כדאי שלא ייכנס ל-commit,
  ורצוי לוודא שהוא ב-.gitignore.
```

ההרגל הזה - לבקש מיפוי מפורש של מה שייך למה, לפני `git add` - חוסך המון conflicts ובלבול בהמשך, במיוחד כשעובדים על כמה דברים בו-זמנית.

## כתיבת הודעות commit - commit messages

סוכן יכול לכתוב הודעת commit טובה, אבל רק אם נותנים לו את ההקשר הנכון. הודעה טובה מסבירה **למה**, לא רק **מה** - ה"מה" כבר נמצא ב-diff עצמו.

```text
> תכתוב הודעת commit לשינויים המסודרים ב-staging. תסביר
  למה השינוי נחוץ, לא רק מה השתנה. תשמור על הפורמט שנהוג
  בפרויקט - תבדוק git log לדוגמאות אחרונות.
```

בדיקת `git log` לפני ניסוח היא צעד חשוב: לכל פרויקט יש קונבנציה - Conventional Commits (`feat:`, `fix:`, `chore:`), פורמט חופשי, או משהו אחר. סוכן שבודק את ההיסטוריה לפני שהוא כותב נשמע כאילו הוא תמיד היה חלק מהצוות.

```bash
$ git log --oneline -5
a3f21c9 fix: prevent duplicate webhook processing on retry
9b04e11 feat: add rate limiting to public API endpoints
7d18aa2 chore: bump prisma to 5.9.0
```

```text
< feat: add CSV export endpoint for orders

  Users can now export their orders within a date range as CSV
  via GET /orders/export. Filters by userId to prevent leaking
  other users' data; returns 400 on missing/invalid date params.
```

שימו לב שההודעה מזכירה במפורש את סינון ה-userId - זו בדיוק סוג הפרט שיעזור למי שיקרא את ההיסטוריה בעוד חצי שנה להבין למה השורה הזו שם.

## יצירת branch ו-PR - branches and pull requests

כלל ברזל: **אף פעם לא עובדים ישירות על main**. גם כשמדובר בשינוי קטן. תמיד מבקשים מהסוכן ליצור branch ייעודי לפני שמתחילים לעבוד:

```text
> תיצור branch חדש בשם feature/orders-csv-export מ-main
  העדכני, ותעבור אליו לפני שנתחיל לממש.
```

```bash
$ git checkout -b feature/orders-csv-export main
Switched to a new branch 'feature/orders-csv-export'
```

בסיום העבודה, דוחפים ומבקשים ליצור PR - כשהסוכן יכול להשתמש ב-`gh` (GitHub CLI) ליצור אותו ישירות, כולל תיאור:

```text
> תדחוף את הbranch ותיצור PR מול main. בתיאור ה-PR, כלול
  תקציר של השינוי, ורשימת בדיקות שהרצת עם התוצאה שלהן.
```

```bash
$ git push -u origin feature/orders-csv-export
$ gh pr create --title "feat: add CSV export endpoint for orders" \
    --body "## Summary
- מוסיף GET /orders/export להורדת הזמנות כ-CSV לפי טווח תאריכים
- מסנן לפי userId - מונע דליפת נתונים בין משתמשים

## Test plan
- [x] בדיקות יחידה: findByDateRange, validation, isolation בין משתמשים
- [x] חבילת הבדיקות המלאה עברה (npm test)
"
```

תיאור PR שכולל test plan מפורש הוא לא רק נוחות לביקורת - הוא מכריח אתכם ואת הסוכן לענות בפירוש על "איך אנחנו יודעים שזה עובד", לא רק "מה השתנה".

## סקירת diff לפני commit - reviewing the diff

זה חוזר מהרצאה 3.2, אבל בהקשר git הוא מקבל ממד נוסף: `git diff --staged` הוא בדיוק מה שייכנס להיסטוריה, ולכן זה הרגע האחרון לתפוס בעיה לפני שהיא "רשמית".

```text
> תראה לי git diff --staged במלואו, קובץ אחר קובץ.
```

בסקירה שימו לב במיוחד לשני דברים שקל לפספס:

- **קבצים שלא היו אמורים להשתנות** - למשל שינוי לא מכוון בקובץ lock, בהגדרות linter, או formatting אוטומטי שרץ על קבצים שלא נגעתם בהם ומייצר diff ענק ולא רלוונטי.
- **secrets או מידע רגיש** - מפתחות API, טוקנים, סיסמאות שנשארו בקוד לבדיקה. סוכן לא תמיד מזהה את זה כבעיה כי מבחינתו זה "רק ערך במשתנה".

```text
> תסרוק את ה-diff המלא ותגיד לי אם יש בו כל מחרוזת שנראית
  כמו API key, טוקן, סיסמה או secret אחר.
```

## פתרון conflicts עם הסוכן - resolving conflicts

כשעולה conflict בעת מיזוג (merge) או ריבייס (rebase), אפשר לתת לסוכן לעזור - אבל צריך זהירות מיוחדת, כי conflict אומר ששני שינויים אמיתיים מתנגשים, ופתרון לא נכון עלול לאבד לוגיקה של אחד הצדדים בשקט.

```text
> יש לי conflict ב-src/services/orderService.ts אחרי rebase
  על main. תראה לי את שני הצדדים של ה-conflict, תסביר מה
  כל צד מנסה לעשות, ורק אז תציע פתרון.
```

```bash
$ git status --short
UU src/services/orderService.ts
```

```text
< בצד "ours" (הbranch שלך) - הוספת findByDateRange לייצוא CSV.
  בצד "theirs" (main) - שינוי חתימת calculateTotal להוסיף
  פרמטר discountCode. שני השינויים לא סותרים לוגית, רק נגעו
  באותו איזור בקובץ. אשלב את שניהם: אשמור את findByDateRange
  כמו שהוא, ואעדכן את הקריאות ל-calculateTotal בקובץ הזה כדי
  שיתאימו לחתימה החדשה עם discountCode.
```

אחרי שהסוכן פותר, זה בדיוק המקום להריץ שוב את הבדיקות - conflict שנפתר "נכון" תחבירית עדיין יכול לשבור לוגיקה:

```text
> אחרי שפתרת את ה-conflict, תריץ את כל חבילת הבדיקות ותראה
  לי שהכל עדיין עובר, כולל הבדיקות של calculateTotal.
```

אם ה-conflict מורכב - נוגע בלוגיקה עסקית עדינה, לא רק בקוד מכני - שקלו לפתור אותו בעצמכם עם הסוכן כעוזר שמסביר את שני הצדדים, במקום לתת לו לבחור פתרון לבד.

## סקירת PR של אחרים - reviewing someone else's pull request

הסוכן לא שימושי רק ל-PR שלכם - הוא כלי טוב גם לסקירת קוד של חברי צוות, בתנאי שמבינים את הגבול שלו: הוא יכול לזהות בעיות טכניות ואי-עקביות, אבל הוא לא יודע את ההקשר העסקי שרק בן אדם מכיר.

```text
> תבדוק את ה-PR הזה: gh pr view 482 --json files,body
  תעבור על השינויים ותסמן לי: קוד שסותר קונבנציה קיימת
  בפרויקט, מקומות שחסרה בהם בדיקה, ושינויים שנראים גדולים
  יותר ממה שהתיאור של ה-PR מרמז.
```

```bash
$ gh pr diff 482 --patch | head -50
```

הסוכן יכול לעבור על ה-diff המלא ולחפש דפוסים בעייתיים - קריאה ל-API בלי טיפול בשגיאות, שינוי בסכימת DB בלי מיגרציה נלווית, פונקציה חדשה שדומה מאוד לפונקציה קיימת (סימן לכפילות קוד). זה שימושי כשלב מקדים לפני שאתם, כבני אדם, משקיעים זמן בסקירה - אבל הוא לא תחליף לסקירה אנושית, בטח לא כשמדובר בלוגיקה עסקית עדינה או בהחלטת ארכיטקטורה. סוכן לא ידע להגיד "זה נוגד את מה שסיכמנו בפגישת התכנון שבוע שעבר" - רק אתם יודעים את זה.

```text
> תסכם לי בשלוש נקודות: מה ה-PR הזה עושה, מה הסיכון הכי
  גדול שאתה רואה בו, ומה היית שואל את המפתח לפני שמאשרים.
```

## rebase מול merge, ועבודה מול branch משותף

כשכמה אנשים עובדים על אותו branch, או כשצריך לעדכן branch פיצ'ר מול main שהתקדם, יש שתי אסטרטגיות עיקריות - merge ו-rebase - וההבדל ביניהן משפיע על היסטוריית git שתישאר לתמיד. שווה לוודא שהסוכן פועל לפי הקונבנציה של הצוות שלכם, לא לפי ברירת מחדל כללית:

```text
> לפני שאתה מעדכן את הbranch שלי מול main, תבדוק בגיטהאב
  (או תשאל אותי) - הצוות הזה עובד עם rebase או עם merge
  commits? תפעל לפי מה שנהוג, לא לפי מה שנוח.
```

נקודה קריטית: **rebase על branch שכבר שותף עם אנשים אחרים (נדחף ל-remote ואחרים עשויים למשוך ממנו) הוא מסוכן**, כי הוא כותב מחדש היסטוריה שכבר קיימת אצל אחרים. אם יש ולו ספק אם מישהו אחר כבר משך את ה-branch, זו בדיוק הסיטואציה שבה שווה לעצור ולשאול, לא לתת לסוכן להחליט לבד:

```text
> האם מישהו אחר יכול להיות שכבר משך את הbranch הזה?
  אם כן, בוא לא נעשה rebase - נשתמש ב-merge כדי לא
  לשבור להם את ההיסטוריה המקומית.
```

## גדרות הגנה - guardrails

| כלל | למה |
|---|---|
| אף פעם לא עובדים ישירות על main | main חייב להישאר יציב; branch מייצר בידוד ומרחב לתקן טעויות |
| כל commit נסקר לפני שהוא נוצר | הודעת commit והתוכן שנכנס הם היסטוריה קבועה, לא טיוטה |
| אף push לא קורה בלי אישור מפורש | push הופך שינוי מקומי לגלוי לצוות (או ל-CI) |
| force push דורש הסכמה מודעת, ובוודאי לא ל-main/master | force push יכול למחוק היסטוריה של אחרים בשקט |
| לפני יצירת PR, מריצים בדיקות מקומית | PR עם בדיקות שנכשלות מבזבז זמן סקירה של אחרים |
| conflicts נפתרים עם הבנה של שני הצדדים, לא רק "תבחר גרסה" | פתרון עיוור עלול לאבד שינוי לוגי אמיתי של אחד הצדדים |

```text
1. > תיצור branch feature/orders-csv-export מ-main.
   < [branch נוצר]

2. > [עבודה על הפיצ'ר - ראו הרצאה 3.2]

3. > תראה לי git status ותציע מה שייך לcommit הזה.
```

כל שלב כאן כולל אישור מפורש שלכם לפני שהסוכן ממשיך לפעולה הבאה שנוגעת בהיסטוריית git - זה בדיוק ההבדל בין "נוח" ל"מסוכן".

## סיכום

- עבודה עם git דרך סוכן היא נוחה מאוד, ולכן היא בדיוק המקום שבו קל לתת לפעולות בלתי הפיכות (commit, push) לקרות בלי סקירה - זה הסיכון המרכזי של ההרצאה.
- לפני staging, מבקשים מהסוכן למפות מפורשות אילו קבצים שייכים ל-commit, כדי לא לערבב שינויים לא קשורים.
- הודעות commit טובות מסבירות למה, לא רק מה, ונכתבות אחרי בדיקת `git log` כדי לשמור על קונבנציית הפרויקט.
- תמיד עובדים על branch ייעודי, לעולם לא ישירות על main - והסוכן יכול ליצור אותו, לדחוף, וליצור PR עם `gh` כולל תיאור ו-test plan.
- סוקרים `git diff --staged` במלואו לפני commit, ומחפשים במפורש קבצים לא רלוונטיים ו-secrets שהוזרקו בטעות.
- ב-conflicts, מבקשים מהסוכן להסביר את שני הצדדים לפני הצעת פתרון, ותמיד מריצים בדיקות אחרי פתרון conflict.
- גדרות ההגנה המרכזיות: אין עבודה על main, אין commit או push בלי אישור מפורש, ואין force push בלי הסכמה מודעת ובוודאי לא על ענפים משותפים.
