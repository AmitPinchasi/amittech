בהרצאה הקודמת ראינו איך לשלוח בקשה למודל ולקבל טקסט בחזרה. אבל מודל שיודע רק לדבר מוגבל: הוא לא יודע מה השעה עכשיו, מה מזג האוויר בתל אביב, כמה שורות יש בקובץ מסוים, או מה יש בבסיס הנתונים שלנו. כל אלה הם נתונים שהמודל לא הודרך עליהם ולא יכול לגשת אליהם. הפתרון נקרא **שימוש בכלים** - נותנים למודל רשימת פונקציות שהוא רשאי "לבקש" מאיתנו להריץ. המודל לא מריץ שום דבר בעצמו; הוא רק מבקש, אנחנו מריצים בקוד שלנו, ומחזירים לו את התוצאה. זה הרעיון שהופך צ'אטבוט פשוט לסוכן שיכול לפעול בעולם.

בהרצאה הזו נבין איך מגדירים כלי, נכיר את **לולאת השימוש בכלים** - הריקוד שבו המודל מבקש קריאה, אנחנו מבצעים, ומחזירים תוצאה - ונבנה דוגמה קטנה ורצה מקצה לקצה.

---

## מה זה שימוש בכלים - tool use

שימוש בכלים (נקרא גם function calling) עובד כך: בבקשה אנחנו מוסיפים שדה `tools` עם רשימת הכלים שהמודל רשאי להשתמש בהם. כל כלי הוא תיאור: שם, מה הוא עושה, ואילו פרמטרים הוא מקבל. המודל **לא** מקבל את הקוד של הכלי - רק את התיאור. הוא מחליט לבד, לפי הבקשה של המשתמש, אם ומתי לקרוא לכלי.

התהליך המדויק:

```text
1. אנחנו שולחים בקשה עם רשימת tools + הודעת המשתמש
2. המודל מחליט שהוא צריך כלי, ומחזיר stop_reason = "tool_use"
   עם בלוק tool_use שמכיל את שם הכלי והקלט
3. אנחנו מריצים את הפונקציה האמיתית בקוד שלנו
4. אנחנו שולחים בקשה חדשה, ומצרפים tool_result עם התוצאה
5. המודל קורא את התוצאה ומנסח תשובה סופית למשתמש
```

הנקודה הקריטית: המודל אף פעם לא מריץ קוד. הוא רק **מבקש** קריאה. הביצוע בפועל - זה באחריותנו, בקוד שלנו. זה מה שנותן לנו שליטה מלאה על מה מותר ומה אסור.

---

## הגדרת כלי - defining a tool

כל כלי מוגדר עם שלושה שדות:

- **name** - שם ברור ותיאורי, למשל `get_weather`.
- **description** - הסבר מפורט מתי ולמה להשתמש בכלי. המודל נשען על זה מאוד.
- **input_schema** - סכמת JSON Schema שמתארת את הפרמטרים.

הנה כלי מזג אוויר לדוגמה:

```python
weather_tool = {
    "name": "get_weather",
    "description": "מחזיר את מזג האוויר הנוכחי בעיר נתונה. השתמש בכלי הזה כשהמשתמש שואל על מזג האוויר.",
    "input_schema": {
        "type": "object",
        "properties": {
            "city": {
                "type": "string",
                "description": "שם העיר, למשל 'תל אביב'",
            },
            "unit": {
                "type": "string",
                "enum": ["celsius", "fahrenheit"],
                "description": "יחידת הטמפרטורה",
            },
        },
        "required": ["city"],
    },
}
```

עצות חשובות: התיאור צריך להיות **מנחה** - לא רק "מה הכלי עושה" אלא "מתי לקרוא לו". שימוש ב-`enum` עבור ערכים מוגבלים מונע טעויות. ומסמנים ב-`required` רק את הפרמטרים שבאמת חובה.

---

## הבקשה הראשונה עם כלי - the first request

עכשיו נשלח בקשה שמצרפת את הכלי:

```python
import anthropic

client = anthropic.Anthropic()

weather_tool = { ... }  # כפי שהגדרנו למעלה

messages = [
    {"role": "user", "content": "מה מזג האוויר בתל אביב?"}
]

response = client.messages.create(
    model="claude-opus-4-8",
    max_tokens=1024,
    tools=[weather_tool],
    messages=messages,
)

print(response.stop_reason)  # ככל הנראה: "tool_use"
```

אם המודל החליט להשתמש בכלי, `stop_reason` יהיה `tool_use`, וה-`content` יכיל בלוק מסוג `tool_use`:

```python
for block in response.content:
    if block.type == "tool_use":
        print(block.name)   # "get_weather"
        print(block.input)  # {"city": "תל אביב", "unit": "celsius"}
        print(block.id)     # מזהה ייחודי - נצטרך אותו לתשובה
```

ה-`block.input` הוא כבר מילון (dict) מפוענח - **תמיד** ניגשים אליו כאובייקט, אף פעם לא מנתחים מחרוזת JSON ידנית.

---

## החזרת התוצאה - the tool result

עכשיו אנחנו מריצים את הפונקציה האמיתית ומחזירים את התוצאה. יש שלושה שלבים:

1. מוסיפים את תשובת המודל (עם בלוק ה-`tool_use`) להיסטוריה.
2. מריצים את הכלי בקוד שלנו.
3. מוסיפים הודעת `user` חדשה עם בלוק `tool_result` שמכיל את התוצאה - ו**חייבים** לכלול את `tool_use_id` שתואם לבלוק ה-`tool_use`.

```python
def run_get_weather(city, unit="celsius"):
    # כאן בפועל היינו קוראים ל-API של מזג אוויר. נדמה תוצאה:
    return f"בעיר {city} עכשיו 26 מעלות {unit} ובהיר."

# מוצאים את בלוק ה-tool_use
tool_block = next(b for b in response.content if b.type == "tool_use")

# מריצים את הכלי האמיתי
result_text = run_get_weather(**tool_block.input)

# מוסיפים את תור המודל להיסטוריה
messages.append({"role": "assistant", "content": response.content})

# מוסיפים את תוצאת הכלי כהודעת user
messages.append({
    "role": "user",
    "content": [{
        "type": "tool_result",
        "tool_use_id": tool_block.id,   # חייב להתאים!
        "content": result_text,
    }],
})

# שולחים שוב - עכשיו המודל ינסח תשובה סופית
final = client.messages.create(
    model="claude-opus-4-8",
    max_tokens=1024,
    tools=[weather_tool],
    messages=messages,
)

for block in final.content:
    if block.type == "text":
        print(block.text)   # "מזג האוויר בתל אביב הוא 26 מעלות ובהיר."
```

אם הכלי נכשל, מחזירים `tool_result` עם `"is_error": True` וטקסט שגיאה מסביר - המודל יבין ויתאושש.

---

## לולאת הסוכן - the agentic loop

בדוגמה למעלה טיפלנו בקריאה אחת. אבל המודל עשוי לרצות לקרוא לכמה כלים ברצף: קודם לבדוק מזג אוויר, ואז לפי התוצאה לחפש מסעדות. לכן בונים **לולאה** שממשיכה עד שהמודל מסיים (כלומר `stop_reason` הוא `end_turn` ולא `tool_use`):

```python
messages = [{"role": "user", "content": "מה מזג האוויר בתל אביב, וכדאי לצאת לטיול?"}]

while True:
    response = client.messages.create(
        model="claude-opus-4-8",
        max_tokens=1024,
        tools=[weather_tool],
        messages=messages,
    )

    # אם המודל סיים - יוצאים מהלולאה
    if response.stop_reason != "tool_use":
        break

    # מוסיפים את תור המודל
    messages.append({"role": "assistant", "content": response.content})

    # מריצים את כל הכלים שהמודל ביקש (יכולים להיות כמה!)
    tool_results = []
    for block in response.content:
        if block.type == "tool_use":
            output = run_get_weather(**block.input)
            tool_results.append({
                "type": "tool_result",
                "tool_use_id": block.id,
                "content": output,
            })

    # מחזירים את כל התוצאות בהודעת user אחת
    messages.append({"role": "user", "content": tool_results})

# התשובה הסופית
final_text = next(b.text for b in response.content if b.type == "text")
print(final_text)
```

שימו לב שכל תוצאות הכלים מאותו תור נשלחות ב**הודעת user אחת**. פיצול שלהן לכמה הודעות "מלמד" את המודל להפסיק לקרוא לכלים במקביל - טעות עדינה שכדאי להימנע ממנה.

---

## הרצת הכלים אוטומטית - the tool runner

לכתוב את הלולאה ידנית זה חינוכי, אבל ה-SDK מציע דרך נוחה יותר: **מריץ הכלים** (tool runner). בגרסת ה-beta של ה-SDK מגדירים כלי כפונקציה עם דקורטור, וה-SDK מנהל את כל הלולאה - קורא ל-API, מריץ את הפונקציות, מחזיר תוצאות, וחוזר עד שהמודל מסיים:

```python
import anthropic
from anthropic import beta_tool

client = anthropic.Anthropic()

@beta_tool
def get_weather(city: str, unit: str = "celsius") -> str:
    """מחזיר את מזג האוויר הנוכחי בעיר.

    Args:
        city: שם העיר, למשל תל אביב.
        unit: יחידת הטמפרטורה, celsius או fahrenheit.
    """
    return f"בעיר {city} עכשיו 26 מעלות {unit} ובהיר."

runner = client.beta.messages.tool_runner(
    model="claude-opus-4-8",
    max_tokens=1024,
    tools=[get_weather],
    messages=[{"role": "user", "content": "מה מזג האוויר בתל אביב?"}],
)

for message in runner:
    print(message)   # הלולאה מנוהלת אוטומטית עד סיום
```

היתרון: אין לולאה ידנית, סכמת הכלי נגזרת אוטומטית מחתימת הפונקציה ומה-docstring, והכל טיפוסי (typed). זו הדרך המומלצת לסוכן עם כלים מותאמים אישית. הלולאה הידנית נשארת שימושית כשצריכים שליטה מלאה שהמריץ לא חושף.

---

## אבטחה ואישור - security

כלים נותנים למודל כוח - וכוח דורש זהירות. אם לכלי יש תופעות לוואי (שליחת מייל, מחיקת נתונים, חיוב כספי), חייבים לשקול "שער אישור" (approval gate) לפני הביצוע. שתי גישות:

- **אימות קלט**: לפני שמריצים, בודקים שהקלט חוקי ובטוח.
- **אישור אנושי**: לפעולות הרסניות, עוצרים ומבקשים אישור מהמשתמש לפני שמריצים בפועל.

הכלל הזהב: המודל **מבקש**, אנחנו **מחליטים אם להריץ**. אף פעם לא מריצים באופן עיוור קלט שהגיע מהמודל, במיוחד לא פקודות מעטפת או שאילתות SQL שנבנו ממחרוזת.

---

## שליטה בבחירת כלי - tool choice

לפעמים אנחנו רוצים לשלוט מתי ואם המודל משתמש בכלים. השדה `tool_choice` נותן בדיוק את זה:

| ערך | התנהגות |
| --- | --- |
| `{"type": "auto"}` | המודל מחליט לבד אם להשתמש בכלי (ברירת מחדל) |
| `{"type": "any"}` | המודל **חייב** להשתמש בכלי כלשהו |
| `{"type": "tool", "name": "..."}` | המודל חייב להשתמש בכלי הספציפי |
| `{"type": "none"}` | המודל לא יכול להשתמש בכלים |

לדוגמה, אם אנחנו בונים מערכת שתמיד צריכה לחלץ נתונים דרך כלי מסוים, נכפה אותו:

```python
response = client.messages.create(
    model="claude-opus-4-8",
    max_tokens=1024,
    tools=[weather_tool],
    tool_choice={"type": "tool", "name": "get_weather"},   # חייב לקרוא לכלי הזה
    messages=[{"role": "user", "content": "מה שלומך?"}],
)
```

---

## כמה כלים במקביל - parallel tool use

כברירת מחדל, המודל יכול לבקש **כמה כלים בבת אחת** בתוך תשובה אחת. למשל, אם משתמש שואל על מזג האוויר בשלוש ערים, המודל עשוי להחזיר שלושה בלוקי `tool_use` בבת אחת. הלולאה שכתבנו כבר מטפלת בזה - היא עוברת על כל הבלוקים ואוספת את כל התוצאות להודעת `user` אחת.

זה חשוב לביצועים: אפשר להריץ את הכלים במקביל (concurrently) במקום בזה אחר זה. אבל הכלל שראינו נשאר קדוש - **כל** התוצאות חוזרות בהודעה **אחת**. אם רוצים לאסור על מקביליות (למשל כשהכלים משנים מצב משותף), אפשר להוסיף `"disable_parallel_tool_use": True` בתוך `tool_choice`.

---

## כלים בצד השרת - server-side tools

עד כה דיברנו על כלים ש**אנחנו** מריצים. אבל ל-Anthropic יש גם כלים שרצים **בצד השרת** שלה - אנחנו רק מכריזים עליהם, והמודל מריץ אותם לבד, בלי לולאת ביצוע אצלנו. הבולטים:

- **חיפוש באינטרנט** - המודל מחפש מידע עדכני מעבר לתאריך האימון.
- **הרצת קוד** - המודל מריץ קוד פייתון בארגז חול מבודד בצד השרת.

מכריזים עליהם בדיוק כמו כלי רגיל, אבל אין פונקציה לממש:

```python
response = client.messages.create(
    model="claude-opus-4-8",
    max_tokens=1024,
    tools=[{"type": "web_search_20260209", "name": "web_search"}],
    messages=[{"role": "user", "content": "מה החדשות האחרונות על Claude?"}],
)
```

התוצאות חוזרות כבלוקי תוכן באותה תשובה. זה נוח כשצריכים יכולת סטנדרטית (חיפוש, הרצת קוד) בלי לבנות אותה בעצמנו.

## הגבלת הלולאה - bounding the loop

הלולאה שכתבנו ממשיכה עד שהמודל מסיים. אבל מה אם המודל "נתקע" ומבקש כלים שוב ושוב בלי להגיע לתשובה? כדי להימנע מלולאה אינסופית, מגבילים את מספר האיטרציות:

```python
max_turns = 10
turns = 0

while turns < max_turns:
    response = client.messages.create(
        model="claude-opus-4-8",
        max_tokens=1024,
        tools=[weather_tool],
        messages=messages,
    )
    if response.stop_reason != "tool_use":
        break
    turns += 1
    # ... הרצת הכלים והוספת התוצאות ...

if turns >= max_turns:
    print("הסוכן חרג ממספר האיטרציות המרבי")
```

זו הגנה בסיסית וחשובה בכל מערכת ייצור. מגבלה של 10 עד 20 איטרציות מתאימה לרוב המשימות; אם מגיעים אליה, כנראה יש בעיה בהגדרת הכלים או בהנחיה. בכלים בצד השרת, אגב, קיים מצב מיוחד `pause_turn` - המודל עצר באמצע ריצה ארוכה ואפשר לחדש אותה על ידי שליחת הבקשה שוב.

## סיכום

בהרצאה הזו למדנו לתת למודל לא רק לדבר, אלא לפעול, דרך שימוש בכלים:

- **שימוש בכלים** מאפשר למודל לבקש שנריץ פונקציות שאנחנו מגדירים. המודל **לא** מריץ קוד - הוא רק מבקש, ואנחנו מבצעים.
- **הגדרת כלי** דורשת `name`, `description` (מנחה - מתי לקרוא לכלי) ו-`input_schema` בפורמט JSON Schema.
- **לולאת השימוש בכלים**: המודל מחזיר `stop_reason = "tool_use"` עם בלוק `tool_use`; אנחנו מריצים; מחזירים `tool_result` עם ה-`tool_use_id` התואם; המודל מנסח תשובה סופית.
- **לולאת הסוכן** ממשיכה עד `stop_reason = "end_turn"`. כל תוצאות הכלים מתור אחד נשלחות בהודעת `user` **אחת**.
- **מריץ הכלים** (`tool_runner`) של ה-SDK מנהל את הלולאה אוטומטית ומייצר סכמות מחתימות פונקציות - הדרך המומלצת.
- ה-`input` של הכלי הוא אובייקט מפוענח - ניגשים אליו כאובייקט, לא מנתחים JSON ידנית.
- **אבטחה**: מאמתים קלט, מגנים על פעולות הרסניות בשער אישור, ואף פעם לא מריצים עיוור קלט מהמודל.

עכשיו שאנחנו יודעים לבנות לולאת כלים בעצמנו, נעבור להרצאה הבאה - שם נכיר את ה-Agent SDK, ספרייה שנותנת לנו סוכן שלם עם כלים מובנים, בלי לכתוב את הלולאה בכלל.
