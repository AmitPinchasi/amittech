אחת המגבלות הבסיסיות של עבודה עם סוכן היא חלון ההקשר - context window. ככל שהשיחה מתארכת ומתמלאת בפלט של חיפושים, לוגים ותוכן קבצים, כך נשאר פחות מקום למה שבאמת חשוב, והסוכן נעשה איטי ומבולבל. בהרצאה הזו נכיר כלי מרכזי שמתמודד עם הבעיה: סוכני משנה - subagents. סוכן משנה הוא עוזר ייעודי שמקבל משימה תחומה, מבצע אותה בחלון הקשר נפרד משלו, ומחזיר רק את הסיכום. נבין מתי זה עוזר, מתי זה דווקא מזיק, איך מגדירים סוכן משנה מותאם, ואיך מריצים כמה במקביל.

## הרעיון המרכזי - context isolation

תארו לעצמכם שאתם באמצע משימה מורכבת, ואתם צריכים למצוא איפה בקוד מוגדרת פונקציה מסוימת. חיפוש כזה עלול לייצר עשרות תוצאות, קטעי קוד ולוגים - כל זה נכנס לשיחה הראשית וגודש אותה, למרות שאחרי שמצאתם את התשובה, רוב הפלט הזה כבר לא רלוונטי.

סוכן משנה פותר בדיוק את זה. במקום שהחיפוש ילכלך את השיחה שלכם, אתם מאצילים אותו לסוכן משנה. הוא עושה את כל העבודה המלוכלכת בחלון ההקשר **שלו**, ומחזיר לכם רק את המסקנה: "הפונקציה מוגדרת בקובץ X שורה Y". השיחה הראשית נשארת נקייה.

זה העיקרון החשוב ביותר להבנה: לכל סוכן משנה יש חלון הקשר טרי ומבודד. הוא לא רואה את היסטוריית השיחה שלכם, את הקבצים שכבר קראתם, או את הסוכני משנה שכבר הרצתם. הסוכן הראשי מרכיב עבורו הודעת האצלה שמסכמת את המשימה, וסוכן המשנה עובד ממנה.

```text
+---- השיחה הראשית (חלון הקשר יקר) ----+
|                                       |
|   "מצא איפה מוגדר handleLogin"        |
|         |                             |
|         v  האצלה                      |
|   +--- סוכן משנה (חלון נפרד) ---+     |
|   | חיפוש -> 40 תוצאות           |     |
|   | קריאת 8 קבצים                |     |
|   | ניתוח                        |     |
|   +--- מחזיר: "auth.ts:52" -----+     |
|         |                             |
|         v  רק הסיכום חוזר             |
|   "נמצא ב-auth.ts שורה 52"           |
+---------------------------------------+
```

## מה סוכני משנה נותנים - benefits

מעבר לשמירה על ההקשר, סוכני משנה נותנים כמה דברים:

- **שמירה על הקשר**: פלט מפורט נשאר אצל סוכן המשנה, רק הסיכום חוזר.
- **אכיפת מגבלות**: אפשר להגביל אילו כלים סוכן משנה יכול להשתמש בהם - למשל, סוכן חוקר שהוא קריאה בלבד ולא יכול לערוך קבצים.
- **התמחות**: פרומפט מערכת ממוקד לתחום מסוים, כמו סוקר קוד או מאתר באגים.
- **שליטה בעלות**: אפשר לנתב משימות למודל זול ומהיר יותר כמו Haiku.

## סוכני המשנה המובנים - built-in subagents

Claude Code מגיע עם כמה סוכני משנה מובנים שהסוכן הראשי משתמש בהם אוטומטית. שלושה עיקריים:

| סוכן | כלים | מטרה |
| ---- | ---- | ---- |
| Explore | קריאה בלבד (בלי Write/Edit) | חיפוש והבנה של בסיס קוד |
| Plan | קריאה בלבד | מחקר קוד לצורך בניית תוכנית |
| general-purpose | כל הכלים | משימות מורכבות שדורשות גם חקירה וגם שינוי |

הסוכן Explore הוא הדוגמה הנפוצה ביותר. כשאתם מבקשים למצוא משהו בקוד, Claude מאציל ל-Explore, וכך תוצאות החיפוש נשארות בהקשר נפרד. שימו לב לפרט חשוב: Explore ו-Plan מדלגים על קריאת קובצי CLAUDE.md ועל מצב ה-git, כדי להישאר מהירים וזולים. שאר סוכני המשנה טוענים את CLAUDE.md כרגיל.

## הגדרת סוכן משנה מותאם - custom subagents

מעבר למובנים, אפשר להגדיר סוכני משנה משלכם. סוכן משנה הוא פשוט קובץ Markdown עם כותרת YAML. המיקום קובע את הטווח:

```text
.claude/agents/       - סוכני פרויקט (נכנסים ל-git, משותפים לצוות)
~/.claude/agents/     - סוכנים אישיים (זמינים בכל הפרויקטים שלכם)
```

הנה דוגמה לקובץ סוכן משנה שסוקר קוד. שמים אותו למשל ב-`.claude/agents/code-reviewer.md`:

```markdown
---
name: code-reviewer
description: Reviews code for quality and best practices. Use proactively after code changes.
tools: Read, Glob, Grep
model: sonnet
---

You are a code reviewer. When invoked, analyze the code and provide
specific, actionable feedback on quality, security, and best practices.
Do not modify files. Report issues grouped by severity.
```

בואו נפרק את השדות. רק `name` ו-`description` הם חובה:

- **name**: מזהה ייחודי באותיות קטנות ומקפים.
- **description**: מתי Claude צריך להאציל לסוכן הזה. זה השדה הכי חשוב - כתבו אותו בבירור, ובמידת הצורך הוסיפו "use proactively" כדי לעודד האצלה.
- **tools**: אילו כלים מותרים. אם משמיטים, הסוכן יורש את כל הכלים. כאן הגבלנו לקריאה בלבד.
- **model**: `sonnet`, `opus`, `haiku`, מזהה מלא כמו `claude-opus-4-8`, או `inherit`. ברירת המחדל היא `inherit` - אותו מודל כמו השיחה הראשית.

הגוף של הקובץ (הטקסט אחרי ה-YAML) הוא פרומפט המערכת של סוכן המשנה. הוא מגדיר את ההתנהגות והטון. סוכן המשנה מקבל רק את הפרומפט הזה בתוספת פרטי סביבה בסיסיים, לא את כל פרומפט המערכת של Claude Code.

הדרך הקלה ביותר ליצור סוכן היא פשוט לבקש מ-Claude: "צור סוכן משנה אישי ב-`~/.claude/agents/` שסורק קבצים ומציע שיפורים, קריאה בלבד, על Sonnet". Claude יכתוב את הקובץ.

## איך מפעילים - invocation

יש שלוש דרכים להפעיל סוכן משנה, מהעדינה לחזקה:

- **שפה טבעית**: פשוט מזכירים את הסוכן בבקשה, ו-Claude מחליט אם להאציל. אין תחביר מיוחד.
  ```text
  השתמש בסוכן test-runner כדי לתקן את הבדיקות שנכשלות
  ```
- **אזכור עם @**: מקלידים `@` ובוחרים את הסוכן מהרשימה. זה מבטיח שהסוכן הספציפי ירוץ, ולא משאיר את הבחירה ל-Claude.
- **סשן שלם**: הדגל `--agent <name>` מפעיל סשן שבו כל השיחה משתמשת בפרומפט ובמודל של אותו סוכן.
  ```bash
  claude --agent code-reviewer
  ```

בנוסף, יש האצלה אוטומטית: Claude קורא את שדה ה-`description` של כל סוכן ומחליט בעצמו מתי להאציל, לפי הבקשה שלכם וההקשר.

## עבודה במקביל - parallel work

אחד השימושים החזקים ביותר הוא הרצת כמה סוכני משנה **במקביל** לחקירות עצמאיות. אם צריך לחקור שלושה מודולים שלא תלויים זה בזה, אפשר לבקש:

```text
חקור את מודולי האימות, מסד הנתונים וה-API במקביל, כל אחד בסוכן משנה נפרד
```

כל סוכן חוקר את התחום שלו בו-זמנית, ואז Claude מסנתז את הממצאים. זה עובד הכי טוב כשנתיבי החקירה אינם תלויים זה בזה. חשוב לדעת: החל מגרסאות אחרונות, סוכני משנה רצים ברקע כברירת מחדל, כך שאתם יכולים להמשיך לעבוד בזמן שהם עובדים.

אפשר גם לשרשר סוכנים: "השתמש בסוקר הקוד כדי למצוא בעיות ביצועים, ואז בסוכן האופטימיזציה כדי לתקן אותן". כל סוכן מסיים ומעביר תוצאות ל-Claude, שמעביר הקשר רלוונטי לסוכן הבא.

## מתי זה מזיק - when it hurts

סוכני משנה הם כלי חזק, אבל לא תשובה לכל דבר. הנה מתי **לא** להשתמש:

- **משימה שדורשת הלוך-ושוב תכוף**: אם צריך דיוק ותיקון איטרטיביים, השיחה הראשית עדיפה. סוכן משנה מתחיל טרי ולוקח לו זמן לאסוף הקשר.
- **שלבים שחולקים הרבה הקשר**: תכנון, מימוש ובדיקה שקשורים זה בזה - עדיף לעשות בשיחה אחת, אחרת כל סוכן צריך לקבל את כל הרקע מחדש.
- **שינוי קטן וממוקד**: לא שווה את התקורה של הפעלת סוכן חדש.

ויש מלכוד עדין וחשוב: **כשסוכני משנה מסיימים, התוצאות שלהם חוזרות לשיחה הראשית**. אם מריצים הרבה סוכני משנה שכל אחד מחזיר תוצאה מפורטת, גם זה גודש את ההקשר. הרווח מגיע דווקא כשסוכן המשנה בולע פלט גדול ומחזיר סיכום קצר. אם הוא מחזיר סיכום ארוך, ביטלתם את היתרון.

הכלל המנחה: השתמשו בסוכן משנה כשהמשימה מייצרת פלט מפורט שאתם לא צריכים בהקשר הראשי, כשאתם רוצים לאכוף הגבלות כלים, או כשהעבודה עצמאית ויכולה להחזיר סיכום.

## בחירת מודל לפי עלות - choosing a model

שדה ה-`model` הוא כלי חשוב לשליטה בעלות. לא כל משימה צריכה את המודל החזק ביותר. הנה השיקול:

```text
opus     - המודל החזק, למשימות מורכבות שדורשות שיפוט עמוק
sonnet   - איזון טוב בין יכולת למהירות, ברירת מחדל טובה לרוב
haiku    - מהיר וזול, למשימות פשוטות וחזרתיות
inherit  - אותו מודל כמו השיחה הראשית (ברירת המחדל)
```

לסוכן חיפוש שרק סורק קבצים ומחזיר מיקום, Haiku לרוב מספיק ועולה הרבה פחות. לסוכן שסוקר ארכיטקטורה ומחליט החלטות עיצוב, שווה Sonnet או Opus. הדפוס: נתבו משימות רועשות אך פשוטות למודל זול, ושמרו את המודל היקר לחשיבה. אפשר גם להשתמש במזהה מלא כמו `claude-haiku-4-5-20251001` אם רוצים לקבע גרסה מדויקת.

## המשך עבודה של סוכן - resuming

כל הפעלה של סוכן משנה יוצרת מופע חדש עם הקשר טרי. אבל לפעמים רוצים להמשיך עבודה של סוכן במקום להתחיל מחדש. אפשר לבקש מ-Claude להמשיך:

```text
השתמש בסוקר הקוד כדי לסקור את מודול האימות
[הסוכן מסיים]

המשך את הסקירה ההיא, ועכשיו נתח גם את לוגיקת ההרשאות
[Claude ממשיך את אותו סוכן עם ההקשר המלא מקודם]
```

הסוכן שממשיך שומר את כל היסטוריית השיחה שלו - קריאות כלים, תוצאות ושיקולים. הוא ממשיך בדיוק מהמקום שבו עצר. שימו לב: סוכני Explore ו-Plan הם חד-פעמיים ולא ניתן להמשיך אותם; למשימות שצריך להמשיך, השתמשו ב-general-purpose או בסוכן מותאם.

## הגבלת כלים והרשאות - restricting tools

כמו שראינו בדוגמה, אפשר להגביל את הכלים. יש שתי דרכים:

```yaml
# רשימת היתר: רק הכלים האלה מותרים
tools: Read, Grep, Glob, Bash
```

```yaml
# רשימת חסימה: יורש הכל חוץ מהכלים האלה
disallowedTools: Write, Edit
```

זו שכבת הגנה שימושית. סוכן חוקר שהוא קריאה בלבד לא יכול לשנות קבצים בטעות, גם אם משהו בתוכן שהוא קרא ניסה להטות אותו. בפרק על MCP הזכרנו הזרקת פרומפט - הגבלת כלים לסוכן משנה היא אחת הדרכים להכיל את הסיכון הזה.

## סיכום

בהרצאה הזו למדנו:

- **סוכן משנה - subagent** הוא עוזר שמקבל משימה תחומה, מבצע אותה ב**חלון הקשר טרי ומבודד** משלו, ומחזיר רק את הסיכום. כך פלט מפורט לא גודש את השיחה הראשית.
- כל סוכן משנה מתחיל נקי: הוא לא רואה את היסטוריית השיחה או הקבצים שכבר נקראו, אלא מקבל הודעת האצלה שמסכמת את המשימה.
- Claude Code כולל סוכנים מובנים: **Explore** ו-**Plan** (קריאה בלבד, לחקירה), ו-**general-purpose** (כל הכלים). Explore ו-Plan מדלגים על CLAUDE.md כדי להישאר מהירים.
- סוכן משנה מותאם הוא קובץ **Markdown עם YAML** ב-`.claude/agents/` (פרויקט) או `~/.claude/agents/` (אישי). שדות חובה: `name` ו-`description`. שדות נפוצים: `tools`, `model`. הגוף הוא פרומפט המערכת.
- מפעילים בשלוש דרכים: שפה טבעית, אזכור עם `@`, או `--agent` לסשן שלם. יש גם האצלה אוטומטית לפי ה-`description`.
- אפשר להריץ כמה סוכנים **במקביל** לחקירות עצמאיות, ולשרשר אותם לזרימות רב-שלביות.
- זה **מזיק** כשהמשימה דורשת הלוך-ושוב תכוף, כשהשלבים חולקים הרבה הקשר, או כשמריצים הרבה סוכנים שכל אחד מחזיר תוצאה ארוכה - כי גם התוצאות החוזרות גודשות את ההקשר.
- הגבלת כלים (`tools` או `disallowedTools`) היא שכבת הגנה: סוכן קריאה בלבד לא יכול לשנות קבצים, מה שמכיל סיכוני הזרקת פרומפט.
