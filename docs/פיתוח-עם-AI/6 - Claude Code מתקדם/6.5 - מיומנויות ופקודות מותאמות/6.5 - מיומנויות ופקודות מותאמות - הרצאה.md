כשעובדים עם סוכן לאורך זמן, מגלים שיש זרימות עבודה שחוזרות על עצמן. בכל פעם שרוצים לפרוס, מדביקים לסוכן את אותה רשימת שלבים. בכל פעם שכותבים endpoint חדש, מזכירים את אותן מוסכמות. זה בזבזני ולא עקבי - לפעמים שוכחים שלב, לפעמים מנסחים אחרת. בהרצאה הזו נכיר את הכלי שפותר את זה: מיומנויות - skills, ופקודות מותאמות - custom commands. מיומנות היא אריזה של זרימת עבודה או ידע לשימוש חוזר, שהסוכן מפעיל בעקביות. נבין את מבנה הקובץ, איך מיומנות מתגלה ומופעלת, ואיך היא שונה מסוכן משנה ומ-CLAUDE.md.

## מהי מיומנות - what is a skill

מיומנות היא קובץ בשם `SKILL.md` שמכיל הוראות. אתם יוצרים אותו פעם אחת, ו-Claude מוסיף אותו ל"ארגז הכלים" שלו. אפשר להפעיל אותו בשתי דרכים: או ש-Claude טוען אותו אוטומטית כשהוא רלוונטי, או שאתם מפעילים אותו ישירות בפקודת slash - למשל `/deploy`.

מתי כדאי ליצור מיומנות? כשאתם מוצאים את עצמכם מדביקים שוב ושוב את אותן הוראות, רשימת בדיקה או תהליך רב-שלבי. או כשקטע ב-CLAUDE.md גדל מ"עובדה" ל"תהליך". היתרון המרכזי על פני CLAUDE.md: תוכן המיומנות **נטען רק כשמשתמשים בו**. חומר עזר ארוך כמעט לא עולה בהקשר עד הרגע שבו הוא נחוץ.

## מבנה הקובץ - SKILL.md

כל מיומנות היא תיקייה עם קובץ `SKILL.md` בתוכה. שם התיקייה הוא הפקודה שתקלידו. הקובץ בנוי משני חלקים: כותרת YAML בין קווי `---` שאומרת ל-Claude מתי להשתמש במיומנות, ותוכן Markdown עם ההוראות עצמן.

הנה דוגמה מלאה. נניח שאנחנו רוצים מיומנות שמסכמת שינויים לא-מקומיטים ומדגישה סיכונים. שומרים אותה ב-`~/.claude/skills/summarize-changes/SKILL.md`:

```yaml
---
description: מסכמת שינויים לא מקומיטים ומדגישה סיכונים. השתמש כשהמשתמש שואל מה השתנה, רוצה הודעת commit, או מבקש לסקור את ה-diff.
---

## השינויים הנוכחיים

!`git diff HEAD`

## הוראות

סכם את השינויים למעלה בשתיים-שלוש נקודות, ואז מנה כל סיכון שאתה מבחין בו:
טיפול חסר בשגיאות, ערכים קשיחים, או בדיקות שצריך לעדכן. אם ה-diff ריק,
אמור שאין שינויים לא מקומיטים.
```

שימו לב לשורה `` !`git diff HEAD` ``. זו תכונה חזקה שנרחיב עליה - **הזרקת הקשר דינמי**. אבל קודם, מיקום הקבצים.

## איפה מיומנויות חיות - where skills live

בדומה לסוכני משנה, המיקום קובע את הטווח:

| מיקום | נתיב | חל על |
| ----- | ---- | ----- |
| אישי - personal | `~/.claude/skills/<name>/SKILL.md` | כל הפרויקטים שלכם |
| פרויקט - project | `.claude/skills/<name>/SKILL.md` | הפרויקט הזה בלבד |
| תוסף - plugin | `<plugin>/skills/<name>/SKILL.md` | היכן שהתוסף מופעל |

מיומנות של פרויקט נכנסת ל-git ומשותפת לצוות. כשמיומנויות חולקות שם ברמות שונות, אישי גובר על פרויקט. הפקודה שתקלידו נגזרת משם התיקייה: מיומנות ב-`.claude/skills/deploy-staging/` תופעל עם `/deploy-staging`.

## שדות הכותרת - frontmatter

כל השדות אופציונליים, אבל `description` מומלץ מאוד כדי ש-Claude ידע מתי להשתמש במיומנות. הנה השדות החשובים:

```yaml
---
name: my-skill
description: מה המיומנות עושה ומתי להשתמש בה
disable-model-invocation: true
allowed-tools: Bash(git add *) Bash(git commit *)
---
```

- **description**: מה המיומנות עושה ומתי. זה מה ש-Claude קורא כדי להחליט אם לטעון אותה אוטומטית. שימו את מקרה השימוש העיקרי ראשון.
- **disable-model-invocation**: אם `true`, רק אתם יכולים להפעיל את המיומנות (עם `/name`), ו-Claude לא יטען אותה לבד. שימושי לזרימות עם תופעות לוואי כמו `/deploy` או `/commit` - לא רוצים ש-Claude יחליט לפרוס כי הקוד "נראה מוכן".
- **allowed-tools**: כלים ש-Claude יכול להשתמש בהם בלי לבקש אישור כשהמיומנות פעילה. זה **לא** מגביל אילו כלים זמינים, אלא מאשר מראש את אלה שברשימה.

## שני סוגי תוכן - content types

כדאי לחשוב על מיומנות באחד משני אופנים:

- **תוכן ידע - reference**: מוסיף ידע ש-Claude מיישם על העבודה הנוכחית. מוסכמות, סגנון, ידע תחום. זה רץ בתוך השיחה כדי שהמודל ישתמש בו לצד ההקשר.
  ```yaml
  ---
  name: api-conventions
  description: מוסכמות תכנון API בבסיס הקוד הזה
  ---

  בכתיבת endpoints:
  - השתמש במוסכמות RESTful
  - החזר פורמט שגיאה עקבי
  - כלול ולידציה של הבקשה
  ```
- **תוכן משימה - task**: נותן ל-Claude שלבים לפעולה ספציפית, כמו פריסה או commit. אלה בדרך כלל דברים שתרצו להפעיל ישירות עם `/name`.

## פקודות מותאמות - custom commands

נקודה חשובה שכדאי להבין: **פקודות מותאמות מוזגו לתוך מיומנויות**. קובץ ב-`.claude/commands/deploy.md` וקובץ ב-`.claude/skills/deploy/SKILL.md` שניהם יוצרים את הפקודה `/deploy` ועובדים אותו דבר. קבצים ישנים ב-`.claude/commands/` ממשיכים לעבוד, אבל מיומנויות מוסיפות יכולות: תיקייה עם קבצי עזר, שליטה במי מפעיל, וטעינה אוטומטית.

לכן, מי שראה בעבר "custom commands" - זה אותו רעיון, וההמלצה היום היא להשתמש במבנה של מיומנות.

## העברת ארגומנטים - arguments

מיומנות יכולה לקבל ארגומנטים דרך ה-placeholder `$ARGUMENTS`. הנה מיומנות שמתקנת issue לפי מספר:

```yaml
---
name: fix-issue
description: תקן issue ב-GitHub
disable-model-invocation: true
---

תקן את issue מספר $ARGUMENTS ב-GitHub לפי תקני הקוד שלנו.

1. קרא את תיאור ה-issue
2. הבן את הדרישות
3. ממש את התיקון
4. כתוב בדיקות
5. צור commit
```

כשמריצים `/fix-issue 123`, ה-placeholder מוחלף ב-`123`. אפשר גם לגשת לארגומנטים לפי מיקום עם `$ARGUMENTS[0]` או הקיצור `$0`, `$1` וכן הלאה.

## הזרקת הקשר דינמי - dynamic context

נחזור לתכונה שראינו קודם. התחביר `` !`<command>` `` מריץ פקודת מעטפת **לפני** שתוכן המיומנות נשלח ל-Claude, ומחליף את ה-placeholder בפלט. כך Claude מקבל נתונים אמיתיים, לא את הפקודה.

```yaml
---
name: pr-summary
description: סכם שינויים ב-pull request
allowed-tools: Bash(gh *)
---

## הקשר ה-PR
- diff: !`gh pr diff`
- תגובות: !`gh pr view --comments`
- קבצים שהשתנו: !`gh pr diff --name-only`

## המשימה שלך
סכם את ה-pull request...
```

כשהמיומנות רצה, כל `` !`...` `` מורץ מיד, הפלט מוחלף במקום ה-placeholder, ו-Claude מקבל פרומפט מלא עם נתוני ה-PR האמיתיים. זו הכנה מקדימה, לא משהו ש-Claude מריץ. הוא רואה רק את התוצאה הסופית.

## קבצי עזר - supporting files

מיומנות היא תיקייה, אז אפשר לצרף לה קבצים נוספים: תבניות, דוגמאות, וסקריפטים. זה שומר על `SKILL.md` ממוקד, כי חומר עזר גדול נטען רק כשצריך אותו:

```text
my-skill/
├── SKILL.md          # הוראות עיקריות (חובה)
├── reference.md      # תיעוד מפורט - נטען בעת הצורך
└── scripts/
    └── helper.py     # סקריפט להרצה, לא לטעינה
```

מפנים לקבצים האלה מתוך `SKILL.md` כדי ש-Claude ידע מה כל קובץ מכיל ומתי לטעון אותו. כלל אצבע: שמרו את `SKILL.md` מתחת ל-500 שורות, והעבירו חומר מפורט לקבצים נפרדים.

## מחזור החיים של התוכן - content lifecycle

כדאי להבין מה קורה כשמיומנות נטענת, כי זה משפיע על איך כותבים אותה. כשאתם או Claude מפעילים מיומנות, התוכן המעובד של `SKILL.md` נכנס לשיחה כהודעה אחת, ו**נשאר שם לשארית הסשן**. Claude Code לא קורא מחדש את הקובץ בכל תור.

לכן, כתבו הוראות שאמורות לחול לאורך המשימה כהנחיות עומדות, לא כשלב חד-פעמי. אם המיומנות אומרת "עצב את הקוד ככה", זה יחול על כל הקוד מכאן והלאה. שימו לב גם לעלות: כל שורה במיומנות היא עלות הקשר חוזרת, אז שמרו על הגוף תמציתי. אמרו **מה** לעשות, לא הרצאה על **למה**.

עוד נקודה: כשמפעילים מיומנות שכבר טעונה עם אותו תוכן, Claude Code רק מוסיף הערה קצרה שהיא כבר טעונה, במקום עותק שני. אבל אם הארגומנטים השתנו או שהזרקת הקשר דינמי הפיקה פלט חדש, התוכן המלא נטען שוב.

## בקרת מי מפעיל - who invokes

ראינו את `disable-model-invocation`, אבל כדאי לראות את התמונה המלאה. שני שדות שולטים במי יכול להפעיל מיומנות:

| כותרת | אתם מפעילים | Claude מפעיל | מתי נטען להקשר |
| ----- | ----------- | ------------ | -------------- |
| (ברירת מחדל) | כן | כן | התיאור תמיד בהקשר, התוכן בהפעלה |
| `disable-model-invocation: true` | כן | לא | התיאור לא בהקשר, התוכן בהפעלה שלכם |
| `user-invocable: false` | לא | כן | התיאור תמיד בהקשר |

השורה השנייה, `disable-model-invocation`, מתאימה לזרימות עם תופעות לוואי - `/deploy`, `/commit`, `/send-slack-message`. לא רוצים ש-Claude יחליט לפרוס לבד. השורה השלישית, `user-invocable: false`, מתאימה לידע רקע שלא הגיוני להפעיל כפקודה - למשל מיומנות שמסבירה איך מערכת ישנה עובדת. Claude צריך לדעת את זה כשרלוונטי, אבל אין טעם ש-`/legacy-context` יהיה פקודה שמשתמש מקליד.

## מיומנויות מובנות - bundled skills

Claude Code מגיע עם מיומנויות מובנות שזמינות בכל סשן. כמה שכדאי להכיר:

```text
/doctor        - בדיקת תקינות של ההגדרות
/code-review   - סקירת קוד של השינויים
/debug         - עזרה בניפוי באגים
/run           - הפעלת האפליקציה לראות שינוי עובד
/verify        - בנייה והרצה לאימות שהשינוי עושה את מה שצריך
```

אלה מיומנויות מבוססות-פרומפט: הן נותנות ל-Claude הוראות מפורטות ומתזמרות את העבודה עם הכלים שלו. מיומנות שלכם עם אותו שם דורסת מובנית - למשל `code-review` ב-`.claude/skills/` שלכם מחליף את המובנה. זו דרך נוחה להתאים התנהגות מובנית לצוות שלכם.

## מיומנות מול סוכן משנה - skill vs subagent

חשוב לא להתבלבל בין השניים. מיומנות רצה כברירת מחדל **בתוך השיחה הראשית** - התוכן שלה נכנס להקשר ומשפיע על הסוכן הנוכחי. סוכן משנה, לעומת זאת, רץ בהקשר **מבודד** ומחזיר סיכום.

הכלל: השתמשו במיומנות כשאתם רוצים פרומפט או זרימה לשימוש חוזר שתרוץ בהקשר הנוכחי. השתמשו בסוכן משנה כשאתם רוצים לבודד עבודה רועשת. אגב, אפשר לשלב: מיומנות עם `context: fork` בכותרת תרוץ דווקא בסוכן משנה מבודד.

## סיכום

בהרצאה הזו למדנו:

- **מיומנות - skill** היא אריזה של זרימת עבודה או ידע לשימוש חוזר, בקובץ `SKILL.md`. היתרון על CLAUDE.md: התוכן נטען רק כשמשתמשים בו, כך שחומר עזר ארוך כמעט לא עולה בהקשר.
- כל מיומנות היא תיקייה עם `SKILL.md`. שם התיקייה הוא הפקודה. המבנה: כותרת YAML (מתי להשתמש) ותוכן Markdown (ההוראות).
- מיקום קובע טווח: `~/.claude/skills/` (אישי), `.claude/skills/` (פרויקט, נכנס ל-git). אישי גובר על פרויקט.
- שדות מרכזיים: `description` (מתי להשתמש), `disable-model-invocation: true` (רק אתם מפעילים, לזרימות עם תופעות לוואי), ו-`allowed-tools` (אישור מראש לכלים).
- **פקודות מותאמות מוזגו למיומנויות**: קובץ ב-`.claude/commands/deploy.md` וב-`.claude/skills/deploy/SKILL.md` שניהם יוצרים `/deploy`.
- ארגומנטים מועברים דרך `$ARGUMENTS` (או `$0`, `$1` לפי מיקום). **הזרקת הקשר דינמי** עם `` !`<command>` `` מריצה פקודה מראש ומזריקה את הפלט לפרומפט.
- אפשר לצרף **קבצי עזר** (תבניות, סקריפטים) לתיקיית המיומנות; שמרו את `SKILL.md` ממוקד.
- הבחנה מסוכן משנה: מיומנות רצה **בהקשר הראשי** ומשפיעה עליו; סוכן משנה רץ **מבודד** ומחזיר סיכום.
