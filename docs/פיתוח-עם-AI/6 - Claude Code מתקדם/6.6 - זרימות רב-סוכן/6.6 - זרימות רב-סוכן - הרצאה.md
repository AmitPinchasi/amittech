הגענו להרצאה האחרונה של הפרק, וכאן נחבר את כל מה שלמדנו. ראינו כלים חיצוניים דרך MCP, סוכני משנה שמבודדים עבודה, הוקים שאוכפים משמעת, ומיומנויות שאורזות זרימות. עכשיו נשאל את השאלה הגדולה: איך מנצחים על משימה שגדולה מדי בשביל סוכן אחד? התשובה היא זרימות רב-סוכן - orchestration של כמה סוכנים שעובדים יחד. נדבר על תכנון-ואז-ביצוע בקנה מידה גדול, על נקודות ביקורת לסקירה, ועל הסיכונים שבאוטונומיה - ואיך נשארים בשליטה. זו הרצאה על שיטת עבודה יותר מאשר על תחביר.

## למה סוכן אחד לא מספיק - the scaling problem

סוכן אחד עובד מצוין על משימה ממוקדת. אבל כשהמשימה גדלה - "בנה מערכת שלמה עם צד לקוח, צד שרת ובדיקות" - נתקלים בשתי מגבלות. הראשונה היא **חלון ההקשר**: משימה גדולה מייצרת כל כך הרבה קוד, לוגים והחלטות, שההקשר מתמלא והסוכן מאבד את החוט. השנייה היא **מיקוד**: סוכן שמנסה להחזיק בראש את כל המערכת בו-זמנית עושה יותר טעויות מסוכן שמתמקד בחלק אחד.

זרימה רב-סוכן פותרת את שתי המגבלות בעיקרון אחד: **חלק וכבוש**. מפרקים את המשימה הגדולה לתת-משימות עצמאיות, כל אחת מקבלת סוכן משנה עם הקשר טרי, וסוכן מתזמר (orchestrator) מרכיב את התוצאות.

## המבנה הבסיסי - orchestrator and workers

הדפוס הנפוץ הוא מתזמר ועובדים. הסוכן הראשי הוא המתזמר: הוא מחזיק את התמונה הגדולה, מחלק עבודה, ומסנתז. העובדים הם סוכני משנה, כל אחד עם משימה תחומה.

```text
+------------------ מתזמר (Orchestrator) ------------------+
|  מחזיק את התוכנית הכוללת ואת התמונה הגדולה               |
|                                                          |
|   +--- Worker A ---+  +--- Worker B ---+  +-- Worker C --+
|   | מודול אימות    |  | מודול DB       |  | מודול API    |
|   | הקשר טרי       |  | הקשר טרי       |  | הקשר טרי     |
|   +----------------+  +----------------+  +--------------+
|          |                   |                   |       |
|          v                   v                   v       |
|        סיכום               סיכום               סיכום     |
|                                                          |
|   המתזמר מסנתז את שלושת הסיכומים לתמונה אחת              |
+----------------------------------------------------------+
```

היתרון: הפלט המפורט של כל עובד נשאר בהקשר שלו, ורק הסיכום חוזר למתזמר. כך המתזמר שומר על ראש נקי גם כשמתחת הפכה עבודה עצומה. זה בדיוק העיקרון של סוכני משנה שראינו בהרצאה 6.3, מיושם בקנה מידה.

## תכנון-ואז-ביצוע - plan then execute

הטעות הנפוצה ביותר בזרימה גדולה היא לקפוץ ישר לביצוע. סוכן שמתחיל לכתוב קוד בלי תוכנית מייצר בלגן שקשה לתקן. הגישה הנכונה מפרידה בין שני שלבים:

- **שלב התכנון**: קודם, הסוכן חוקר ומייצר תוכנית - אילו תת-משימות יש, באיזה סדר, ומה התלויות ביניהן. בשלב הזה אף שורת קוד לא נכתבת. Claude Code תומך בזה עם מצב תכנון (plan mode), שבו הסוכן חוקר בקריאה בלבד ומציג תוכנית לפני שהוא נוגע בקוד.
- **שלב הביצוע**: רק אחרי שאישרתם את התוכנית, הסוכן מבצע אותה, תת-משימה אחר תת-משימה.

ההפרדה הזו קריטית בקנה מידה גדול. תוכנית כתובה היא **חוזה** שאתם יכולים לבדוק לפני שמושקעת עבודה. הרבה יותר זול לתקן שורה בתוכנית מאשר למחוק יום של קוד שנכתב לפי הבנה שגויה.

דפוס מומלץ: בקשו מהסוכן לכתוב את התוכנית לקובץ (למשל `PLAN.md`), עברו עליה, תקנו, ורק אז אמרו לו להתחיל. כשהתוכנית בקובץ, גם סוכני המשנה יכולים לקרוא ממנה את החלק שלהם.

## נקודות ביקורת - review checkpoints

זרימה רב-סוכן בלי בקרה היא מתכון לצרות. אתם עלולים לחזור אחרי שעה ולגלות שהסוכן בנה את הדבר הלא נכון, ביעילות מרשימה. הפתרון הוא **נקודות ביקורת** - עצירות מכוונות שבהן אתם (או סוכן ייעודי) בודקים את העבודה לפני שממשיכים.

יש שתי דרכים לנקודות ביקורת:

- **ביקורת אנושית**: אחרי כל תת-משימה גדולה, הסוכן עוצר ומציג מה עשה. אתם מאשרים או מתקנים. זה איטי יותר אבל בטוח.
- **ביקורת אוטומטית**: סוכן משנה ייעודי לסקירה (למשל `code-reviewer` מהרצאה 6.3) בודק את העבודה של סוכן אחר. שילוב חזק: עובד מבצע, סוקר בודק, והמתזמר מחליט אם להמשיך.

אפשר לאכוף נקודות ביקורת גם עם הוקים. הוק על אירוע `Stop` שמריץ בדיקות ולא נותן לסוכן לסיים כל עוד הן אדומות, הוא נקודת ביקורת אוטומטית שאי אפשר לדלג עליה. כאן כל הכלים של הפרק מתחברים: MCP מביא נתונים, סוכני משנה מבודדים עבודה, הוקים אוכפים, ומיומנויות מתקננות את התהליך.

## דוגמה: בניית פיצ'ר - worked example

בואו נראה איך זה נראה בפועל עבור פיצ'ר בינוני. נניח "הוסף מערכת התראות למשתמשים":

```text
1. תכנון (plan mode)
   הסוכן חוקר את בסיס הקוד, כותב PLAN.md:
   - schema של התראות ב-DB
   - endpoint ליצירה ולקריאה
   - רכיב UI
   - בדיקות
   -> אתם עוברים על התוכנית ומאשרים

2. ביצוע במקביל (סוכני משנה)
   - Worker A: מוסיף את ה-schema וה-migration
   - Worker B: כותב את ה-endpoints
   (B תלוי ב-A, אז רץ אחריו; UI אפשר במקביל)

3. נקודת ביקורת
   - סוכן code-reviewer סוקר את הקוד של A ו-B
   - הוק Stop מריץ את הבדיקות

4. סינתזה
   המתזמר מרכיב, מריץ בדיקות אינטגרציה, ומדווח לכם
```

שימו לב לתלויות: לא הכל רץ במקביל. תת-משימות תלויות רצות בסדר, ורק עצמאיות רצות יחד. זיהוי התלויות הוא בדיוק מה שקורה בשלב התכנון.

## סוכני משנה מול צוותי סוכנים - subagents vs teams

כדאי להבחין בין שתי צורות של ריבוי סוכנים ב-Claude Code, כי הן מתאימות למשימות שונות:

- **סוכני משנה - subagents**: עובדים בתוך סשן אחד. הסוכן הראשי מאציל, מקבל סיכומים, ומסנתז. זה מה שראינו בהרצאה 6.3, וזה המנגנון הבסיסי לרוב הזרימות.
- **צוותי סוכנים וסוכני רקע - agent teams**: לריבוי סשנים עצמאיים שרצים במקביל, כל אחד עם ההקשר שלו, שאפשר לנטר ממקום אחד. זה מתאים למקבילות מתמשכת שחורגת מחלון הקשר בודד.

לרוב המשימות, סוכני משנה בסשן אחד מספיקים. כשהמשימה כל כך גדולה שאפילו המתזמר לא יכול להחזיק את הסינתזה, שוקלים צוותי סוכנים. הכלל: התחילו מהפשוט, ושדרגו רק כשנתקלים במגבלה אמיתית.

נקודת מפתח לגבי תקשורת בין סוכנים: החל מגרסאות אחרונות, סוכן משנה מתייחס להודעות מהסוכן ששיגר אותו כהכוונת משימה רגילה, כולל תיקוני מסלול תוך כדי. אבל שתי מגבלות תמיד בתוקף - שום הודעה מסוכן אחר אינה נחשבת לאישור הרשאה מטעמכם, ושום הודעה מסוכן לא יכולה לשנות את הגדרות ההרשאה או ה-CLAUDE.md. רק מערכת ההרשאות או ההודעות שלכם מעניקות אישור. זו הגנה חשובה: גם בזרימה אוטונומית, אתם נשארים השער האחרון להחלטות רגישות.

## עומק ההאצלה - nesting depth

סוכן משנה יכול בעצמו לשגר סוכני משנה, וכך נבנה עץ. זה שימושי כשתת-משימה מתפצלת בעצמה - למשל סוקר שמשגר מאמת נפרד לכל ממצא. הפלט של הרמות הפנימיות לא מגיע לשיחה הראשית, רק הסיכום של הסוכן העליון חוזר אליכם.

יש מגבלת עומק קבועה על העץ הזה, כדי למנוע רקורסיה אינסופית. מבחינה מעשית, נדיר להזדקק ליותר משתיים-שלוש רמות. עץ עמוק מדי הוא בדרך כלל סימן שהמשימה לא פורקה נכון - עדיף לשטח אותה לתת-משימות מקבילות מאשר לקנן לעומק.

## הסיכונים - the risks

כוח מביא אחריות. זרימה רב-סוכן אוטונומית טומנת סיכונים אמיתיים שחייבים להכיר:

- **סחף ממטרה - drift**: ככל שהסוכן רץ יותר זמן בלי בקרה, כך גדל הסיכוי שהוא ייסחף מהכוונה המקורית. תת-משימה שהובנה מעט שגוי מתפשטת לכל מה שבא אחריה.
- **הגברת שגיאות - error amplification**: אם המתזמר מעביר לעובד הקשר שגוי, כל העובדים בונים על בסיס פגום. שגיאה אחת בתכנון הופכת לעשר שגיאות בביצוע.
- **גודש בהקשר**: אירוני, אבל אם כל סוכן משנה מחזיר סיכום ארוך, ההקשר של המתזמר מתמלא בדיוק כמו שרצינו למנוע. סיכומים חייבים להיות קצרים.
- **עלות ואיבוד שקיפות**: הרבה סוכנים במקביל עולים יותר, וקשה יותר לעקוב מה כל אחד עושה. קל לאבד את התחושה מה בדיוק קורה.

## איך נשארים בשליטה - staying in control

הנה הכללים המעשיים לשמור על שליטה בזרימה רב-סוכן:

- **תכננו לפני שמבצעים**, תמיד. אשרו את התוכנית בכתב.
- **הגבילו את היקף כל סוכן משנה**. משימה תחומה עם הקשר ברור נסחפת פחות.
- **הגבילו כלים**. סוכן חוקר צריך להיות קריאה בלבד. תנו הרשאות כתיבה רק למי שחייב.
- **הכניסו נקודות ביקורת**. אל תתנו לזרימה לרוץ שעה בלי עצירה. עדיף לבדוק אחרי כל שלב.
- **דרשו סיכומים קצרים**. אמרו במפורש לסוכני המשנה להחזיר מסקנה תמציתית, לא דוח מלא.
- **התחילו קטן**. לפני שמאצילים מערכת שלמה, נסו זרימה של שני-שלושה סוכנים ובדקו שהיא מתנהגת כצפוי.

העיקרון על: אוטונומיה היא לא "שגר ושכח". היא "שגר, בדוק בנקודות מפתח, ותקן מוקדם". ככל שהמשימה גדולה יותר, כך נקודות הביקורת חשובות יותר. הסוכן מגביר את התפוקה שלכם, אבל השיפוט וההכוונה נשארים באחריותכם.

## איך זה נראה ב-Claude Code - in practice

בואו נחבר את הרעיונות לפעולות קונקרטיות בכלי. זרימה רב-סוכן טיפוסית מורכבת מהצעדים האלה:

- **נכנסים למצב תכנון**: מבקשים מ-Claude תוכנית לפני ביצוע. הוא חוקר בקריאה בלבד ומציג תוכנית. אתם מאשרים או מתקנים לפני שנוגעים בקוד.
- **מבקשים מקבילות מפורשת**: "חקור את שלושת המודולים במקביל בסוכני משנה נפרדים". Claude משגר סוכני משנה, ואתם רואים אותם בפאנל מתחת לשורת הפקודה.
- **מנטרים תוך כדי**: הפאנל מראה כל סוכן ואת מצבו. אפשר לפתוח סוכן, לראות את התמליל שלו, ואף לשלוח לו הודעת תיקון תוך כדי ריצה.
- **מוסיפים אכיפה**: הוק `Stop` שמריץ בדיקות, וסוכן `code-reviewer` שסוקר. אלה נקודות הביקורת האוטומטיות.

השילוב הזה הוא בדיוק הסינתזה של הפרק. כל כלי שלמדנו תורם את חלקו: MCP מרחיב את הכלים שהעובדים יכולים להשתמש בהם, סוכני משנה מבודדים את העבודה הרועשת, מיומנויות מתקננות את התהליכים החוזרים, והוקים אוכפים את המשמעת. אף אחד מהם לבדו לא פותר משימה גדולה, אבל ביחד הם מאפשרים לכם לתזמר עבודה שהיא הרבה מעבר למה שסוכן בודד יכול.

## סיכום

בהרצאה הזו למדנו:

- סוכן אחד נתקל בשתי מגבלות במשימה גדולה: **חלון ההקשר** מתמלא, וה**מיקוד** נפגע. זרימה רב-סוכן פותרת ב**חלק וכבוש**.
- הדפוס המרכזי הוא **מתזמר ועובדים**: הסוכן הראשי מחזיק את התמונה הגדולה ומחלק עבודה לסוכני משנה עם הקשר טרי, שכל אחד מחזיר רק סיכום.
- **תכנון-ואז-ביצוע**: קודם תוכנית כתובה (עם plan mode או קובץ `PLAN.md`) שאתם מאשרים, ורק אז ביצוע. תוכנית היא חוזה שזול לתקן לפני שמושקעת עבודה.
- **נקודות ביקורת** מונעות בזבוז: ביקורת אנושית אחרי כל שלב גדול, סוכן סוקר ייעודי, או הוק `Stop` שמריץ בדיקות. כאן כל כלי הפרק מתחברים.
- מזהים **תלויות**: תת-משימות תלויות רצות בסדר, עצמאיות רצות במקביל.
- הסיכונים: **סחף ממטרה**, **הגברת שגיאות**, **גודש בהקשר** מסיכומים ארוכים, ועלות ואיבוד שקיפות.
- שליטה: תכננו לפני ביצוע, הגבילו היקף וכלים לכל סוכן, הכניסו נקודות ביקורת, דרשו סיכומים קצרים, והתחילו קטן. אוטונומיה היא "שגר, בדוק, תקן מוקדם" - לא "שגר ושכח". השיפוט נשאר באחריותכם.
