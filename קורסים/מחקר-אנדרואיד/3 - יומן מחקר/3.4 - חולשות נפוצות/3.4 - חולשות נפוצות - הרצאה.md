## חולשות נפוצות באפליקציות אנדרואיד - Common Android App Vulnerabilities
עד עכשיו למדנו כלים ושיטות לחקור אפליקציות אנדרואיד.
עכשיו נלמד מה לחפש - מהן החולשות הנפוצות שאנחנו מצפים למצוא כשאנחנו חוקרים אפליקציה.

## אחסון מידע לא מאובטח - Insecure Data Storage
זו אחת החולשות הנפוצות ביותר.
כמו שלמדנו בשיעור על מידע של אפליקציות, מפתחים לפעמים שומרים מידע רגיש בצורה לא מאובטחת:

### מידע רגיש בSharedPreferences
```xml
<string name="password">mySecret123</string>
<string name="api_key">sk-1234567890abcdef</string>
```
סיסמאות, טוקנים ומפתחות API שנשמרים בטקסט גלוי בקבצי XML.

### מידע רגיש במסד נתונים
```sql
SELECT * FROM users;
-- username: admin, password: admin123, token: eyJhbG...
```
מסדי נתונים SQLite שמכילים סיסמאות או טוקנים בלי הצפנה.

### מידע רגיש בלוגים
מפתחים לפעמים שוכחים להסיר שורות debug שמדפיסות מידע רגיש ללוגים:
```java
Log.d("AUTH", "Token: " + authToken);
Log.d("DEBUG", "Password: " + password);
```
אפשר לראות את הלוגים האלו עם `adb logcat`.

### איך לחפש
- בדקי את כל הקבצים ב`/data/data/<package>/`
- חפשי בlogcat מידע רגיש תוך כדי שימוש באפליקציה
- חפשי בקוד קריאות ל`Log.d`, `Log.i`, `Log.e` עם מידע רגיש

## רכיבים מיוצאים - Exported Components
כשרכיב (activity, service, receiver, provider) מוגדר כ-exported בmanifest, כל אפליקציה במכשיר יכולה לגשת אליו.

### פעילות מיוצאת - Exported Activity
```xml
<activity android:name=".AdminPanelActivity" android:exported="true" />
```
כאן כל אפליקציה יכולה לפתוח את מסך האדמין בלי הזדהות:
```bash
adb shell am start -n com.example.app/.AdminPanelActivity
```

### ספק תוכן מיוצא - Exported Content Provider
```xml
<provider android:name=".UserDataProvider"
          android:authorities="com.example.app.provider"
          android:exported="true" />
```
כל אפליקציה יכולה לקרוא את המידע:
```bash
adb shell content query --uri content://com.example.app.provider/users
```
ואולי גם לכתוב או למחוק מידע.

### מקבל שידור מיוצא - Exported Broadcast Receiver
```xml
<receiver android:name=".CommandReceiver" android:exported="true">
    <intent-filter>
        <action android:name="com.example.EXECUTE" />
    </intent-filter>
</receiver>
```
כל אפליקציה יכולה לשלוח broadcast ולהפעיל את הreceiver:
```bash
adb shell am broadcast -a com.example.EXECUTE --es command "delete_all"
```

### איך לחפש
- פתחי את הmanifest בjadx וחפשי `exported="true"`
- שימי לב - מandroid 12 ומעלה, חובה להגדיר exported במפורש. בגרסאות ישנות, רכיב עם intent-filter הוא exported כברירת מחדל
- נסי לגשת לכל רכיב מיוצא עם am ובדקי מה קורה

## Deeplinks לא מאובטחים - Insecure Deeplinks
הרבה אפליקציות רושמות deeplinks שמאפשרות לפתוח מסכים ספציפיים דרך URL:
```xml
<intent-filter>
    <action android:name="android.intent.action.VIEW" />
    <data android:scheme="myapp" android:host="transfer" />
</intent-filter>
```
אם האפליקציה לא מוודאת שהבקשה מגיעה ממקור מהימן, תוקף יכול ליצור אתר זדוני עם לינק כזה:
```html
<a href="myapp://transfer?to=attacker&amount=1000">לחץ כאן</a>
```
והמשתמש שילחץ על הלינק יבצע פעולה באפליקציה בלי לדעת.

### איך לחפש
- חפשי intent-filters עם scheme מותאם אישית בmanifest
- בדקי אם האפליקציה מוודאת את הפרמטרים שהיא מקבלת בdeeplink
- נסי לשלוח deeplinks עם פרמטרים לא צפויים:
```bash
adb shell am start -a android.intent.action.VIEW -d "myapp://transfer?to=hacker&amount=9999"
```

## תקשורת לא מאובטחת - Insecure Communication

### שימוש בHTTP במקום HTTPS
אפליקציות שמתקשרות בHTTP שולחות מידע בטקסט גלוי, כולל סיסמאות וטוקנים.
אפשר לזהות את זה בburp - אם רואים בקשות HTTP (בלי S), זו חולשה.

בנוסף, אפשר לחפש בקוד:
```java
// חולשה - HTTP
String url = "http://api.example.com/login";

// תקין - HTTPS
String url = "https://api.example.com/login";
```

### חוסר בSSL Pinning
גם אם האפליקציה משתמשת בHTTPS, אם אין SSL pinning תוקף שנמצא באותה רשת יכול לבצע man-in-the-middle עם סרטיפיקט מזויף.
כזכור, למדנו איך לעקוף SSL pinning - אם אנחנו מצליחים לעקוף אותו בקלות, ייתכן שהוא לא מומש כראוי.

## מפתחות ומחרוזות בקוד - Hardcoded Secrets
מפתחים לפעמים שמים מפתחות וסודות ישירות בקוד:
```java
String API_KEY = "AIzaSyB1234567890abcdef";
String SECRET = "super_secret_key_12345";
String ADMIN_PASSWORD = "admin123";
```

### איך לחפש
חפשי בjadx מחרוזות שמכילות מילים כמו:
- key, secret, password, token, api
- כתובות URL של שרתים פנימיים
- credentials, auth
## הזרקת קוד בWebView - WebView Vulnerabilities
הרבה מאוד אפליקציות משתמשות בWebView (מנוע אנדרואיד לדפדפן) כדי להציג תוכן מהאינטרנט.
למשל, אם האפליקציה רוצה לפתוח עמוד אינטרנט, או קישור- היא כנראה תשתמש בwebview.

### הפעלת JavaScript
המון פעמים, נראה שאפליקציות אפילו מאנבלות javascript בwebview (לרוב)
```java
webView.getSettings().setJavaScriptEnabled(true);
```
אם הWebView טוען תוכן ממקור לא מאובטח (HTTP), תוקף יכול להזריק javascript דרך הרשת.
ואפילו מעבר- הרבה פעמים נראה שאפליקציות פותחות עם intent-ים אפשרות להעביר url-ים שבflow מסוים עושות webview.loadUrl() לקלט שלנו.
וכך, בהינתן שליחת intent לאפליקציה אנחנו יכולים להעביר url שלנו שנטען אחרכך בwebview עם אפשרות להריץ javascript (חולשת xss)

### גישור JavaScript לJava
נניח ומצאנו xss, או איזשהי הזרקת javascript לwebview לפעמים, כשהאפליקציה מקפנגת את הwebview- היא מוסיפה אפשרות להריץ פונקציות java מסוימות בjavascript שלנו.
```java
webView.addJavascriptInterface(new MyJSInterface(), "Android");
```
הפונקציה `addJavascriptInterface` חושפת מתודות java לjavascript שרץ בWebView.
המון פעמים נמצא הרבה מאוד דברים מעניינים שנוכל לעשות עם הפונקציות האלו. (תמיד שווה הסתכלות)

### איך לחפש
- חפשי בjadx שימוש בWebView
- בדקי אם `setJavaScriptEnabled(true)` מופעל
- בדקי אם יש `addJavascriptInterface`
- בדקי מאיפה הWebView טוען תוכן - HTTP? מקור חיצוני?
- חפשי בmanifest את המילים url, webview, web כדי לחפש פעילויות שאולי מקבלות דרך intent קישורים, (כמובן יש לבדוק שם exported=true)

## הרשאות מיותרות - Excessive Permissions
אפליקציות שמבקשות הרשאות שהן לא באמת צריכות - למשל אפליקציית פנס שמבקשת גישה לאנשי קשר, למיקום ולמצלמה.
זה לא בהכרח חולשה טכנית, אבל זה סימן לאפליקציה חשודה שאולי אוספת מידע.

### איך לבדוק
- פתחי את הmanifest ובדקי את רשימת ההרשאות
- שאלי - האם ההרשאה הזו הגיונית לפונקציונליות של האפליקציה?

## סיכום
כשחוקרים אפליקציה, כדאי לעבור על הרשימה הזו בצורה שיטתית:
1. בדקי אחסון מידע - SharedPreferences, databases, logcat
2. בדקי רכיבים מיוצאים - activities, providers, receivers ומחפשים מילים כמו exec, shell, webview, web, url, admin, config
3. בדקי deeplinks - האם יש validation
4. בדקי תקשורת - HTTP/HTTPS, SSL pinning
5. חפשי מפתחות וסודות בקוד
6. בדקי WebView - JavaScript enabled, JS interfaces
7. בדקי הרשאות - האם הגיוניות לאפליקציה
