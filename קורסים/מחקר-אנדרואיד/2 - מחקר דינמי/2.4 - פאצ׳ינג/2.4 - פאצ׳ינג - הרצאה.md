## פיצ׳פוץ׳ - Patching
עד עכשיו למדנו שני סוגי מחקר:
- מחקר סטטי - קראנו קוד בjadx בלי להריץ את האפליקציה
- מחקר דינמי - השתמשנו בfrida כדי לעשות hook לפונקציות בזמן ריצה

עכשיו נלמד טכניקה שלישית - **פאצ'ינג** (patching).
בפאצ'ינג אנחנו משנים את הקוד של האפליקציה עצמה, בונים אותה מחדש, חותמים אותה, ומתקינים את הגרסה המשונה על המכשיר.
בניגוד לfrida שמבצעת שינויים בזמן ריצה (ושנעלמים כשסוגרים את האפליקציה), בפאצ'ינג השינוי קבוע - האפליקציה עצמה שונתה.

## תהליך העבודה - Patching Workflow
התהליך המלא נראה ככה:
1. פירוק הAPK עם apktool (מקבלים קוד smali)
2. עריכת קוד הsmali
3. בנייה מחדש של הAPK עם apktool
4. חתימת הAPK
5. התקנה על המכשיר

בואו נעבור על כל שלב.

## שלב 1 - פירוק - Decompile
כמו שלמדנו בשיעור על הAPK, הכלי apktool מפרק apk לתיקייה עם כל הקבצים שלו, כולל קוד smali.
```bash
apktool d app.apk -o app_patched
```
הדגל `-o` מציין את שם תיקיית הפלט.
בתיקייה שנוצרה נמצא תיקיית `smali/` שמכילה את כל קוד הsmali של האפליקציה.

## שלב 2 - עריכת קוד - Smali Editing
### תזכורת - מה זה smali?
כזכור, smali הוא פורמט קריא-אדם של הbytecode של dalvik.
כשkotlin/java מקומפלים לאנדרואיד, הם הופכים לbytecode בפורמט dex. הsmali הוא ייצוג טקסטואלי של הdex, בדומה לassembly.

### סוגי משתנים - Types
בsmali סוגי המשתנים מסומנים באותיות:

| סוג | סימון |
|---|---|
| int | I |
| boolean | Z |
| long | J |
| float | F |
| double | D |
| void | V |
| מחלקה | Lcom/example/ClassName; |
| מערך | [I (מערך של int) |

### רגיסטרים - Registers
בsmali משתמשים ברגיסטרים במקום משתנים.
יש שני סוגי רגיסטרים:
- `v0, v1, v2...` - רגיסטרים לוקליים
- `p0, p1, p2...` - רגיסטרים של פרמטרים (p0 הוא this במתודות לא סטטיות)

בתחילת כל מתודה מוגדר מספר הרגיסטרים:
```smali
.method public checkPassword(Ljava/lang/String;)Z
    .registers 4
```
כאן יש 4 רגיסטרים, מתוכם p0 הוא this, וp1 הוא הפרמטר password.

### פקודות נפוצות - Common Instructions
הנה הפקודות הנפוצות ביותר שנתקל בהן:

**העברת ערכים:**
```smali
const/4 v0, 0x1          # v0 = 1
const-string v0, "hello"  # v0 = "hello"
move v0, v1               # v0 = v1
```

**החזרת ערך:**
```smali
return v0          # החזרת ערך
return-void        # החזרת void
```

**קריאה למתודה:**
```smali
invoke-virtual {p0, p1}, Lcom/example/MyClass;->myMethod(Ljava/lang/String;)V
```
כאן קוראים למתודה `myMethod` על האובייקט `p0` עם הפרמטר `p1`.

**השוואות וקפיצות:**
```smali
if-eqz v0, :label    # אם v0 == 0 קפוץ ל label
if-nez v0, :label    # אם v0 != 0 קפוץ ל label
if-eq v0, v1, :label  # אם v0 == v1 קפוץ ל label
goto :label           # קפוץ ל label
```

### דוגמה מעשית - Practical Example
נניח שיש לנו פונקציה שבודקת סיסמה:
```java
public boolean checkPassword(String password) {
    if (password.equals("secret123")) {
        return true;
    }
    return false;
}
```

בsmali היא תיראה בערך ככה:
```smali
.method public checkPassword(Ljava/lang/String;)Z
    .registers 4

    const-string v0, "secret123"

    invoke-virtual {p1, v0}, Ljava/lang/String;->equals(Ljava/lang/Object;)Z

    move-result v0

    if-eqz v0, :cond_false

    const/4 v0, 0x1
    return v0

    :cond_false
    const/4 v0, 0x0
    return v0
.end method
```

כדי לעקוף את הבדיקה, אפשר פשוט לשנות את הפונקציה שתמיד תחזיר true:
```smali
.method public checkPassword(Ljava/lang/String;)Z
    .registers 4

    const/4 v0, 0x1
    return v0
.end method
```

מחקנו את כל הלוגיקה והחזרנו תמיד 1 (true).

### איך למצוא את הקוד הנכון - Finding the Right Code
טיפ חשוב - אל תנסי לקרוא smali מאפס. השתמשי בjadx כדי להבין את הלוגיקה בjava, ואז חפשו את הפונקציה המתאימה בקבצי הsmali.
מבנה התיקיות של הsmali תואם למבנה החבילות:
```
smali/com/example/app/MainActivity.smali
```
תואם למחלקה:
```
com.example.app.MainActivity
```

## שלב 3 - בנייה מחדש - Rebuild
אחרי שערכנו את הsmali, נבנה מחדש את הAPK:
```bash
apktool b app_patched -o patched.apk
```
הפקודה `b` (שזה build) בונה מחדש את הAPK מהתיקייה שפירקנו ומייצרת קובץ APK חדש.

## שלב 4 - חתימה - Signing
כל APK חייב להיות חתום דיגיטלית כדי שאנדרואיד יסכים להתקין אותו.
כשאנחנו בונים APK מחדש, החתימה המקורית כבר לא תקינה ואנחנו צריכים לחתום עליו מחדש.

קודם ניצור מפתח חתימה (צריך לעשות את זה רק פעם אחת):
```bash
keytool -genkey -v -keystore my-key.keystore -alias mykey -keyalg RSA -keysize 2048 -validity 10000
```
הפקודה תבקש סיסמה ופרטים נוספים - אפשר למלא מה שרוצים, זה לא משנה לצורך מחקר.

אחרי שיש לנו מפתח, נחתום את הAPK:
```bash
apksigner sign --ks my-key.keystore --ks-key-alias mykey patched.apk
```
אם apksigner לא מותקן, אפשר למצוא אותו בתוך הAndroid SDK:
```bash
~/Library/Android/sdk/build-tools/<version>/apksigner
```

## שלב 5 - התקנה - Installation
עכשיו נתקין את הAPK המשונה על המכשיר:
```bash
adb install patched.apk
```
אם האפליקציה המקורית כבר מותקנת, נצטרך קודם להסיר אותה:
```bash
adb uninstall com.example.app
adb install patched.apk
```
חייבים להסיר את המקורית כי החתימה שלנו שונה מהחתימה המקורית, ואנדרואיד לא מאפשר לעדכן אפליקציה עם חתימה שונה.

## מתי להשתמש בפאצ'ינג ומתי בפרידה - Patching vs Frida
**פאצ'ינג** מתאים כאשר:
- רוצים שינוי קבוע שנשמר גם אחרי הפעלה מחדש
- רוצים להוסיף קוד חדש לאפליקציה
- רוצים לשנות משאבים (תמונות, טקסטים, layouts)
- רוצים להסיר קוד לגמרי

**פרידה** מתאימה כאשר:
- רוצים לנתח את האפליקציה בזמן ריצה (לראות ערכי משתנים, פרמטרים)
- רוצים שינוי זמני לצורך בדיקה
- האפליקציה מבצעת בדיקות integrity על עצמה
- רוצים לעשות hook לפונקציות של המערכת (לא רק של האפליקציה)

בפועל, הרבה פעמים משתמשים בשני הכלים ביחד.
