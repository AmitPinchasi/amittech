## שרת המערכת - System Server
בשיעור על הזיגוט למדנו שהדבר הראשון שהזיגוט עושה אחרי שעלה, הוא ליצור את הsystem_server.
הsystem_server הוא הפרוסס הכי חשוב באנדרואיד - הוא מריץ את כל השירותים שגורמים לאנדרואיד לעבוד.
בלי system_server, אנדרואיד הוא סתם לינוקס רגיל בלי ממשק, בלי אפליקציות, בלי כלום.

## מה הsystem_server מריץ - What system_server Runs
הsystem_server מריץ עשרות שירותי מערכת. כל שירות אחראי על תחום אחר.
כל השירותים רצים **באותו פרוסס** (system_server), כל אחד בthread נפרד.

השירותים מחולקים לשלוש קבוצות לפי סדר הטעינה:

### שירותי אתחול - Bootstrap Services
שירותים שחייבים לעלות ראשונים כי שירותים אחרים תלויים בהם:

**ActivityManagerService (AMS)**
השירות הכי מרכזי באנדרואיד. הוא אחראי על:
- ניהול מחזור החיים של כל הactivities (יצירה, השהיה, סגירה)
- ניהול פרוססים - מחליט אילו פרוססים לסגור כשנגמר הזכרון
- שליחה וקבלה של intents ו-broadcasts
- בקשה מהזיגוט ליצור פרוססים חדשים
- ניהול tasks ו-back stack

כשאמרנו שactivity manager שולח intent-ים, בפועל זה הAMS שרץ בsystem_server.

**PackageManagerService (PMS)**
אחראי על ניהול כל האפליקציות המותקנות:
- התקנה והסרה של APKs
- ניהול הרשאות - מי מקבל איזו הרשאה
- פרסור (parsing) של AndroidManifest.xml
- ניהול חתימות דיגיטליות

כשמריצים `adb install app.apk`, הבקשה מגיעה לPMS.

**PowerManagerService**
ניהול צריכת החשמל - מצב שינה, wake locks, בהירות מסך.

**DisplayManagerService**
ניהול מסכים - רזולוציה, צפיפות, מסכים חיצוניים.

### שירותי ליבה - Core Services

**BatteryService**
ניטור מצב הסוללה, טמפרטורה, מתח.

**UsageStatsService**
מעקב אחרי שימוש באפליקציות - כמה זמן כל אפליקציה רצה, מתי נפתחה לאחרונה.

**WebViewUpdateService**
ניהול עדכונים של WebView.

### שירותים נוספים - Other Services

**WindowManagerService (WMS)**
ניהול החלונות על המסך:
- סדר שכבות (z-order) של חלונות
- אנימציות
- טיפול באירועי מגע והעברתם לחלון הנכון
- ניהול הstatus bar ו-navigation bar

**ConnectivityService**
ניהול חיבורי רשת - WiFi, סלולר, Bluetooth, VPN.
הוא מחליט איזה חיבור פעיל ומנתב את התעבורה.

**NetworkPolicyManagerService**
מדיניות רשת - חסימת אפליקציות מאינטרנט ברקע, מגבלות data.

**NotificationManagerService**
ניהול הnotifications - הצגה, סדר עדיפויות, ערוצים.

**LocationManagerService**
ניהול שירותי מיקום - GPS, מיקום לפי רשת, geofencing.

**AudioService**
ניהול אודיו - עוצמת קול, ניתוב לרמקול/אוזניות, focus.

**CameraService**
ניהול גישה למצלמה.

**AlarmManagerService**
ניהול התראות מתוזמנות (alarms) - מפעיל אפליקציות בזמנים מתוזמנים.

**InputManagerService**
ניהול קלט - מגע, מקלדת, כפתורים פיזיים.

**ClipboardService**
ניהול הclipboard - העתק/הדבק בין אפליקציות.

ויש עוד עשרות שירותים נוספים. אפשר לראות את כל השירותים שרצים:
```bash
adb shell service list
```

## תקשורת עם השירותים - Binder IPC
איך אפליקציה מתקשרת עם שירותי system_server?
התשובה: **Binder**.

כפי שלמדנו, Binder הוא מנגנון IPC (Inter-Process Communication) ייחודי לאנדרואיד.
בואו נבין לעומק איך זה עובד.

### למה לא sockets רגילים?
בתיאוריה, אפליקציות יכלו לתקשר עם system_server דרך sockets או pipes.
אבל Binder מציע יתרונות משמעותיים:
- **אבטחה** - הקרנל מוודא את הUID של השולח, אי אפשר לזייף זהות
- **ביצועים** - Binder מעביר מידע בcopy אחד בלבד (kernel buffer), לא שניים כמו בsockets
- **נוחות** - Binder מאפשר לקרוא לפונקציות בפרוסס אחר כאילו הן לוקליות (RPC)

### איך Binder עובד
```
אפליקציה                    system_server
    |                              |
    |  -- "startActivity()" -->    |
    |       (Binder transaction)   |
    |                      ActivityManagerService
    |                              |
    |  <-- result --               |
```

1. האפליקציה רוצה להפעיל activity חדש
2. היא קוראת ל-`startActivity()` שמאחורי הקלעים שולחת Binder transaction
3. הקרנל מעביר את הבקשה לsystem_server (לthread של הAMS)
4. הAMS מטפל בבקשה ומחזיר תשובה דרך Binder

כל זה קורה דרך הדרייבר `/dev/binder` בקרנל.

### הProxy Pattern
בקוד java אנחנו יכולים לתקשר עם Binder בתקשורת מוסתרת מהמפתח.
כשאפליקציה קוראת ל:
```java
ActivityManager am = getSystemService(Context.ACTIVITY_SERVICE);
```
היא מקבלת אובייקט **proxy** שנראה כמו ActivityManager רגיל, אבל כל קריאה לפונקציה שלו בעצם שולחת Binder transaction לsystem_server.

המבנה:
```
אפליקציה:
ActivityManager (proxy/client) --> IActivityManager.Stub.Proxy

Binder driver (/dev/binder)

system_server:
  ActivityManagerService (server) --> IActivityManager.Stub
```

### צפייה בBinder transactions
אפשר לראות את הBinder transactions בזמן אמת:
```bash
adb shell cat /sys/kernel/debug/binder/transactions
```
או לראות סטטיסטיקות:
```bash
adb shell cat /sys/kernel/debug/binder/stats
```

## הפקודה dumpsys
הכלי `dumpsys` מאפשר לנו לשאול כל שירות בsystem_server על המצב שלו.
זה כלי מחקר מאוד חזק.

### מידע על activities
```bash
adb shell dumpsys activity activities
```
מציג את כל הactivities שרצים כרגע, כולל הback stack.

### מידע על אפליקציה ספציפית
```bash
adb shell dumpsys package com.example.app
```
מציג הכל על אפליקציה - הרשאות, activities, services, receivers, providers, חתימה, גרסה, ועוד.

### מידע על הרשאות
```bash
adb shell dumpsys package permissions
```

### מידע על חיבורי רשת
```bash
adb shell dumpsys connectivity
```

### מידע על סוללה
```bash
adb shell dumpsys battery
```

### מידע על notifications
```bash
adb shell dumpsys notification
```

### כל השירותים
```bash
adb shell dumpsys -l
```
מציג רשימה של כל השירותים שאפשר לעשות להם dump.

## system_server מבחינת מחקר
### למה זה חשוב
הsystem_server הוא נקודה מרכזית במחקר אנדרואיד:

**חקירת הרשאות**
הPMS (PackageManagerService) הוא שמנהל הרשאות. אם נמצא חולשה בPMS, נוכל אולי לעקוף בדיקות הרשאות.

**חקירת intents**
הAMS מנהל את כל הintent-ים. הבנה של איך הוא מעבד intents עוזרת לנו למצוא חולשות בintent handling.

**חקירת Binder**
כל תקשורת עם שירותי מערכת עוברת דרך Binder. ניתוח Binder transactions יכול לחשוף מידע על מה אפליקציות עושות.

 **מבחינת privilege escalation**
אם נמצא חולשה בsystem_server, אנחנו מקבלים הרשאות system - שזה כמעט כמו root. הsystem_server רץ עם UID 1000 (system) שיש לו הרבה מאוד הרשאות.

### קריסת system_server
אם הsystem_server קורס, אנדרואיד יעשה reboot אוטומטי (soft reboot) - כי בלעדיו המערכת לא יכולה לעבוד.
אפשר לראות את זה בפעולה:
```bash
adb shell kill $(adb shell pidof system_server)
```
המכשיר יבצע restart מהיר (לא reboot מלא של הקרנל, אלא restart מהזיגוט).
